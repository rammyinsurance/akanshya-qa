What is Playwright?

Playwright is Microsoft's end-to-end testing framework. It drives real browsers (Chromium, Firefox, WebKit) to automate user flows — clicks, form fills, navigation, API calls — and verify that your web app behaves correctly.

Why QA teams pick Playwright

  • Auto-waiting — no more sleep(5); it waits for elements automatically
  • One API, three browsers — Chrome, Firefox, and Safari engines out of the box
  • Fast & parallel — tests run in isolated browser contexts simultaneously
  • Trace viewer — visual timeline of every action, screenshots, network calls
  • Codegen — record your clicks and it writes the test code for you
  • Network interception — mock APIs, throttle, block requests
  • Mobile emulation — test responsive design with device profiles

🛠️ Setup

Prerequisites

  • Node.js 18+ — download from nodejs.org
  • A code editor (VS Code recommended — has an official Playwright extension)
  • Terminal / Command Prompt

Install Playwright in 3 commands

mkdir my-qa-project && cd my-qa-project npm init playwright@latest npx playwright test

The init command asks:

  1. TypeScript or JavaScript? → TypeScript (recommended)
  2. Tests folder name? → tests
  3. Add GitHub Actions? → Yes (useful for CI)
  4. Install browsers? → Yes (downloads Chromium, Firefox, WebKit ~300MB)

Project structure after init

my-qa-project/
├── tests/                  ← your test files (*.spec.ts)
│   └── example.spec.ts
├── tests-examples/         ← demo tests (can delete)
├── playwright.config.ts    ← main config (browsers, timeouts, reporters)
├── package.json
└── .github/workflows/      ← CI pipeline (if you said yes)

VS Code extension (recommended)

Install Playwright Test for VSCode by Microsoft. You'll get:

  • Run individual tests from a gutter icon next to each test
  • Record new tests with one click
  • Step-through debugging with breakpoints
  • Pick locators by hovering elements in the browser
💡 Quick start: This guide ships with ready-to-run test files in the tests/ folder. Copy the whole project folder, run npm install, then npx playwright test.
BEGINNER

1. Beginner Examples

1.1 Your first test — navigate and check title

Goal: open a page and verify the title tests/beginner/01-first-test.spec.ts 
import { test, expect } from '@playwright/test';

test('has correct title', async ({ page }) => {
    await page.goto('https://playwright.dev/');
    await expect(page).toHaveTitle(/Playwright/);
});

Run it: npx playwright test 01-first-test

1.2 Click a link and verify navigation

Goal: click "Get started" and check the new heading appears tests/beginner/02-click-link.spec.ts 
import { test, expect } from '@playwright/test';

test('get started link', async ({ page }) => {
    await page.goto('https://playwright.dev/');
    await page.getByRole('link', { name: 'Get started' }).click();
    await expect(page.getByRole('heading', { name: 'Installation' })).toBeVisible();
});

1.3 Fill a form

Goal: fill inputs and submit a login form tests/beginner/03-fill-form.spec.ts 
import { test, expect } from '@playwright/test';

test('login form', async ({ page }) => {
    await page.goto('https://the-internet.herokuapp.com/login');

    await page.getByLabel('Username').fill('tomsmith');
    await page.getByLabel('Password').fill('SuperSecretPassword!');
    await page.getByRole('button', { name: 'Login' }).click();

    await expect(page.getByText('You logged into a secure area!')).toBeVisible();
});

1.4 Locators — the right way to find elements

Playwright recommends user-facing locators (role, label, text) over CSS selectors. They're more resilient when markup changes.

Locator Example When to use
getByRole getByRole('button', { name: 'Submit' }) Buttons, links, headings — 80% of cases
getByLabel getByLabel('Email') Form inputs with labels
getByPlaceholder getByPlaceholder('Search...') Inputs with only placeholders
getByText getByText('Welcome back') Visible text on page
getByTestId getByTestId('submit-btn') When devs add data-testid attributes
locator locator('.my-class') Last resort — CSS/XPath

1.5 Common assertions (expect)

Cheat sheet
// Visibility
await expect(locator).toBeVisible();
await expect(locator).toBeHidden();

// Text content
await expect(locator).toHaveText('Exact text');
await expect(locator).toContainText('partial');

// Attributes & values
await expect(locator).toHaveValue('input value');
await expect(locator).toHaveAttribute('href', '/home');
await expect(locator).toBeEnabled();
await expect(locator).toBeChecked();

// Counts
await expect(page.getByRole('listitem')).toHaveCount(5);

// Page-level
await expect(page).toHaveURL(/.*dashboard/);
await expect(page).toHaveTitle('Dashboard');
💡 Auto-retry: Every expect automatically retries for up to 5 seconds. No need to write waitForTimeout — Playwright waits until the element is ready.
INTERMEDIATE

2. Intermediate Examples

2.1 Page Object Model (POM)

Wrap each page's interactions into a class. Makes tests readable and easy to maintain when the UI changes.

The page object tests/intermediate/pages/LoginPage.ts 
import { Page, Locator } from '@playwright/test';

export class LoginPage {
    readonly page: Page;
    readonly username: Locator;
    readonly password: Locator;
    readonly submit: Locator;
    readonly flash: Locator;

    constructor(page: Page) {
        this.page = page;
        this.username = page.getByLabel('Username');
        this.password = page.getByLabel('Password');
        this.submit   = page.getByRole('button', { name: 'Login' });
        this.flash    = page.locator('#flash');
    }

    async goto() {
        await this.page.goto('https://the-internet.herokuapp.com/login');
    }

    async login(user: string, pass: string) {
        await this.username.fill(user);
        await this.password.fill(pass);
        await this.submit.click();
    }
}
The test using the POM tests/intermediate/01-pom.spec.ts 
import { test, expect } from '@playwright/test';
import { LoginPage } from './pages/LoginPage';

test('successful login via POM', async ({ page }) => {
    const login = new LoginPage(page);
    await login.goto();
    await login.login('tomsmith', 'SuperSecretPassword!');
    await expect(login.flash).toContainText('You logged into a secure area!');
});

import { test, expect } from '@playwright/test';

test.describe('Dashboard suite', () => {
    test.beforeEach(async ({ page }) => {
        await page.goto('https://example.com/login');

    test('shows user name', async ({ page }) => {
        await expect(page.getByTestId('user-name')).toHaveText('QA Tester');
    });

    test('shows notifications', async ({ page }) => {
        await expect(page.getByRole('button', { name: /notifications/i })).toBeVisible();
    });
});

2.3 API testing (no browser needed)

tests/intermediate/03-api.spec.ts 
import { test, expect } from '@playwright/test';

test('GET request returns users', async ({ request }) => {
    const res = await request.get('https://reqres.in/api/users?page=2');
    expect(res.status()).toBe(200);

    const body = await res.json();
    expect(body.data.length).toBeGreaterThan(0);
    expect(body.data[0]).toHaveProperty('email');
});

test('POST creates a user', async ({ request }) => {
    const res = await request.post('https://reqres.in/api/users', {
        data: { name: 'QA Bot', job: 'Tester' }
    });
    expect(res.status()).toBe(201);
    const body = await res.json();
    expect(body.name).toBe('QA Bot');
});

2.4 Reusing authentication state

Log in once, save the session, reuse it across all tests — huge time-saver.

Global setup auth.setup.ts 
import { test as setup } from '@playwright/test';

const authFile = 'playwright/.auth/user.json';

setup('authenticate', async ({ page }) => {
    await page.goto('https://example.com/login');
    await page.getByLabel('Email').fill('qa@test.com');
    await page.getByLabel('Password').fill('pass123');
    await page.getByRole('button', { name: 'Sign in' }).click();
    await page.waitForURL('**/dashboard');
    await page.context().storageState({ path: authFile });
});

2.5 File upload & download

tests/intermediate/04-files.spec.ts 
import { test, expect } from '@playwright/test';
import path from 'path';

test('upload a file', async ({ page }) => {
    await page.goto('https://the-internet.herokuapp.com/upload');
    await page.setInputFiles('#file-upload', path.join(__dirname, 'sample.txt'));
    await page.click('#file-submit');
    await expect(page.locator('#uploaded-files')).toContainText('sample.txt');
});

test('download a file', async ({ page }) => {
    await page.goto('https://the-internet.herokuapp.com/download');
    const downloadPromise = page.waitForEvent('download');
    await page.getByRole('link', { name: /\.txt$/ }).first().click();
    const download = await downloadPromise;
    await download.saveAs('/tmp/' + download.suggestedFilename());
});

2.6 Handling dialogs, new tabs, iframes

Common patterns
// JavaScript alert/confirm/prompt
page.on('dialog', dialog => dialog.accept());
await page.getByRole('button', { name: 'Delete' }).click();

// New tab opens
const [newTab] = await Promise.all([
    page.context().waitForEvent('page'),
    page.getByRole('link', { name: 'Open in new tab' }).click(),
]);
await newTab.waitForLoadState();

// Iframe
const frame = page.frameLocator('#my-iframe');
await frame.getByRole('button', { name: 'Inside frame' }).click();
ADVANCED

3. Advanced Examples

3.1 Network interception & API mocking

Mock backend responses to test edge cases (errors, slow responses, empty data) without needing a real broken backend.

tests/advanced/01-mocking.spec.ts 
import { test, expect } from '@playwright/test';

test('shows empty state when API returns no users', async ({ page }) => {
    // Intercept the API before navigation
    await page.route('**/api/users', route => {
        route.fulfill({
            status: 200,
            contentType: 'application/json',
            body: JSON.stringify({ data: [] }),
        });
    });

    await page.goto('https://my-app.com/users');
    await expect(page.getByText('No users found')).toBeVisible();
});

test('shows error banner when API fails', async ({ page }) => {
    await page.route('**/api/users', route => route.fulfill({ status: 500 }));
    await page.goto('https://my-app.com/users');
    await expect(page.getByRole('alert')).toContainText('Something went wrong');
});

3.2 Visual regression testing

tests/advanced/02-visual.spec.ts 
import { test, expect } from '@playwright/test';

test('homepage visual snapshot', async ({ page }) => {
    await page.goto('https://playwright.dev/');
    await expect(page).toHaveScreenshot('homepage.png', {
        maxDiffPixelRatio: 0.01,   // allow 1% difference
        fullPage: true,
    });
});

test('button component snapshot', async ({ page }) => {
    await page.goto('https://my-app.com/components');
    await expect(page.getByRole('button', { name: 'Primary' }))
        .toHaveScreenshot('primary-button.png');
});

First run creates a baseline image. Later runs compare against it. Update with npx playwright test --update-snapshots.

3.3 Mobile emulation

In playwright.config.ts add mobile projects 
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
    projects: [
        { name: 'chromium',     use: { ...devices['Desktop Chrome'] } },
        { name: 'firefox',      use: { ...devices['Desktop Firefox'] } },
        { name: 'webkit',       use: { ...devices['Desktop Safari'] } },
        { name: 'Mobile Chrome', use: { ...devices['Pixel 5'] } },
        { name: 'Mobile Safari', use: { ...devices['iPhone 13'] } },
    ],
});

3.4 Custom fixtures

Build reusable "ingredients" — a logged-in user, a pre-seeded todo list, an admin session.

tests/advanced/fixtures.ts 
import { test as base } from '@playwright/test';
import { LoginPage } from '../intermediate/pages/LoginPage';

type MyFixtures = {
    loginPage: LoginPage;
    loggedInPage: LoginPage;
};

export const test = base.extend<MyFixtures>({
    loginPage: async ({ page }, use) => {
        const login = new LoginPage(page);
        await login.goto();
        await use(login);
    },
    loggedInPage: async ({ page }, use) => {
        const login = new LoginPage(page);
        await login.goto();
        await login.login('tomsmith', 'SuperSecretPassword!');
        await use(login);
    },
});

export { expect } from '@playwright/test';
Using the fixture
import { test, expect } from './fixtures';

test('logged-in user sees welcome', async ({ loggedInPage }) => {
    await expect(loggedInPage.flash).toContainText('secure area');
});

3.5 Parallel execution & sharding

Playwright runs tests in parallel by default. For big suites, split across machines:

npx playwright test --shard=1/4 npx playwright test --shard=2/4

Each CI machine runs 1/4 of the tests simultaneously.

3.6 Accessibility testing

Install the Axe plugin:

npm i -D @axe-core/playwright
tests/advanced/03-a11y.spec.ts 
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('home page has no a11y violations', async ({ page }) => {
    await page.goto('https://playwright.dev/');
    const results = await new AxeBuilder({ page }).analyze();
    expect(results.violations).toEqual([]);
});

🌐 Cross-Browser Testing

Playwright ships with three browser engines. Configure which ones run in playwright.config.ts.

🟢 Chromium

Powers Chrome & Edge. Fastest, most features, great DevTools integration.

npx playwright test --project=chromium

🦊 Firefox

Mozilla's Gecko engine. Catches CSS & JS quirks Chrome doesn't.

npx playwright test --project=firefox

🧭 WebKit

Safari's engine (works on Windows/Linux too). Essential if you have iOS users.

npx playwright test --project=webkit

📱 Mobile

Emulate iPhone, Pixel, iPad — real viewport + touch + user-agent.

npx playwright test --project="Mobile Chrome"

Running across all browsers at once

npx playwright test

Without --project, every test runs in every configured browser. If you have 10 tests × 3 browsers = 30 test runs.

Headed mode (see the browser)

npx playwright test --headed

Slow motion (great for demos & debugging)

npx playwright test --headed --slow-mo=1000

🔍 Debugging & Reports

1. UI Mode (best for writing tests)

npx playwright test --ui

Opens a GUI where you can pick tests, step through actions, inspect the DOM at each step, and watch them run live.

2. Debug mode (Playwright Inspector)

npx playwright test --debug

Pauses at each line, lets you step through, and suggests locators.

3. Codegen (record tests by clicking)

npx playwright codegen https://my-app.com

Opens a browser — every action you perform gets turned into test code in real-time.

4. Trace Viewer (post-mortem for failures)

npx playwright show-trace trace.zip

After a test fails in CI, download the trace and view the full timeline: screenshots at every step, console logs, network calls, DOM snapshots.

5. HTML Report

npx playwright show-report

After any run, this opens a polished HTML report with pass/fail, durations, screenshots, videos, and traces for failures.

💡 Pro tip: Enable trace: 'retain-on-failure' and video: 'retain-on-failure' in playwright.config.ts. You'll only capture artifacts when tests fail, keeping CI fast.

⚙️ CI/CD with GitHub Actions

If you chose "Add GitHub Actions" during setup, you already have a workflow file. Here's a production-ready version:

.github/workflows/playwright.yml 
name: Playwright Tests
on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: 20
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright browsers
      run: npx playwright install --with-deps
    - name: Run tests
      run: npx playwright test
    - uses: actions/upload-artifact@v4
      if: always()
      with:
        name: playwright-report
        path: playwright-report/
        retention-days: 30

Running tests on a schedule

Add this to run every morning at 6 AM UTC (smoke tests against production):

on:
  schedule:
    - cron: '0 6 * * *'

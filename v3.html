Q26 What is the difference between verification and validation? ▼
BasicIntermediate
✦ Model Answer

Verification — "Are we building the product right?" It's a static process checking that the software conforms to specifications. Examples: code reviews, requirement reviews, design walkthroughs. No execution required.

Validation — "Are we building the right product?" It's a dynamic process — actually running the software to ensure it meets the user's needs. Examples: functional testing, UAT, system testing. Execution required.

Scenario: If a requirement says "login button must be blue" and the developer builds a blue button that opens the wrong page — verification passes (blue button exists) but validation fails (wrong page). Both are necessary.

Q31 What is the Software Testing Life Cycle (STLC)? ▼
Basic
✦ Model Answer

STLC is the sequence of activities performed during the testing process. The phases are:

  • 1. Requirement Analysis: Review requirements, identify testable requirements, clarify ambiguities. Output: Requirements Traceability Matrix (RTM)
  • 2. Test Planning: Define scope, strategy, resources, timeline, risks. Output: Test Plan document
  • 3. Test Case Design: Write detailed test cases and test scripts. Output: Test cases, test data
  • 4. Test Environment Setup: Prepare test environment, install software, configure data
  • 5. Test Execution: Execute test cases, log defects, update status
  • 6. Test Closure: Evaluate exit criteria, prepare test summary report, lessons learned
Q32 What are the 7 principles of software testing? ▼
Basic
✦ Model Answer
  • 1. Testing shows presence of defects — it reduces the probability of undiscovered defects but cannot prove absence
  • 2. Exhaustive testing is impossible — test with risk-based and priority-driven approaches
  • 3. Early testing saves money — defects found in requirements are cheaper to fix than in production
  • 4. Defect clustering — most defects are found in a small number of modules (80/20 rule)
  • 5. Pesticide paradox — repeating the same tests finds fewer new bugs; tests must evolve
  • 6. Testing is context dependent — safety-critical testing differs from e-commerce testing
  • 7. Absence of errors fallacy — a bug-free system that doesn't meet user needs still fails
Q33 Explain the defect lifecycle with all statuses. ▼
Basic

Defect statuses: New → Assigned → Open → Fixed → Retest → Verified → Closed. Also: Rejected (not a bug), Deferred (later release), Reopened (fix didn't work), Duplicate.

Q34 — Equivalence Partitioning: Divides input data into valid/invalid partitions; test one value per partition. Example: Age 18–65 allowed → test 40 (valid partition), 10 (invalid <18), 70 (invalid >65).

Q35 — BVA (Boundary Value Analysis): Tests values at boundaries. For age 18–65: test 17, 18, 19, 64, 65, 66. Defects hide at boundaries.

Q36 — Decision Tables: When multiple conditions determine an action. Create a table with all condition combinations (Y/N). Ensures all paths tested.

Q37 — State Transition: Models application states and transitions. Test: valid transitions work, invalid transitions are blocked. Example: Patient appointment states (Requested → Confirmed → Completed / Cancelled).

Q38 — Functional vs Non-Functional: Functional = WHAT system does (features). Non-functional = HOW it performs (performance, security, usability, compatibility).

Q39 — Regression Testing: Verifies new code changes haven't broken existing functionality. Run after: bug fixes, new features, refactoring, releases. I've automated regression using Selenium in current project — 65% coverage saves manual effort.

Q40 — Smoke vs Sanity: Smoke = broad surface test on new build (should it load?). Sanity = narrow test after bug fix (does the specific fix work?). Smoke fails → build rejected. Sanity isolates fix verification.

Q41 — Risk-Based Testing: Prioritise tests by likelihood and impact of failure. High-impact business flows first, historical defect hotspots, complex areas, recent changes. Use defect heatmaps to guide effort.

Q42 — Good Bug Report: Contains: ID, summary, severity/priority, environment, steps to reproduce, expected result, actual result, screenshot/logs, test data, assignee. Must be reproducible, specific, actionable.

Q43 — Severity vs Priority: Severity = technical impact (I set). Priority = business urgency (PO sets). Example: CEO's misspelled name = low severity, high priority.

Q44 — Exploratory Testing: Simultaneous design and execution. Tester learns and intuitively finds bugs. Use session-based testing (time-boxed 60-min sessions). Good for new features, supplement automated tests to find unknown unknowns.

Q45 — RTM (Requirement Traceability Matrix): Maps requirements to test cases. Columns: Requirement ID | Description | Test Case ID | Status. Ensures full coverage. I maintain in Azure DevOps.

Q46 — UAT: User Acceptance Testing — end-users validate software meets their needs. My role: prepare business-friendly test scenarios, facilitate UAT sessions, log defects, obtain sign-off.

Q47 — Integration Testing: Verifies modules work together. Approaches: Big Bang (all at once), Top-Down (use stubs), Bottom-Up (use drivers), Sandwich (hybrid). In Salesforce project, I test Salesforce ↔ backend system integration.

Q48 — Test Plan: Document outlining testing approach. Sections: Objective, Scope (in/out), Strategy, Entry/Exit Criteria, Resources, Schedule, Risks, Tools, Deliverables.

Q49 — Black/White/Grey Box: Black = no code knowledge (functional testing). White = full code knowledge (unit/code coverage). Grey = partial knowledge (integration testing, understanding architecture but testing from user perspective).

Q50 — Cannot Reproduce: 1) Verify report is complete. 2) Try same environment/data. 3) Check if test data or environment changed. 4) Try different browser. 5) Review logs. 6) Schedule joint session to watch reporter. Never close without due diligence.

Q51 — Entry/Exit Criteria: Entry = conditions to START testing (build deployed, smoke passed, test data ready). Exit = conditions to STOP (all cases executed, no open P1, coverage met, sign-off done).

Q52 — Mobile Testing: Tests on mobile devices. Checks: responsive design, touch interactions, mobile browsers, app installation, offline support, data usage, battery impact. I've done compatibility testing of Salesforce Lightning on mobile.

Q53 — Shift-Left Testing: Move QA left in timeline — involve early (requirements, design stage). Benefits: cheaper defect fixes, clearer requirements, prevent defects not just detect them.

Q54 — E2E Testing: Tests complete workflows from start to finish across integrated systems. Example: Patient appointment workflow (login → search patient → create appointment → confirm → validate billing record created in backend).

Q55 — Good Test Case: ID, descriptive title, pre-conditions, step-by-step actions (one per step), test data, expected result, actual result, status. Cover positive/negative/boundary. Independent — one test ≠ depend on another's result.

Q56 — TDM (Test Data Management): Creating, maintaining, managing test data. Critical: never use real prod data (especially healthcare). Synthetic data, data masking, reset between cycles, transactions for rollback, versioning.

Q57 — Defect Leakage: Defects escaping to production. Metric: (Bugs found in production / Total bugs found) × 100. Minimise via: strong regression coverage, risk-based testing, BDD clarity, exploratory testing, RCA on escaped defects.

Q58 — QA Metrics: Test Execution Rate, Pass/Fail Rate, Defect Density, Defect Leakage, Automation Coverage %, Automation ROI, Defect Removal Efficiency, MTTD, Defect Age, Team Velocity.

Q59 — Test Strategy vs Test Plan: Strategy = organisational/high-level (WHY, HOW broadly). Plan = project-specific (WHAT, WHEN, WHO, WHERE for THIS project).

Q60 — HP ALM/QC Experience: Used at Revamp Consulting. Features: requirements/test case management, defect tracking, test execution, integration with tools. Transitioned to Azure DevOps at Capgemini.

Q61 — Test Levels: Unit (developer), Integration (components together), System (complete app), Acceptance/UAT (business users).

Q62 — Pair Testing: Two testers on same machine, simultaneous. Benefits: finds more bugs, knowledge transfer, dev-QA alignment, speeds complex scenarios.

Q63 — QA Effort Estimation: Use story points + historical data + test case count + regression overhead + 10–15% buffer. Present during sprint planning. Flag if testing scope > sprint capacity.

Q64 — Acceptance Criteria: Conditions story must satisfy. I ensure testable using SMART (Specific, Measurable, Achievable, Relevant, Time-bound) and BDD Given-When-Then format.

Q65 — Test Summary Report: End-of-cycle report. Contents: scope, objectives, test counts (planned/executed/passed/failed), defects (total/open/closed by severity), coverage, risks, sign-off recommendation.

Q66 — Re-testing vs Regression: Re-testing = specific defect fix verification (only affected test). Regression = broad app testing to ensure fix didn't break other areas. Both after bug fix — re-test first, then regression.

Q67 — Performance Testing: Tests system under load. Types: Load (normal), Stress (beyond capacity), Spike (sudden load), Endurance/Soak (sustained load to find memory leaks), Volume (large data). Tools: JMeter, Gatling, k6. Outside my primary scope but collaborate with perf team.

Q68 — Security Testing (QA perspective): Auth & auth, SQL injection, XSS, session management, data encryption. OWASP Top 10. In healthcare, PHI protection and HIPAA compliance especially important.

Q69 — Compatibility Testing: Works across browsers (Chrome, Firefox, Edge, Safari), OS (Windows, macOS, Linux), devices (desktop, tablet, mobile), networks. For Salesforce project: Chrome, Edge primary with Firefox spot checks.

Q70 — RCA (Root Cause Analysis): Identify WHY defect occurred, not just fix it. Process: gather facts, use 5 Whys, categorise cause, define corrective actions, document, share findings. Improves future test design and processes.

Q71 — API Importance: APIs = microservices backbone. Defects here impact multiple consumers. API testing faster, more stable, less flaky than UI testing. Validates business logic independent of UI.

Q72 — JIRA vs Azure DevOps: JIRA = better project management, customisable (needs plugin for test management). Azure DevOps = all-in-one (test plans, CI/CD built-in), seamless pipeline integration. For MS stack, ADO is more cohesive.

Q73 — Test Execution Report vs Summary: Execution Report = daily/sprint operational (progress, pass/fail, blocked). Summary Report = end-cycle for stakeholders (overall assessment, risks, sign-off).

Q74 — Accessibility Testing: Ensures usable by disabled users. WCAG standards, keyboard navigation, screen reader compatible, alt text, colour contrast, focus indicators, form labels. Healthcare = Section 508 compliance required.

Q75 — Environment Issues: Immediately communicate. Escalate with specifics. Pivot to unblocked work. Track as blocked item. Work with DevOps on SLA. Post-incident: add health checks to CI.

Q76 — Testing Pyramid: Bottom = Unit tests (most, fast). Middle = Integration/API. Top = UI/E2E (fewest, slowest). Message: don't build inverted pyramid. Aim ~60–70% API/integration automation, UI automation for critical E2E only.

Q77 — Flaky Tests: Pass/fail inconsistently without code change. Causes: timing (use explicit waits not Thread.Sleep), test data conflicts (isolated data, cleanup), environment instability (retries, health checks), hardcoded waits, order-dependent tests. Quarantine and fix as tech debt.

Q78 — Cross-Browser Testing: Verify consistent behaviour across browsers. I use Selenium C# with DriverFactory supporting Chrome/Firefox/Edge. Azure Pipelines run same suite on multiple browsers in parallel.

Q79 — Test Stubs/Drivers/Mocks: Stub = dummy lower-level component (top-down). Driver = dummy upper-level component (bottom-up). Mock = programmable object recording interactions, verifies method calls with params (unit testing). Moq (C#), Mockito (Java).

Q81 What is Selenium? Explain its architecture. ▼
Basic

Selenium is an open-source browser automation framework. Components: WebDriver (direct browser communication via drivers), Grid (parallel execution), IDE (record-playback). WebDriver uses W3C protocol. Flow: Test Script → Bindings → Browser Driver → Browser → AUT → Response.

Q82 — Locator Strategies: ID (fastest), Name, CSS Selector (most reliable), XPath, LinkText, PartialLinkText, TagName, ClassName. Prefer: ID > Name > CSS > XPath.

Q83 — XPath: XML path language navigates DOM. Absolute (from root, brittle). Relative (partial path, flexible). Syntax: //tagname[@attribute='value']. Examples: //button[contains(text(),'Login')], //input[@placeholder='Enter email'], //tr[td/text()='John']/following-sibling::tr.

Q84 — CSS Selectors: More stable than XPath. Syntax: tag.class#id[attribute='value'] > descendant + sibling. Examples: input#username, .form-group > input[type='email'], [data-testid='submit-btn']. Modern, preferred.

Q85 — Implicit Wait: WebDriver waits for element to appear before throwing NoSuchElementException. Set once globally. Example: driver.Manage().Timeouts().ImplicitWait = TimeSpan.FromSeconds(10). Less control, not recommended for mixed scenarios.

Q86 — Explicit Wait (WebDriverWait): PREFERRED. Waits for specific condition before proceeding. Conditions: ElementExists, ElementVisible, ElementClickable, AlertPresent. Example: wait.Until(EC.ElementToBeClickable(By.Id("btn"))).Click(). Deterministic, reduces flakiness.

Q87 — Fluent Wait: Polls every X milliseconds, catches specific exceptions, times out after Y seconds. More flexible than explicit. Example: new WebDriverWait(driver, timeout).IgnoreExceptionTypes(typeof(NoSuchElementException)).Until(EC.ElementExists(by));

Q88 — Actions Class: Simulates user interactions (keyboard, mouse). Methods: MoveToElement, Click, DoubleClick, RightClick, SendKeys, Drag & Drop. Example: new Actions(driver).MoveToElement(elem).Click().Perform();

Q89 — Handle Multiple Windows: Use WindowHandles. Get all handles, switch to target window. Example: driver.WindowHandles[1] or driver.SwitchTo().Window(handle);

Q90 — Handle Frames/iFrames: Elements inside iFrame not accessible directly. Must switch first: driver.SwitchTo().Frame("frameId") or driver.SwitchTo().Frame(frameElement). Then interact. Switch back: driver.SwitchTo().DefaultContent().

Q91 — Handle JS Alerts: driver.SwitchTo().Alert().Accept() (OK), .Dismiss() (Cancel), .SendKeys("text") (input), .Text (get message).

Q92 — Handle Dropdowns: Use Select class if "select" tag. Example: Select s = new Select(driver.FindElement(By.Id("dropdown"))); s.SelectByValue("option1"); For custom dropdowns, click to open then click option.

Q93 — Stale Element Reference: Element no longer on DOM (page refresh, element removed). Fix: re-find element, use explicit waits, retry logic. Exception: StaleElementReferenceException.

Q94 — Shadow DOM: Encapsulated DOM not directly accessible by standard locators (common in modern web apps, Salesforce). Work-around: execute JavaScript to pierce shadow DOM, or use Playwright (better support).

Q95 — Headless Mode: Browser runs without UI. Faster, good for CI. Chrome: ChromeOptions opts = new ChromeOptions(); opts.AddArguments("--headless"); IWebDriver driver = new ChromeDriver(opts);

Q96 — Dynamic Elements: Elements rendered with IDs containing timestamps or random values. Use partial matching: //button[contains(@id, 'submit')]. Or wait for element by visibility.

Q97 — DriverFactory Pattern: Centralised WebDriver creation/teardown. Returns driver based on browser name. Manages driver lifecycle — ensures Quit() called. Reduces duplicate code.

Q98 — Base Test Class: Contains setup (Initialize driver, navigate to URL, open browser), teardown (Close driver, clean cookies), common utilities. Test classes inherit from base. DRY principle.

Q99 — Element Not Clickable: Element exists but can't click (covered by another element, disabled, hidden). Cause: overlap, dynamic content, JavaScript. Fix: explicit wait for clickable, scroll to element, JavaScript click as fallback.

Q100 — Page Refresh During Test: driver.Navigate().Refresh(). Causes stale element exceptions. Use explicit waits to re-sync after refresh.

Q221 — Why BDD: Shifts focus from "how to test" to "what should system do" (behaviour). Gherkin is readable by BA/business, not just devs. Creates living documentation. Scenarios are acceptance tests AND specs. Reduces ambiguity in requirements.

Q222 — Feature File Structure: Feature (title), Scenario (test case), Given (precondition), When (action), Then (expected). Example:

Q223 — Step Definitions: C# methods mapped to Gherkin steps. Use attributes: [Given], [When], [Then]. Example:

Q224 — Scenario Outline + Examples: Data-driven Gherkin. One scenario template, multiple examples. Example:

Runs scenario twice with different data.

Q225 — Data Tables: Pass structured data to steps. Example:

Given patient enters the following details: | Field | Value | | Email | user@example.com | | Password | secret |

Q227 — ScenarioContext for State Management: Passes data between steps. Example: ScenarioContext.Add("userId", "12345"); later: var id = ScenarioContext.Get("userId");. Avoids tight coupling.

Q228 — Tags: Mark scenarios for selective execution. Example: @smoke, @regression, @wip. Run: [Tags("smoke")]. Filters test runs. Azure Pipelines respects tags.

Q229 — Step Reusability: Write generic steps. Example: [When("user clicks {button}")] — {button} is parameter. Step definition parses parameter. Reduces step duplication.

Q230 — Regular Expressions in Steps: Match flexible step syntax. Example: [When(@"user enters (\w+) and (\w+)")] — captures email and password as groups. Powerful for parameterisation.

Q231 — SpecFlow.json Configuration: Configure SpecFlow behaviour. Specifies feature language, test runner, unit test providers. Rarely modified in typical projects.

Q232 — Reporting in SpecFlow: Generate HTML reports. SpecFlow.Plus.LivingDoc creates documentation + test results. Excellent for stakeholders — live, executable documentation.

Q233 — Loose Coupling to UI: BDD scenario describes WHAT (business behaviour). Step definitions describe HOW (using Page Objects). If UI changes, update POM + steps, not scenarios. Scenarios remain stable, reusable across platforms.

Q234 — Maintaining Feature Files: Keep scenarios concise, readable, business-focused. Avoid technical details in Gherkin. Example: avoid "Assert.AreEqual" in scenarios; keep it in step definitions.

Q351 — Scrum Roles: PO (prioritises backlog, speaks for business), SM (removes blockers, facilitates ceremonies), Dev Team (devs + QA, delivers increments). Cross-functional, self-organising team. I've worked as QA and sometimes SM.

Q352 — Sprint (2 weeks typical): Time-boxed iteration. Team commits to work, delivers increment. Goal: predictable output, fast feedback loop, incremental delivery.

Q353 — Daily Standup (15 min): What I did yesterday, what I'm doing today, blockers. Keeps team aligned. I report: tests executed, defects found, automation progress, blockers (e.g., environment down). Quick sync, not a meeting.

Q354 — Sprint Planning (4 hours for 2-week sprint): Team pulls stories from backlog, estimates effort (story points + QA hours), commits to sprint. I estimate QA effort, flag testability issues.

Q355 — Backlog Refinement (1 hour mid-sprint): Review upcoming stories, clarify requirements, break down epics. I ask: testable? Missing scenarios? Acceptance criteria clear? Prepares stories for next sprint.

Q356 — Sprint Review (1 hour, last day): Team demonstrates completed work to stakeholders. I present QA results: coverage, pass rate, defects, risks. Builds confidence in quality.

Q357 — Retrospective (1 hour, last day): "What went well? What can improve?" Generate action items. I bring data: flaky tests, automation gains, defect trends. Drive continuous improvement.

Q358 — Definition of Done (DoD): Checklist for story completion. From QA perspective: acceptance criteria tested, tests written, no P1 defects, regression updated, automation added, code reviewed. DoD = story cannot move to Done unless all checked.

Q359 — Story Points: Relative complexity estimate (1, 2, 3, 5, 8...). Not hours. Based on team's historical velocity. Easier than absolute estimation, accounts for unknowns.

Q360 — Velocity: Total story points completed per sprint. I track: if S1=40pts, S2=40pts, S3=42pts, average velocity=40pts/sprint. Use to forecast: 400pt backlog ÷ 40pts = 10 sprints to complete.

Q361 — Burndown Chart: X-axis = sprint days, Y-axis = remaining story points. Ideal line diagonal from sprint start to zero. Actual line should follow ideal. If above ideal, at-risk sprint.

Q362 — Scope Creep: Adding work mid-sprint without removing other work. Kills velocity, lowers quality. I enforce: "What are we removing?" Nothing is free. Scrum Master's job to protect.

Q363 — Three Amigos: BA + Dev + QA 15-min conversation about a story BEFORE coding. BA explains business intent, Dev asks technical questions, QA raises test scenarios, edge cases. Align on AC before development. Prevents rework.

Q364 — Acceptance Criteria (AC): Conditions story must satisfy. Testable, unambiguous. Format: "Given X, When Y, Then Z". Example: "Given patient logs in with valid credentials, When patient clicks dashboard, Then patient sees appointments list." I validate AC early.

Q365 — User Stories: Format: "As a [role], I want [action], so that [benefit]." Example: "As a nurse, I want to update patient medications quickly, so that I can focus on patient care." Business-oriented, not technical.

Q366 — Bug Triage (in Scrum): P1 bugs → fix immediately in current sprint. P2 bugs → next sprint backlog. P3/P4 → backlog (lower priority). Severity = technical impact, Priority = business urgency. PM/PO decides priority.

Q367 — Blocking Issues: In standups: "I'm blocked because [reason]." Examples: environment down, waiting on design, dependency delay. SM immediately works to unblock. Don't let team sit idle.

Q368 — Release Planning: Work backward from release date. If release in 8 weeks and velocity 40pts/sprint, can deliver 320pts of work. Prioritise backlog to fit 320pts. Anything beyond → next release.

Q369 — Kanban vs Scrum: Kanban = continuous flow, WIP limits, no sprints. Better for support/maintenance teams. Scrum = time-boxed sprints, fixed commitments. Better for product dev. I prefer Scrum for healthcare — provides rhythm, predictability.

Q370 — Lean Principles in Agile: Eliminate waste (unnecessary meetings, documentation). Amplify learning (retros, pair programming). Fast feedback loops. Deliver early and often. Empower teams.

Q384 — Error Handling in API Tests: Test negative paths: invalid input (400), missing auth (401), resource not found (404), server errors (500). Ensure graceful failure, proper error messages. Not just happy path.

Q385 — Idempotency: Repeated calls produce same result. GET always idempotent. POST typically not (creates duplicate records). Ensure APIs designed for safe retry. Example: POST with idempotency key header prevents duplicates if retried.

Q388 — Timeout Handling: API calls can hang. Set timeouts: Postman (Settings → Timeouts), RestSharp (client.Timeout = TimeSpan.FromSeconds(10)). Test: verify timeout error, circuit breaker prevents cascade failures.

Q389 — Database Validation After API Call: API updates backend. Verify via SQL query that changes persisted. Example: Call POST /patients → Create patient record. Then Query DB: SELECT * FROM patients WHERE id = ? → Verify record exists with correct data.

Q471 How do you handle Shadow DOM elements in Salesforce Lightning? ▼
Advanced

Challenge: Shadow DOM encapsulates elements. Standard Selenium selectors can't pierce shadow boundary. Salesforce Lightning uses Shadow DOM extensively.

Solutions I've Used:

  • 1. JavaScript Execution: Execute JS to pierce shadow DOM. Example: `driver.ExecuteScript("return document.querySelector('lightning-card').shadowRoot.querySelector('button');");`. Get element reference, then interact.
  • 2. Playwright (better): Playwright's `>>` combinator pierces shadow boundary. Example: `page.locator('lightning-card >> button')`. Handles shadow automatically.
  • 3. Wait for DOM Attachment: Shadow elements may render asynchronously. Use explicit wait for element visibility before interacting.
  • 4. Salesforce-Specific Locators: Some Salesforce components expose test-friendly attributes. Use `[data-testid]` where available.

Workaround for Selenium C# (current): Helper method using JS to find shadow elements. Document in framework, reusable. Future: migrate to Playwright for native shadow support.

Q472 Design a custom wait condition for a complex application scenario. ▼
Advanced

Scenario: In healthcare app, after clicking "Submit Appointment," a multi-step confirmation process runs. Page shows loading spinner, then modal with confirmation, then success toast. How do you wait for completion?

Custom Wait Condition (C# + Selenium):

public class ConfirmationCompleteWait : IWait<IWebDriver> { public T Until<T>(Func<IWebDriver, T> condition) { var spinner = driver.FindElements(By.ClassName("loading-spinner")); if (spinner.Count > 0 && spinner[0].Displayed) { // Spinner visible, wait for it to disappear var wait = new WebDriverWait(driver, TimeSpan.FromSeconds(15)); wait.Until(EC.InvisibilityOfElementLocated(By.ClassName("loading-spinner"))); } // Spinner gone, verify success toast appeared var toast = wait.Until(EC.ElementIsVisible(By.ClassName("success-toast"))); return (T)(object)toast; } } // Usage new ConfirmationCompleteWait().Until(d => d); // Waits for entire flow

Benefit: Readable, reusable, handles complex async scenarios elegantly. Better than multiple independent waits.

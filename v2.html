Q31 What is Software Testing? Why is it important? ▶
Basic

Software testing is the process of evaluating a software system to detect differences between given input and expected output, ensuring the product meets defined requirements and user expectations.

Why important: It prevents defects from reaching production (cost of fixing a bug in production is 100x more than catching it in development), ensures reliability and security, builds user trust, and reduces business risk — especially critical in healthcare systems where we work."

Q32 What is SDLC vs STLC? ▶
Basic

SDLC (Software Development Life Cycle): The complete process of planning, creating, testing, and deploying software. Phases: Requirements → System Design → Architecture → Coding → Testing → Deployment → Maintenance.

STLC (Software Testing Life Cycle): A subset of SDLC focused specifically on testing. Phases:

  • Requirement Analysis — understand what to test, identify testable requirements
  • Test Planning — define scope, approach, resources, timeline, risks
  • Test Case Design — write test cases, test data, scripts
  • Test Environment Setup — configure hardware/software prerequisites
  • Test Execution — run tests, log defects
  • Test Closure — generate reports, analyze metrics, document lessons learned
Q33 What are the different types of testing you have done? ▶
Basic

From my experience, I've performed:

  • Functional Testing — validating features against requirements
  • Regression Testing — ensuring new changes don't break existing functionality
  • Smoke Testing — quick build validation post-deployment
  • Sanity Testing — focused check of a specific bug fix or feature
  • Integration Testing — testing interactions between Salesforce modules and downstream systems
  • UAT (User Acceptance Testing) — with business stakeholders/clients
  • API Testing — using Postman to test REST endpoints
  • Exploratory Testing — unscripted, experience-based testing
  • Risk-Based Testing — prioritizing tests based on business risk
  • Mobile Testing — cross-device validation
Q34 What is the difference between Smoke Testing and Sanity Testing? ▶
Basic

Smoke Testing: A broad, shallow test of the major functionality of a new build to determine if it's stable enough for further testing. It's like asking "does the application start and does the main flow work?" If smoke fails, the build is rejected.

Sanity Testing: A narrow, deep test of a specific area — typically after a bug fix or minor change — to verify that fix works and hasn't broken related functionality. It's subset testing, not full regression.

Analogy: Smoke = checking if the car starts. Sanity = checking if the specific brake repair was done correctly.

Q35 What is Regression Testing? When do you perform it? ▶
Basic

Regression testing verifies that existing functionality hasn't been broken by new code changes, bug fixes, or configuration changes. It's performed:

  • After every code commit (via automated CI/CD pipelines in our case)
  • After bug fixes — to verify the fix and check for side effects
  • Before each sprint release
  • After environment or infrastructure changes

"In our project, we run a full automated regression suite overnight via Azure DevOps pipelines after each significant merge to the main branch."

Q36 What is Boundary Value Analysis (BVA)? ▶
Basic

BVA is a test design technique that focuses on testing values at the boundaries of valid and invalid equivalence partitions. Defects tend to cluster at boundaries.

Example: Age field accepts 18–60. BVA test values: 17 (just below min), 18 (min boundary), 19 (just above min), 59 (just below max), 60 (max boundary), 61 (just above max). So you test 6 values instead of the entire 18–60 range."

Q37 What is Equivalence Partitioning? ▶
Basic

EP divides input data into partitions (classes) where all values in a partition are expected to be treated identically by the software. You test one representative value from each partition.

Example: Same age field (18–60). Partitions: below 18 (invalid), 18–60 (valid), above 60 (invalid). Test one value from each: 10, 35, 70. This reduces the number of test cases significantly while maintaining coverage.

Q38 What is a Test Plan? What does it contain? ▶
Basic

A Test Plan is a formal document describing the scope, approach, resources, and schedule of testing activities. Contents:

  • Test objectives and scope (in scope / out of scope)
  • Test approach and strategy (manual/automated, test types)
  • Entry and exit criteria
  • Test schedule and milestones
  • Resource allocation
  • Risk register and mitigation
  • Tools and environments
  • Deliverables
Q39 What are Entry and Exit Criteria in testing? ▶
Basic

Entry Criteria — conditions that must be met before testing begins. Example: Build is deployed to test environment, all unit tests pass, test data is prepared, test environment is stable.

Exit Criteria — conditions that must be met before testing ends/release. Example: 95%+ test cases executed, 0 open P1/P2 defects, defect leakage below threshold, regression suite passes, UAT sign-off received."

Q40 What is a Defect Life Cycle? ▶
Basic

The defect life cycle (bug life cycle) tracks a defect from discovery to closure:

  • New → Defect logged by QA
  • Assigned → Assigned to developer
  • Open → Developer starts working
  • Fixed → Developer marks as fixed
  • Retest → QA verifies the fix
  • Closed → Fix verified, defect closed
  • Reopened → If fix didn't work, back to Open
  • Deferred → Postponed to future sprint
  • Rejected/Not a Bug → Working as designed
Q41 What is Severity vs Priority in defects? ▶
Basic

Severity = technical impact on the system. How badly is the functionality broken? (Critical, Major, Minor, Trivial)

Priority = business importance. How urgently must this be fixed?

Classic examples:

  • High Severity / Low Priority: Application crashes on a rarely-used admin page
  • Low Severity / High Priority: Company logo is misspelled on the homepage
  • High Severity / High Priority: Payment processing fails (both technical and business impact)
Q42 What should a good bug report contain? ▶

Q43 What is Exploratory Testing? ▶
Mid

Exploratory testing is simultaneous learning, test design, and test execution — the tester actively controls the design of the tests as those tests are performed. Unlike scripted testing, there are no pre-written test cases.

It's most valuable for: new features with incomplete requirements, finding edge-case defects that scripted tests miss, understanding complex workflows, and rapid feedback in early sprints.

"I use time-boxed session-based exploratory testing with a charter: e.g., 'Explore the patient registration workflow for unusual input combinations for 60 minutes.' I document findings in a session sheet."

Q44 What is Risk-Based Testing? ▶
Mid

Risk-based testing prioritizes test effort based on the likelihood and impact of failures. High-risk areas (business critical, complex code, frequently changed areas, areas with past defects) get more thorough testing.

"In our healthcare project, patient data handling and authorization flows are highest risk — any failure there has legal, HIPAA, and patient safety implications. So those areas get the most automation coverage, most thorough manual testing, and are included in every regression cycle."

Q45 What is the difference between Black Box, White Box, and Grey Box testing? ▶
Basic

Black Box: Testing without knowledge of internal code. Tester provides input and validates output against requirements. This is most of what QA teams do — functional, regression, UAT.

White Box: Testing with full knowledge of internal code structure. Used for unit testing, code coverage analysis. Developers typically do this.

Grey Box: Partial knowledge of internals. Tester knows the architecture and data flows but not the code. Useful for integration testing, API testing, database validation.

Q46 What is UAT (User Acceptance Testing)? ▶
Basic

UAT is the final phase of testing performed by end users or business stakeholders to validate that the software meets business requirements and is ready for production deployment. It's the last gate before go-live.

"In our healthcare project, I've coordinated UAT sessions with the client's business users — preparing UAT test scripts based on business scenarios, setting up the UAT environment, walking users through scenarios, and capturing sign-off documentation."

Q47 What is End-to-End Testing? ▶
Basic

E2E testing validates complete user workflows from start to finish — simulating real-world usage across multiple systems, databases, and interfaces.

"Example in our project: E2E test for 'patient referral' — starts with a referral being created in the Salesforce portal, triggers an integration event, updates the scheduling system, sends a notification, and the QA validates each step including UI, database state, and API responses."

Q48 How do you write a good test case? ▶
Basic

A good test case has:

  • Unique Test Case ID
  • Clear Title — describes what is being tested
  • Pre-conditions — what must be true before execution
  • Test Steps — numbered, precise, one action per step
  • Test Data — exact values to use
  • Expected Result — specific, verifiable outcome
  • Actual Result — filled post-execution
  • Pass/Fail Status

Principles: atomic (one objective per test), reusable, traceable to a requirement, independent of other tests."

Q49 What is Traceability Matrix? ▶
Mid

A Requirements Traceability Matrix (RTM) is a document that maps requirements to test cases, ensuring every requirement has test coverage and every test case traces to a requirement. It helps identify gaps in coverage and ensures nothing is missed.

Typically a table: Requirement ID → Requirement Description → Test Case IDs → Test Status

Q50 What is the difference between Verification and Validation? ▶
Basic

Verification: Are we building the product right? — checking that the software conforms to its specification. Static activities: reviews, inspections, walkthroughs. "Is the design correct?"

Validation: Are we building the right product? — checking that the final software meets user needs. Dynamic activities: testing. "Does it do what the user actually needs?"

Q51 What is Shift-Left Testing? ▶
Mid

Shift-left testing means moving testing activities earlier in the SDLC — involving QA in requirements review, design discussions, and code reviews rather than waiting for a finished build. The earlier a defect is found, the cheaper it is to fix (by 10–100x).

"In our Agile project, I participate in backlog grooming and sprint planning — this IS shift-left. By raising testability concerns before coding starts, we prevent entire categories of defects."

Q52 What is Static Testing vs Dynamic Testing? ▶
Basic

Static Testing: Testing without executing the code. Includes code reviews, peer reviews, walkthroughs, inspections of requirements, design documents, test cases.

Dynamic Testing: Testing by executing the code and observing behavior. Includes functional testing, regression, automation, performance testing."

Q53 What is Defect Leakage and Defect Seepage? ▶
Mid

Defect Leakage: Defects that escape QA testing and are found in production by end users. Formula: (Defects in Production / Total Defects) × 100. Lower is better — indicates QA effectiveness.

Defect Seepage: Defects that escape one phase of testing and are found in the next phase (e.g., from system testing to UAT). Measures phase-level effectiveness."

Q54 What is Test Coverage? ▶
Mid

Test coverage measures how much of the software is tested. Types:

  • Requirement Coverage — % of requirements with at least one test case
  • Code Coverage — % of code lines executed during testing (unit test level)
  • Functional Coverage — % of functional areas tested
  • Automation Coverage — % of test cases that are automated

"I track automation coverage as a KPI on our QA dashboard — currently at 70%+ for regression."

Q55 What is Integration Testing? What are its types? ▶
Mid

Integration testing validates the interaction between two or more integrated components/modules. Types:

  • Big Bang — all modules integrated at once and tested
  • Top-Down — testing from top-level modules downward, using stubs for lower modules
  • Bottom-Up — testing from bottom-level modules upward, using drivers for higher modules
  • Incremental/Sandwich — combination of top-down and bottom-up
Q56 What is Decision Table Testing? ▶
Mid

Decision Table Testing is a black-box testing technique used for complex business logic with multiple conditions and their combinations. A decision table maps all combinations of conditions to resulting actions.

Example: Bank loan eligibility based on Age, Income, and Credit Score:

Age Income Credit Score Loan Eligible
Yes Yes Yes Approve
Yes Yes No Reject

Decision tables ensure all logical combinations are tested — prevents gaps in coverage.

Q57 What is State Transition Testing? ▶
Mid

State Transition Testing tests a system that moves through different states based on events or conditions. You draw a state diagram, then create test cases for valid transitions, invalid transitions, and boundary states.

Example: Order lifecycle in e-commerce: Pending → Confirmed → Shipped → Delivered → Returned

Test cases cover: valid transitions (Pending→Confirmed), invalid transitions (Delivered→Pending), all state combinations, and state-specific business rules (e.g., can only cancel while Pending).

Q58 What is Negative Testing? ▶
Basic

Negative testing validates that the application handles invalid, unexpected, or out-of-range inputs gracefully without crashing or behaving unexpectedly.

Examples:

  • Entering letters in a numeric field (age field gets "abc")
  • Submitting forms with empty required fields
  • SQL injection attempts in login fields
  • Uploading oversized files
  • Negative or zero values where positive expected
  • Special characters in name fields

Expected behavior: graceful error message, no crash, no data corruption.

Q59 What is the difference between Retesting and Regression Testing? ▶
Basic

Retesting: Re-executing a specific failed test case after the bug is fixed to confirm the fix works. Narrow scope, focused on the fix.

Regression Testing: Re-executing a broader set of test cases (entire test suite or a large portion) to confirm that the bug fix didn't break existing functionality. Wider scope, prevents side effects.

Example: If a "Login" button bug is fixed, Retesting = verify the Login button works. Regression = verify Login button, Dashboard, User Profile, and all related features still work.

Q60 What are the key QA metrics you track? ▶
Mid

Key QA metrics I track:

  • Defect Density — defects per story point or per module (identifies high-risk areas)
  • Defect Leakage Rate — (Production defects / Total defects) × 100 (measures QA effectiveness)
  • Automation Coverage % — percentage of test cases that are automated
  • Test Execution Rate — % of planned tests executed in sprint
  • Test Pass/Fail Ratio — indicates product quality
  • Defect Aging — how long defects stay open (identifies workflow bottlenecks)
  • Requirements Coverage % — % of requirements with test cases
  • First-Time Pass Rate — % of tests that pass on first execution (identifies poorly designed tests)
Q61 What is Compatibility Testing? ▶
Basic

Compatibility testing verifies the application works correctly across different combinations of:

  • Browsers: Chrome, Firefox, Safari, Edge
  • Operating Systems: Windows, macOS, Linux
  • Devices: Desktop, Tablet, Mobile
  • Screen Resolutions: 1024×768, 1440×900, 1920×1080, mobile sizes
  • Versions: Different browser versions, OS versions
  • Configurations: Different RAM, Processors, network speeds

"In our healthcare project, we verify the Salesforce portal works on Chrome, Firefox, and Safari at minimum, across desktop and tablet screens."

Q62 What is Performance Testing? Explain Load, Stress, and Spike Testing. ▶
Mid

Performance testing verifies the application meets performance requirements under various load conditions.

Types:

  • Load Testing: Verify application performance under expected normal load. Example: 1000 concurrent users.
  • Stress Testing: Push application beyond limits to find breaking point. Example: 5000 concurrent users (beyond expected).
  • Spike Testing: Sudden large increase in load. Example: normal 100 users → suddenly 2000 users. Tests recovery.
  • Soak/Endurance Testing: Normal load over prolonged period (hours/days) to find memory leaks or degradation.
  • Volume Testing: Large data volume to verify performance. Example: 10 million records in database.

Tools: JMeter, k6, Gatling, LoadRunner

Q63 What is Error Guessing in Testing? ▶
Basic

Error Guessing is a test technique based on intuition and experience where the tester guesses likely error-prone areas without formal rules. It relies on domain knowledge and past defects.

Examples:

  • Testing boundary values (0, -1, 99999) based on past off-by-one errors
  • Testing after software updates where crashes commonly occur
  • Testing data type mismatches (string vs number)
  • Testing timeout scenarios in medical equipment integrations

"In healthcare QA, I use error guessing to focus on HIPAA violations, data loss scenarios, and critical system failures based on industry knowledge."

Q64 What is Ad-hoc Testing? ▶
Basic

Ad-hoc testing is informal, unscripted testing without pre-planned test cases or documentation. The tester explores the application freely, trying unexpected interactions and edge cases based on intuition.

When to use:

  • After completing formal scripted testing for exploratory coverage
  • When time is very limited and cannot document detailed tests
  • To find unusual/creative defects that scripts miss
  • Early in development for quick smoke-type verification

Pros: Fast, finds unexpected defects. Cons: Not reproducible, hard to measure coverage, dependent on tester skill.

Q65 What is Test Estimation? ▶
Mid

Test Estimation is predicting the effort (hours/days) required to complete testing activities for a project or sprint.

Estimation techniques:

  • Work Breakdown Structure (WBS): Break testing into tasks, estimate each task, sum total
  • Three-Point Estimation: Pessimistic + Optimistic + Most Likely / 3
  • Comparison/Analogy: Compare to similar past projects
  • Historical Data: Track past actuals, use for future estimates
  • Function Point Analysis: Estimate based on functionality complexity

"I use WBS + historical data comparison. I track actuals vs estimated for every sprint to continuously improve accuracy."

Q66 What is Testability? ▶
Basic

Testability refers to how easily a requirement or software component can be tested. Software with high testability is easier to test.

Factors that improve testability:

  • Clear, unambiguous requirements
  • Observable system behavior (logs, outputs, data can be verified)
  • Controllable preconditions (test data can be set up easily)
  • Defined expected results
  • Small, focused modules (easier to test isolated functionality)
  • Stable element locators (for UI automation)
  • API endpoints for test data setup (vs UI-only setup)

"During backlog grooming, I review stories for testability gaps and work with BA to clarify acceptance criteria before development starts."

Q67 What is Usability Testing? ▶
Basic

Usability testing evaluates how easy and intuitive the application is for end users. It assesses user experience — whether users can accomplish their goals with the application efficiently.

What is evaluated:

  • Navigation flow — is it logical and intuitive?
  • Clarity of labels and instructions
  • Error messages — are they helpful?
  • Accessibility — can users with disabilities use it?
  • Visual design — is it clean and professional?
  • Task completion rate — can users complete required tasks?
  • Time to complete tasks

Usability testing often involves real users, observation sessions, and sometimes surveys/interviews.

Q68 What is Accessibility Testing? ▶
Basic

Accessibility testing verifies the application is usable by people with disabilities. It ensures compliance with accessibility standards like WCAG 2.1 (Web Content Accessibility Guidelines).

What is tested:

  • Keyboard Navigation: All functions accessible via keyboard, not just mouse
  • Screen Reader Compatibility: Works with NVDA (open-source screen reader)
  • Color Contrast: Sufficient contrast for visually impaired users (WCAG AA/AAA)
  • ARIA Labels: Proper semantic HTML and ARIA attributes for assistive tech

  • Alt Text: Images have descriptive alt text
  • Form Labels: Labels associated with inputs

Q69 What is Security Testing (basics)? ▶
Mid

Security testing identifies vulnerabilities and weaknesses in the application's security posture.

Common security testing scenarios:

  • SQL Injection: Attempt to inject SQL code in input fields to bypass authentication or extract data
  • XSS (Cross-Site Scripting): Inject JavaScript code to execute in other users' browsers
  • CSRF (Cross-Site Request Forgery): Trick authenticated users into performing unwanted actions
  • Authentication Bypass: Can I login without valid credentials?
  • Authorization Flaws: Can I access data I shouldn't have access to?
  • Data Exposure: Is sensitive data (passwords, PHI) exposed in transit or at rest?
  • Insecure API Endpoints: Are API endpoints properly authenticated?

"In healthcare QA, security testing is critical — we verify HIPAA compliance, PHI access controls, audit logging, and data encryption."

Q70 What is Mutation Testing? ▶
Mid

Mutation testing deliberately introduces small changes (mutations) into the source code to check if existing test cases can detect them. It measures test suite effectiveness.

Process: Code has a bug intentionally inserted → Run test suite → If tests fail, mutation is "killed" (good); if tests pass, mutation survives (bad — tests are weak).

Example mutations: Change > to >=, change + to -, remove a statement, change true to false

Metric: Mutation Score = (Killed Mutations / Total Mutations) × 100. Higher = better test suite quality.

"If mutation testing shows 60% score, it means 40% of intentional bugs wouldn't be caught by our tests — time to improve test coverage."

Q71 How do you handle flaky tests? ▶
Advanced

Flaky tests are tests that pass sometimes and fail other times, unpredictably. They undermine confidence in automation. My approach to handling them:

Step 1: Identify the flakiness root cause

  • Timing issues (element not ready, async operations) → Add explicit waits
  • Test data dependency (dependent on order, shared state) → Isolate test data
  • Environment instability (slow network, intermittent service) → Use stubs/mocks, verify environment stability
  • Genuine application issue (race condition in code) → Report to development

Step 2: Fix the root cause, not just retry — Blind retries mask the problem and waste time.

Step 3: Quarantine and track — Mark flaky test with @flaky tag, exclude from critical paths, monitor improvement.

Q72 What is Test Debt? ▶
Mid

Test Debt (or QA Technical Debt) is accumulated untested code, missing test cases, or poor-quality automation that reduces confidence in releases. It's similar to technical debt but in the quality domain.

Examples of test debt:

  • Features released without test coverage
  • Automation scripts that are poorly written, hard to maintain
  • Flaky tests that are ignored instead of fixed
  • Test frameworks that are outdated or unsupported
  • Manual test cases that should have been automated

How I manage test debt: I reserve 10–15% of sprint capacity for test debt reduction — framework maintenance, automating high-value manual tests, fixing flaky tests, improving test documentation.

Q73 What is Configuration Testing? ▶
Basic

Configuration testing verifies that the software works correctly with different hardware, software, and system configurations that it's supposed to support.

Examples of configurations to test:

  • Different JDK versions (11, 13, 17)
  • Different database versions (SQL Server 2019 vs 2022)
  • Different application server versions (Tomcat 9 vs 10)
  • Different RAM/processor configurations
  • Different network conditions (fast vs slow internet)
  • Different character encodings (UTF-8, ASCII)

"In our Salesforce project, we verify compatibility across Salesforce API versions and Lightning configurations."

Q74 What is A/B Testing? ▶
Basic

A/B Testing compares two versions (A and B) of a feature with different user groups to determine which performs better based on key metrics.

Process: Version A (control) vs Version B (test variant) → split users into two groups → measure metrics → determine winner based on statistical significance

Metrics measured: conversion rate, click-through rate, error rate, task completion time, user satisfaction

QA role in A/B testing: Verify both versions are functionally correct, test isolation between variants (users don't see conflicting versions), verify analytics tracking is accurate, test rollback procedures.

Q75 What is Monkey Testing? ▶
Basic

Monkey Testing (also called Fuzz Testing) is random, unstructured testing where the application receives random inputs, random mouse clicks, or random keyboard presses to test robustness and error handling.

Purpose: Find unexpected crashes, memory leaks, or graceless failure scenarios.

Example: Generate random strings and submit them into every form field; click random parts of the UI in random order; press random keyboard combinations.

When useful: During early development phases, after major code changes, to stress-test error handling. Less useful for detailed functional testing.

Tools: Chaos Monkey (Netflix), OSS-Fuzz, or custom scripts

Q76 What is Gorilla Testing? ▶
Basic

Gorilla Testing is intensive testing of a specific module or feature for a prolonged time. One tester focuses deeply on one area, exhaustively testing it from multiple angles, finding edge cases and corner scenarios.

Example: Spend an entire day testing just the "Patient Registration" form — test all field combinations, boundary values, error messages, various browsers, accessibility, performance.

Purpose: Deep quality assurance in specific critical areas. It's more focused than monkey testing but exhaustive within scope.

Q77 What is Pairwise Testing (Combinatorial Testing)? ▶
Mid

Pairwise Testing (Combinatorial Testing) is a data-driven testing approach that tests combinations of input parameters. Instead of testing all possible combinations (which is exponential), it tests all pairwise combinations — ensuring every pair of parameters is covered.

Example: If testing 3 parameters with 3 values each = 27 combinations possible. Pairwise reduces this to ~9 test cases while maintaining good coverage.

Tools: PICT (Microsoft), AllPairs

"Pairwise testing is useful for configuration testing, API parameter combinations, and permission-based testing where combinations are numerous."

Q78 What is Orthogonal Array Testing? ▶
Advanced

Orthogonal Array Testing is a systematic approach for reducing test cases while maximizing coverage. It uses orthogonal arrays (mathematical constructs) to select test cases such that all interactions are covered with minimal redundancy.

Benefit: Reduces test count significantly compared to brute-force combinations.

When used: For systems with many parameters and limited test budget, configuration testing, regression testing of large feature sets.

Q79 What is Cause-Effect Testing? ▶
Mid

Cause-Effect Testing links input conditions (causes) to resulting outputs (effects). It uses a graphical notation called a "cause-effect graph" to map these relationships and derive test cases.

Process: Identify causes (input conditions) → Identify effects (outputs) → Map relationships → Create truth table → Derive test cases from truth table

This technique is useful for complex business logic and similar to Decision Table testing but graphical.

Q80 What is the difference between Validation and Verification? ▶
Basic

Verification: "Are we building the product correctly?" — Checking that software conforms to its specification and design. Static activities: reviews, inspections, walkthroughs, static analysis. Asks "Does the code match the design?"

Validation: "Are we building the right product?" — Checking that the final software meets actual user needs and business requirements. Dynamic activities: testing, execution. Asks "Does it solve the user's problem?"

Analogy: Verification = building the house according to blueprints. Validation = ensuring the house is what the homeowner actually wanted.

Q81 What is Shift-Left Testing? ▶
Mid

Shift-Left Testing means moving testing activities earlier in the Software Development Life Cycle (SDLC) — involving QA earlier, not just at the end. The "left" refers to the left side of project timeline (earlier phases).

Shift-left practices:

  • QA participates in requirements review (identify ambiguous or untestable requirements)
  • QA reviews design documents before coding starts
  • QA writes test cases during design phase, not after coding
  • QA participates in code reviews
  • Developers write unit tests before writing feature code (TDD)
  • Automate functional tests earlier in sprint

Benefit: Bugs caught earlier cost 10–100x less to fix than bugs found later.

"In our Agile project, I participate in backlog grooming and design discussions — I'm involved from day 1, not the end."

Q82 What is Static Testing? ▶
Basic

Static Testing is testing without executing the code or application. It involves analysis of software artifacts — requirements, design documents, code, test cases.

Static testing activities:

  • Code Review: Peer review of code for bugs, style, design issues
  • Walkthrough: Author guides reviewers through the code/document
  • Inspection: Formal process with defined roles, checking against criteria
  • Static Analysis Tools: Automated checking for code quality, security issues, style violations (SonarQube, Checkstyle)

Benefit: Find defects before execution, lower cost, especially good for high-risk areas.

Q83 What is Dynamic Testing? ▶
Basic

Dynamic Testing is testing by executing the code or application and observing its behavior. The application is running during the test.

Dynamic testing types:

  • Functional testing (manual or automated)
  • Performance testing
  • Security testing
  • Usability testing
  • Regression testing
  • Compatibility testing

Dynamic vs Static: Static finds defects without running code (faster, lower cost). Dynamic catches defects in execution context and system behavior.

Q84 What is Continuous Testing? ▶
Mid

Continuous Testing is executing automated tests continuously throughout the development process — on every code commit, build, or deployment. It's a key practice in DevOps/CI-CD.

How it works: Developer commits code → CI pipeline triggers → automated tests run automatically → results reported immediately → if tests fail, pipeline stops or notifies team

Benefits: Faster feedback, early defect detection, reduced risk of breaking changes, higher code quality, faster releases

"In our project, our Azure DevOps pipeline runs smoke tests automatically on every commit. Full regression runs nightly. This is continuous testing."

Q85 What is Test-Driven Development (TDD)? ▶
Mid

Test-Driven Development (TDD) is a development practice where automated unit tests are written BEFORE the feature code. The flow is: Red → Green → Refactor.

Process:

  • Red: Write a failing test for the feature not yet implemented
  • Green: Write minimal code to make the test pass
  • Refactor: Clean up code while keeping test passing
  • Repeat for next feature

Benefits: Comprehensive test coverage from the start, lower defect rate, better code design, confidence in refactoring

"TDD is a developer practice, but as a QA lead I encourage it — teams that practice TDD have significantly fewer defects."

Q86 What is Behavior-Driven Development (BDD)? ▶
Mid

Behavior-Driven Development (BDD) is an extension of TDD that specifies behavior in a business-readable format (Gherkin language). It bridges the gap between business stakeholders and technical teams.

BDD flow: Business defines expected behavior in Gherkin (Given/When/Then) → Developers implement feature to pass scenarios → Automated tests serve as acceptance criteria and regression tests

Tools: Cucumber, SpecFlow (which I use), Behave, Pytest-BDD

"BDD with SpecFlow is core to our project — BA writes scenarios, team implements, automated tests verify. One source of truth."

Q87 What is Contract Testing? ▶
Mid

Contract Testing verifies that service interactions conform to an agreed "contract" — ensuring components integrate correctly. The contract specifies expected request/response formats, status codes, and data types.

Consumer-Driven Contract Testing (CDCT): The consumer (client) defines the contract, producer (service) implements to satisfy it. Prevents producer from breaking consumer with API changes.

Tools: Pact (most popular), Spring Cloud Contract

Example: Frontend expects Patient API to return {id, name, dob, insuranceId}. If backend adds/removes fields, contract test fails immediately — catching integration issues early.

Q88 What is Visual Regression Testing? ▶
Mid

Visual Regression Testing automatically checks if the UI appearance (layout, colors, fonts, sizes) has changed unexpectedly. It compares screenshots of the UI against baseline images.

How it works: Take screenshot → Compare against baseline image → Flag visual differences

Tools: Percy, Applitools, BackstopJS, Puppeteer

Use case: After a CSS framework update or designer changes, run visual tests to ensure no unintended layout breaks across all pages/components.

Limitation: Need to review and approve changes manually — not fully automated, requires judgment.

Q89 What is Baseline Testing? ▶
Basic

Baseline Testing establishes a reference point (baseline) for comparing performance, functionality, or other metrics in future tests. The baseline is the expected/acceptable performance level.

Example: Website loads in 1.5 seconds on first visit (baseline). Future performance tests compare against this baseline — if new code causes 3-second load time, it exceeds baseline and is flagged.

Use cases: Performance baselines, automation coverage baselines, defect rate baselines, test execution time baselines

"I establish baseline metrics at project start, then track trends — we can see if quality is improving or degrading over sprints."

Q90 What is Incremental Testing? ▶
Mid

Incremental Testing tests software components incrementally as they are developed — not waiting for the entire system to be built. New components are integrated with previously tested components and tested together.

Example: Component A is tested → Component A+B tested together → Component A+B+C tested together (not testing C in isolation)

Benefits: Detect integration issues early, easier to pinpoint defects, faster feedback

Contrast with Big Bang Integration: Big Bang = develop everything, then test all integrated (risky, defects hard to locate). Incremental = test as you go (preferred).

Q102 What is the Page Object Model (POM)? Why do you use it? ▶
MidC# Selenium

POM is a design pattern that creates an object repository for web UI elements. Each page of the application has a corresponding Page class that contains: (1) element locators, (2) action methods for that page. Test classes only call page methods — they don't contain locators.

Q103 What are the different types of locators in Selenium? ▶

Q105 What are Implicit Wait, Explicit Wait, and Fluent Wait in Selenium? ▶
MidC# Selenium

Implicit Wait: Sets a global wait for all FindElement calls. Driver polls the DOM for the specified time before throwing NoSuchElementException. Applied once, affects all.

Explicit Wait (WebDriverWait): Waits for a specific condition to be true before proceeding. More precise — applied per element or condition.

Fluent Wait: Like explicit wait but with configurable polling frequency and exception ignoring.

// Implicit Wait driver.Manage().Timeouts().ImplicitWait = TimeSpan.FromSeconds(10); // Explicit Wait — PREFERRED WebDriverWait wait = new WebDriverWait(driver, TimeSpan.FromSeconds(15)); IWebElement el = wait.Until(ExpectedConditions.ElementToBeClickable(By.Id("btn"))); // Fluent Wait DefaultWait<IWebDriver> fluentWait = new DefaultWait<IWebDriver>(driver); fluentWait.Timeout = TimeSpan.FromSeconds(20); fluentWait.PollingInterval = TimeSpan.FromMilliseconds(500); fluentWait.IgnoreExceptionTypes(typeof(NoSuchElementException));
⚠️Never mix Implicit and Explicit waits — it causes unpredictable wait times. Use explicit waits throughout your framework.

Q161 What is BDD (Behavior Driven Development)? ▶
BasicBDD

BDD is a software development approach that encourages collaboration between developers, QA, and non-technical business stakeholders. Tests are written in plain English-like language (Gherkin) that everyone can understand.

Core idea: Define system behavior from the user's perspective using scenarios that describe GIVEN a context, WHEN an action occurs, THEN the expected outcome happens.

Benefits: Shared understanding of requirements, living documentation, tests as acceptance criteria, reduced ambiguity.

"At Capgemini, I write SpecFlow feature files with BAs to ensure our automated tests directly reflect business requirements. When a business analyst reviews a Gherkin scenario and confirms it matches their understanding, we've eliminated a whole class of bugs."

Q162 What is SpecFlow? How does it differ from Cucumber? ▶
BasicBDD

SpecFlow is the .NET implementation of BDD/Cucumber. It uses the same Gherkin syntax for feature files but integrates with .NET/C# test frameworks (NUnit, MSTest, xUnit) and Visual Studio tooling.

Cucumber is the original BDD framework primarily for Java, Ruby, and JavaScript.

They're conceptually identical — same Gherkin syntax, same Given/When/Then structure — but SpecFlow is the right choice for C# projects. SpecFlow+ (now SpecFlow.Plus.LivingDoc) adds living documentation features.

Q171 — Step reuse across feature files: Steps are automatically reused — any step matching the regex pattern in any feature file will use the same step definition. Design step definitions to be generic and reusable.

Q172 — SpecFlow Living Documentation: SpecFlow.Plus.LivingDoc generates HTML living documentation from feature files — shows which scenarios pass/fail. Great for stakeholder communication — a non-technical business analyst can see exactly what's tested and its current status.

Q173 — BDD Anti-patterns to avoid: (1) UI-centric steps (Given I click on the 'Submit' button) — steps should describe WHAT, not HOW. (2) One step per method always — combine related actions. (3) Sharing mutable state incorrectly — use DI properly. (4) Too many scenarios — pick representative cases, not every data combination.

Q174 — Gherkin best practices: Write from the user's perspective. Steps should be declarative (what) not imperative (how). Use Background for shared setup. Use Scenario Outline for data-driven cases. Keep features focused — max 10 scenarios per feature file.

Q175 — Context Injection (DI in SpecFlow): SpecFlow automatically resolves constructor dependencies using its built-in BoDi IoC container. Shared objects (driver, pages, state) are registered and injected — ensuring the same instance is shared across step definition classes within a scenario.

Q274 What is a User Story? How does QA ensure it's testable? ▶
Basic

User Story format: "As a [user type], I want [action], so that [benefit]"

Example: "As a nurse, I want to quickly update a patient's medication list, so that I can ensure accurate records without navigating multiple pages."

  • Are edge cases considered? (empty list, max limits, special characters?)
  • Is there a clear expected result? (what success looks like)

If a story fails these checks, I raise concerns in backlog grooming before it enters the sprint."

Q275 What is Velocity in Agile? How do you use it for forecasting? ▶
Mid

Velocity = total story points completed in a sprint. It's a measure of team capacity.

Example: Sprint 1 completed 40 points, Sprint 2 completed 38 points, Sprint 3 completed 42 points. Average velocity = 40 points/sprint.

How I use velocity: If backlog has 400 points remaining and our velocity is 40 pts/sprint, we forecast 10 sprints to complete all work. This informs release planning — when can we ship?

Velocity trends: If velocity is declining (35 → 30 → 25), it signals problems — team is slower, more blockers, or scope is expanding. I investigate: is QA creating a bottleneck? Are defects taking longer to fix? Is environment instability slowing us?

Q276 What is the "Three Amigos" approach? How does it prevent defects? ▶
Mid

Three Amigos = Business Analyst + Developer + QA having a conversation about a User Story BEFORE coding starts.

My experience with Three Amigos: In backlog grooming, we pull a story and discuss: "What does this feature do? What could go wrong?" BA explains business intent, Developer asks about technical constraints, I ask about edge cases and test scenarios.

Example conversation:

  • BA: "Users should be able to upload patient records as PDF files."
  • Dev: "We support files up to 50MB via S3."
  • QA: "What happens if someone uploads 55MB? A non-PDF? An empty file? What's the success message?"
  • All three align on acceptance criteria before a single line of code is written.

Result: Fewer surprises. Developers don't code something that QA can't test or BA doesn't want. Defects that would've been found late are prevented early."

Q277 How do you handle scope creep in a sprint? ▶
Mid

Scope creep = adding new stories or work mid-sprint without removing other work. Kills velocity and creates quality issues.

My approach: When someone wants to add a story mid-sprint, I ask: "What are we removing?" Nothing is free. If a stakeholder insists on adding a critical bug fix, we identify a low-priority story to move out. The Scrum Master and I enforce this — it protects team capacity and QA ability to thoroughly test.

Example scenario: Friday before sprint end, a P1 production bug is reported. We don't just add it to the sprint. Instead: (1) Developers fix it immediately, (2) I test the fix the same day, (3) We move a lower-priority story out to maintain sprint commitment. This prevents burnout and maintains quality standards."

Q278 What is Technical Debt in QA? How do you manage it? ▶
Mid

Test/QA Technical Debt = shortcuts taken that haunt you later. Examples:

  • Automation scripts written poorly — hard to maintain, break frequently
  • Test cases with missing coverage — you know features aren't tested but haven't written tests
  • Flaky tests that everyone ignores — reduce confidence in automation
  • Manual tests that should be automated — waste time every sprint
  • Framework using outdated libraries — security vulnerabilities, compatibility issues

How I manage it: I advocate for reserving 10–15% of sprint capacity for "QA refinement" — framework maintenance, fixing flaky tests, updating deprecated dependencies, automating high-value manual tests. This is non-negotiable. Without it, debt snowballs and becomes unmanageable.

My quarterly QA debt assessment: I audit: automation framework health, test coverage gaps, slow test areas, maintenance burden. I present findings to leadership with impact: "We have 8 flaky tests affecting team trust. Fixing them is 1 sprint investment that improves team velocity long-term."

Q411 What is Playwright? How does it differ from Selenium? ▶
MidPlaywright

Playwright is a modern end-to-end testing framework by Microsoft that supports Chromium, Firefox, and WebKit browsers. Key differences from Selenium:

  • Auto-wait — Playwright automatically waits for elements to be actionable before interacting (no explicit waits needed for most cases)
  • Network interception — built-in API mocking, request/response modification
  • Shadow DOM — handled natively without JavaScript hacks
  • Parallel execution — workers run tests in parallel by default
  • Trace viewer — built-in time-travel debugging with screenshots at each step
  • Cross-browser with same API — Chromium, Firefox, WebKit (Safari engine) from one codebase
  • No driver management — no ChromeDriver/GeckoDriver needed

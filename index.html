Q26 What is the difference between verification and validation? ▼
BasicIntermediate
✦ Model Answer

Verification — "Are we building the product right?" It's a static process checking that the software conforms to specifications. Examples: code reviews, requirement reviews, design walkthroughs. No execution required.

Validation — "Are we building the right product?" It's a dynamic process — actually running the software to ensure it meets the user's needs. Examples: functional testing, UAT, system testing. Execution required.

Scenario: If a requirement says "login button must be blue" and the developer builds a blue button that opens the wrong page — verification passes (blue button exists) but validation fails (wrong page). Both are necessary.

Q31 What is the Software Testing Life Cycle (STLC)? ▼
Basic
✦ Model Answer

STLC is the sequence of activities performed during the testing process. The phases are:

  • 1. Requirement Analysis: Review requirements, identify testable requirements, clarify ambiguities. Output: Requirements Traceability Matrix (RTM)
  • 2. Test Planning: Define scope, strategy, resources, timeline, risks. Output: Test Plan document
  • 3. Test Case Design: Write detailed test cases and test scripts. Output: Test cases, test data
  • 4. Test Environment Setup: Prepare test environment, install software, configure data
  • 5. Test Execution: Execute test cases, log defects, update status
  • 6. Test Closure: Evaluate exit criteria, prepare test summary report, lessons learned
Q32 What are the 7 principles of software testing? ▼
Basic
✦ Model Answer
  • 1. Testing shows presence of defects — it reduces the probability of undiscovered defects but cannot prove absence
  • 2. Exhaustive testing is impossible — test with risk-based and priority-driven approaches
  • 3. Early testing saves money — defects found in requirements are cheaper to fix than in production
  • 4. Defect clustering — most defects are found in a small number of modules (80/20 rule)
  • 5. Pesticide paradox — repeating the same tests finds fewer new bugs; tests must evolve
  • 6. Testing is context dependent — safety-critical testing differs from e-commerce testing
  • 7. Absence of errors fallacy — a bug-free system that doesn't meet user needs still fails
Q33 Explain the defect lifecycle with all statuses. ▼
Basic
✦ Model Answer
  • New: Defect logged by tester
  • Assigned: Assigned to developer
  • Open: Developer starts investigating
  • Fixed: Developer has fixed and moved to QA
  • Retest: Tester retests the fix
  • Verified: Fix confirmed by tester
  • Closed: Defect resolved and closed
  • Rejected: Developer says it's not a bug
  • Deferred: Will be fixed in a later release
  • Reopened: Tester finds the fix didn't work
  • Duplicate: Same defect already reported
Q34 What is Equivalence Partitioning? Give a real example. ▼
Basic
✦ Model Answer

Equivalence Partitioning (EP) is a black-box test design technique where input data is divided into valid and invalid partitions. The assumption is that all values in a partition will be treated the same way, so testing one value per partition is sufficient.

Real Example — Age field for a health insurance form (18–65 years allowed):
• Valid partition: 18–65 → test with 40 (one value covers whole partition)
• Invalid partition 1: below 18 → test with 10
• Invalid partition 2: above 65 → test with 70
• Invalid partition 3: non-numeric → test with "abc"
Q35 What is Boundary Value Analysis? How is it different from EP? ▼
Basic
✦ Model Answer

Boundary Value Analysis (BVA) tests values at the boundaries of valid and invalid partitions — because defects tend to hide at boundaries.

BVA is a complement to EP, not a replacement. EP tests one value per partition; BVA tests boundary values specifically.

Age 18–65 example using BVA:
• 17 (just below lower boundary) — invalid
• 18 (lower boundary) — valid
• 19 (just above lower boundary) — valid
• 64 (just below upper boundary) — valid
• 65 (upper boundary) — valid
• 66 (just above upper boundary) — invalid
Q36 What is a Decision Table and when do you use it? ▼
BasicIntermediate
✦ Model Answer

A Decision Table is a test design technique used when multiple conditions/rules must be combined to determine an action. It ensures all combinations of conditions are tested.

Example — Loan approval logic:
Conditions: Credit Score Good (Y/N) | Income Sufficient (Y/N)

| Credit Score | Income | Result |
| Y | Y | Approved |
| Y | N | Rejected |
| N | Y | Review |
| N | N | Rejected |

Use when: multiple business rules interact, complex conditional logic exists.
Q37 What is State Transition Testing? ▼
Intermediate
✦ Model Answer

State Transition Testing is used when an application behaves differently based on its current state. You model all possible states and the transitions between them.

Healthcare example — Patient Appointment:
States: Requested → Confirmed → In Progress → Completed / Cancelled
Test that valid transitions work and invalid ones (e.g., jumping from Requested to Completed without Confirmed) are blocked.
Q38 Difference between Functional and Non-Functional Testing? ▼
Basic
✦ Model Answer

Functional Testing — tests WHAT the system does. Validates business requirements and application features. Examples: login works, data is saved correctly, search returns right results.

Non-Functional Testing — tests HOW the system performs. Examples:

  • Performance Testing: How fast does the app respond under load?
  • Security Testing: Is data protected from unauthorised access?
  • Usability Testing: Is the UI intuitive for users?
  • Compatibility Testing: Does it work on all supported browsers/OS?
  • Accessibility Testing: Is it usable by people with disabilities?
Q39 What is Regression Testing and when is it performed? ▼
Basic
✦ Model Answer

Regression Testing verifies that new code changes have not broken existing functionality. It is performed:

  • After every bug fix (to ensure the fix didn't introduce new issues)
  • After adding new features
  • After code refactoring
  • Before every production release
  • After Salesforce platform upgrades in my current project

In my projects, I've automated the core regression suite using Selenium C# so it can be triggered automatically via Azure DevOps pipelines after each deployment — reducing manual regression effort by 65%+.

Q40 What is the difference between Smoke Testing and Sanity Testing? ▼
Basic
✦ Model Answer

Smoke Testing: A surface-level test performed on a new build to verify it is stable enough for further testing. It covers basic functionality — can we log in? Can we navigate key pages? If smoke fails, the build is rejected.

Sanity Testing: A focused, narrow test performed after a bug fix or minor change to verify that the specific fix works correctly — without running the full regression. It's a quick check before a re-release.

Memory aid: Smoke = "Does the app smoke (catch fire)?" — broad check. Sanity = "Is this fix sane/sensible?" — narrow check.
Q41 What is Risk-Based Testing? How do you prioritise test cases? ▼
Intermediate
✦ Model Answer

Risk-Based Testing (RBT) prioritises testing based on the likelihood and impact of failure. The goal is to allocate testing effort to areas of highest risk.

My prioritisation approach:

  • Business impact: Core revenue/business flows (e.g., patient admission in healthcare) → highest priority
  • Defect history: Modules with most historical defects → higher coverage
  • Complexity: Complex integrations and data transformations → thorough testing
  • Change impact: Recently modified code → regression focus
  • Frequency of use: Features used by 80% of users → always in scope

I use heat maps and defect trend data from Azure DevOps to make this data-driven rather than subjective.

Q43 What is the difference between Severity and Priority? ▼
Basic
✦ Model Answer

Severity — the technical impact of the defect on the system. Set by the QA engineer. (High = system crash; Low = UI typo)

Priority — the business urgency of fixing the defect. Set by the Product Owner / Business. (High = must fix before release; Low = can wait)

High Severity, Low Priority: An error on the company's "About Us" page that throws an exception — system-level issue but no business impact.

Low Severity, High Priority: The company CEO's name is misspelled on the home page — cosmetic but must fix before the press release tomorrow.
Q44 What is Exploratory Testing and when do you use it? ▼
Intermediate
✦ Model Answer

Exploratory Testing is simultaneous test design and execution — the tester learns about the system while testing it, using experience and intuition to find defects that scripted tests miss.

When I use it:

  • When a new feature lands and test cases aren't fully written yet
  • After a major release to "tour" the application and find unexpected interactions
  • When time is limited and I need to find the most impactful bugs fast
  • To supplement automated testing — automation finds known unknowns; exploration finds unknown unknowns

I use session-based testing — time-boxed (e.g., 60-minute) sessions with a defined charter to keep exploration focused.

Q45 What is a Requirement Traceability Matrix (RTM)? ▼
Basic
✦ Model Answer

An RTM maps each business requirement to one or more test cases, ensuring full test coverage. It has columns like: Requirement ID | Requirement Description | Test Case ID | Test Case Status.

It helps answer: "Do we have tests for every requirement?" and "Which requirements are at risk?" I maintain RTMs in Azure DevOps linking user stories to test cases, giving stakeholders traceability from business need all the way to test result.

Q47 What is Integration Testing and what are the approaches? ▼
Intermediate
✦ Model Answer

Integration Testing verifies that different modules/components work correctly together.

Approaches:

  • Big Bang: All components integrated at once and tested — simple but hard to isolate failures
  • Top-Down: Start from top-level modules, use stubs for lower-level — follows user flow
  • Bottom-Up: Start from lower-level modules, use drivers for upper — good for testing core logic first
  • Sandwich (Hybrid): Combination of top-down and bottom-up

In my Salesforce project, integration testing validates data flows between Salesforce modules and backend healthcare systems.

Q49 What is the difference between Black Box, White Box, and Grey Box Testing? ▼
Basic
✦ Model Answer

Black Box Testing: Tester has no knowledge of internal code. Tests based on requirements and expected behaviour. Examples: functional testing, UAT.

White Box Testing: Tester knows the internal code/logic. Tests code paths, branches, conditions. Examples: unit testing, code coverage analysis.

Grey Box Testing: Partial knowledge of internals. Tester uses knowledge of architecture/data flow to design better tests while still testing from a user perspective. Examples: integration testing with DB access, API testing.

As a QA Lead I primarily do black and grey box testing — I have enough system knowledge to design more targeted tests, but I primarily validate from the user/business perspective.

Q50 How do you handle a "Cannot Reproduce" defect? ▼
Intermediate
✦ Model Answer
  • Verify the defect was reported with complete steps to reproduce, test data, and environment details
  • Try to reproduce it myself in the same environment and data state
  • Check if the test data or environment has changed since the defect was reported
  • Try reproducing in a different browser or machine
  • Review application logs to find evidence the error occurred
  • If still can't reproduce, mark as "Cannot Reproduce" and schedule a joint session with the reporter to watch them reproduce it live
  • Never close a defect as invalid without due diligence — it may reappear in production

Q55 How do you write effective test cases? ▼
Basic
✦ Model Answer

A good test case is clear, reusable, and maintainable. My checklist:

  • Unique Test Case ID — for traceability
  • Descriptive title — states what's being tested and expected behaviour
  • Pre-conditions: System state required before execution
  • Step-by-step actions: Precise, unambiguous, one action per step
  • Test Data: Specific data values to use
  • Expected Result: Exact, verifiable outcome
  • Actual Result: Filled during execution
  • Pass/Fail Status
  • Cover positive, negative, and boundary scenarios
  • Keep them independent — one test case should not depend on another's result

Q57 What is Defect Leakage and how do you minimise it? ▼
Intermediate
✦ Model Answer

Defect Leakage = defects that are not caught during testing and escape to production. It's a key QA metric I track.

Defect Leakage % = (Bugs found in production / Total bugs found) × 100

How I minimise it:

  • Strong regression coverage — automated suite runs on every deployment
  • Risk-based testing to focus on high-probability failure areas
  • BDD acceptance criteria reviewed before development starts
  • Exploratory testing sessions before each release
  • Root cause analysis on every escaped defect to prevent recurrence

Q59 What is the difference between Test Strategy and Test Plan? ▼
Intermediate
✦ Model Answer

Q63 How do you estimate testing effort for a sprint? ▼

Q64 What is Acceptance Criteria and how do you ensure it's testable? ▼
Intermediate
✦ Model Answer

Q66 What is the difference between Re-testing and Regression Testing? ▼
Basic
✦ Model Answer

Re-testing (Defect Verification): Tests the specific defect that was fixed to confirm it is now resolved. Only the test case(s) related to that defect are re-executed.

Regression Testing: Tests the broader application to ensure the bug fix has not inadvertently broken other working functionality. Covers areas potentially impacted by the code change.

Both are performed after a bug fix — re-testing first, then regression.

Q70 What is Root Cause Analysis in defect management? ▼
Intermediate
✦ Model Answer

Root Cause Analysis (RCA) is the process of identifying the fundamental reason why a defect occurred — not just fixing the symptom. I use it especially for high-severity production defects.

My RCA process:

  • Gather facts: when, where, how did it occur?
  • Use the 5 Whys technique — keep asking "Why?" until the root cause is found
  • Categorise root cause: requirements gap? Code logic error? Missing test case? Environment issue?
  • Define corrective actions: fix the process, not just the defect
  • Document and share — RCA findings inform future test case additions and process improvements

Q76 What is the testing pyramid? ▼
Intermediate
✦ Model Answer

The Testing Pyramid (Mike Cohn) describes the ideal distribution of test types:

  • Bottom (most) — Unit Tests: Fast, cheap, isolated. Developers write these. Highest volume.
  • Middle — Integration/API Tests: Test component interactions. Medium volume.
  • Top (least) — UI/E2E Tests: Slowest, most expensive, most fragile. Selenium tests live here. Lowest volume.

The message: don't build an "ice cream cone" (inverted pyramid) — too many slow E2E tests and not enough fast unit tests. In my projects I aim for ~60-70% automation at API/integration level and use UI automation for critical E2E flows only.

Q77 What is meant by "Flaky Tests" and how do you handle them? ▼
Intermediate
✦ Model Answer

Flaky tests are automated tests that pass and fail inconsistently without any code change — they erode trust in the automation suite.

Common causes and my fixes:

  • Timing/synchronisation issues: Replace Thread.Sleep with explicit waits (WebDriverWait)
  • Test data conflicts: Ensure tests use isolated, pre-seeded data and clean up after themselves
  • Environment instability: Retry mechanism + environment health checks
  • Hardcoded waits: Remove all Thread.Sleep — use dynamic waits
  • Order-dependent tests: Make each test fully independent

I tag flaky tests as "quarantined" and track them in a tech-debt backlog to fix systematically.

Q79 What is a Test Stub vs a Test Driver vs a Mock? ▼
Advanced
✦ Model Answer
  • Stub: A dummy module that replaces a lower-level component not yet developed. It provides pre-programmed responses. Used in top-down integration testing.
  • Driver: A dummy module that replaces an upper-level component to call the module being tested. Used in bottom-up integration testing.
  • Mock: A programmable object that records interactions and can verify that specific methods were called with specific parameters. More powerful than stubs — used in unit testing. Tools: Moq (C#), Mockito (Java).

Q81 What is Selenium? Explain its architecture. ▼
Basic
✦ Model Answer

Selenium is an open-source framework for automating web browsers. It consists of:

  • Selenium WebDriver: Directly communicates with the browser via browser drivers (ChromeDriver, GeckoDriver). Most widely used component.
  • Selenium Grid: Enables parallel test execution on multiple machines and browsers
  • Selenium IDE: Browser extension for record-and-playback (prototype/demo use)

Architecture flow: Test Script → Selenium WebDriver (Language Bindings) → Browser Driver (ChromeDriver) → Browser (Chrome) → Application under test → Response back

WebDriver uses the W3C WebDriver protocol (replaced the JSON Wire Protocol). Commands are sent as HTTP requests to the browser driver.

Q26 What is the difference between verification and validation? ▼
BasicIntermediate
✦ Model Answer

Verification — "Are we building the product right?" It's a static process checking that the software conforms to specifications. Examples: code reviews, requirement reviews, design walkthroughs. No execution required.

Validation — "Are we building the right product?" It's a dynamic process — actually running the software to ensure it meets the user's needs. Examples: functional testing, UAT, system testing. Execution required.

Scenario: If a requirement says "login button must be blue" and the developer builds a blue button that opens the wrong page — verification passes (blue button exists) but validation fails (wrong page). Both are necessary.

Q31 Explain the Software Testing Life Cycle (STLC). ▼
BasicIntermediate
✦ Model Answer

STLC consists of specific phases to ensure quality goals are met:

  • Requirement Analysis: QA interacts with stakeholders to understand requirements. Are they testable?
  • Test Planning: Preparation of Test Strategy/Plan, resource allocation, and effort estimation.
  • Test Case Development: Creating detailed test cases, test data, and RTM.
  • Environment Setup: Configuring the test environment (servers, databases, devices).
  • Test Execution: Running the tests and logging defects.
  • Test Closure: Analysis of results, creating Test Summary Reports, and lesson-learned sessions.
Q32 What are the 7 principles of software testing? ▼
Basic
✦ Model Answer
  • 1. Testing shows presence of defects — it reduces the probability of undiscovered defects but cannot prove absence
  • 2. Exhaustive testing is impossible — test with risk-based and priority-driven approaches
  • 3. Early testing saves money — defects found in requirements are cheaper to fix than in production
  • 4. Defect clustering — most defects are found in a small number of modules (80/20 rule)
  • 5. Pesticide paradox — repeating the same tests finds fewer new bugs; tests must evolve
  • 6. Testing is context dependent — safety-critical testing differs from e-commerce testing
  • 7. Absence of errors fallacy — a bug-free system that doesn't meet user needs still fails
Q33 Explain the defect lifecycle with all statuses. ▼
Basic

Defect statuses: New → Assigned → Open → Fixed → Retest → Verified → Closed. Also: Rejected (not a bug), Deferred (later release), Reopened (fix didn't work), Duplicate.

Q36 Explain equivalence partitioning and boundary value analysis with examples. ▼
Basic
✦ Model Answer

Equivalence Partitioning (EP): Divides input data into logical groups (partitions) where each value in a group is expected to be processed similarly. We test only one value from each partition.

Boundary Value Analysis (BVA): Focuses on testing at the boundaries of these partitions, where bugs are most likely to occur (minimum, maximum, just inside, and just outside).

Example: A field accepts age between 18 and 60.
EP: Valid (18-60), Invalid (>60), Invalid (<18).
BVA: 17, 18, 19 and 59, 60, 61.
Q37 What is traceability matrix and why is it important? ▼
BasicIntermediate
✦ Model Answer

A Requirement Traceability Matrix (RTM) is a document (usually a table) that maps requirements to the test cases designed to verify them. It ensures 100% test coverage.

Importance:

  • It proves that every requirement has at least one test case assigned to it.
  • It helps in Impact Analysis: if a requirement changes, I know exactly which test cases need updating.
  • It serves as proof for stakeholders and auditors that all business needs have been verified.
Q38 How do you perform root cause analysis of production defects? ▼
AdvancedProcess
✦ Model Answer

I follow a structured "5 Whys" approach or a fishbone diagram:

  1. Gather Data: Collect logs, screenshots, environment details, and steps to reproduce.
  2. Identify the Gap: Determine why the bug wasn't caught in QA. Was it a missing test case? An environment difference? Or a data-specific issue?
  3. Root Cause: Dig deep. Example: Why did the server crash? (Memory leak). Why? (Unclosed DB connection). Why? (Missing finally block in code).
  4. Corrective Action: Fix the bug and add a regression test.
  5. Preventive Action: Update the test strategy or add code review checklists to ensure similar issues don't recur.
Q39 Difference between verification and validation. ▼
Basic
✦ Model Answer

Verification: "Are we building the product right?" It is a static process involving document reviews, walkthroughs, and inspections. It happens before the code is executed.

Validation: "Are we building the right product?" It is a dynamic process where we execute the code to see if it meets user requirements. Examples include functional testing and UAT.

Q40 Explain decision table testing technique. ▼
Intermediate
✦ Model Answer

Decision table testing is used for features containing complex business logic with multiple true/false conditions. We create a table listing all possible combinations of inputs (Conditions) and the resulting actions (Outcomes).

Example: A banking app allows transfer if (1) Account is active and (2) Balance > Transfer Amount. Each combination (Y/Y, Y/N, N/Y, N/N) becomes a test scenario.
Q41 What is state transition testing? ▼
Intermediate
✦ Model Answer

State transition testing is used when the application behavior depends on "states" and certain events that trigger transitions between them. We test all valid transitions and ensure invalid transitions are blocked.

Example: An ATM. States: Idle → Card Inserted → Pin Entered → Transaction Completed. We test that you can't go from 'Idle' to 'Transaction Completed' without entering a PIN.

Q42 How do you test a REST API manually? ▼
IntermediateProcess
✦ Model Answer

I use tools like Postman or Swagger:

  • Check Status Codes: Ensure 200/201 for success, 400 for bad requests, 401 for unauthorized, etc.
  • Verify JSON/XML Payload: Ensure the response body matches the expected schema and data types.
  • Header Validation: Check Content-Type, Authorization, and Cache-Control headers.
  • Verify Data Integrity: If a POST call creates a record, I run a GET call or check the Database manually to ensure it exists.
  • Negative Testing: Send invalid payloads or missing mandatory fields to verify error messages.
Q43 What is exploratory testing? How do you document it? ▼
Intermediate
✦ Model Answer

Exploratory testing is simultaneous learning, test design, and execution. It's not unscripted but rather relies on the tester's intuition and experience to find bugs that predefined scripts might miss.

Documentation: I use Test Charters (specifying the goal, duration, and scope) and document "Session Reports" detailing what was tested, bugs found, and areas needing more focus.

Q44 Difference between smoke, sanity, and regression testing. ▼
Basic
✦ Model Answer
  • Smoke: Broad but shallow testing to ensure the build is stable enough for further testing. It tests the "critical path."
  • Sanity: Narrow but deep testing performed after a bug fix or minor change to verify that the specific problem is resolved.
  • Regression: Extensive testing to ensure that new changes or bug fixes haven't broken existing, untouched features.
Q45 How do you handle non-reproducible bugs? ▼
Intermediate
✦ Model Answer
  • Capture details: I immediately record the environment, time, user role, and browser version.
  • Review Logs: Check server-side and browser console logs for hidden errors.
  • Try different data: Sometimes bugs are triggered only by specific data patterns.
  • Peer session: Work with the developer to trigger it on their machine.
  • "Monitor" status: If it still can't be reproduced, I log it with a "Cannot Reproduce" status but keep it open for a few days to see if others encounter it.
Q46 What is defect leakage and defect age? ▼
Intermediate
✦ Model Answer

Defect Leakage: When a bug is not caught by the QA team but is found by the end-user in production. Formula: (Bugs in Prod / Total Bugs) * 100.

Defect Age: The time difference between when a bug was reported and when it was closed. It helps measure the efficiency of the dev-QA cycle.

Q47 Explain pairwise testing with an example. ▼
Advanced
✦ Model Answer

Pairwise (All-Pairs) testing is a combinatorial technique that assumes most bugs are caused by the interaction of at most two parameters. It significantly reduces the number of test cases while maintaining strong coverage.

Example: Testing a printer with 3 params: OS (Win, Mac), Connection (USB, Wifi), Paper (Plain, Photo).
Instead of 2*2*2 = 8 cases, pairwise selects combinations (e.g., Win/USB, Win/Wifi, Mac/USB, Mac/Wifi) ensuring every pair is tested.
Q48 How do you test microservices architecture manually? ▼
AdvancedProcess
✦ Model Answer
  • Component Testing: Test each service in isolation using Postman.
  • Contract Testing: Ensure Service A sends the exact format Service B expects.
  • End-to-End flows: Test a user journey that spans multiple services (e.g., Auth Service → Patient Service → Billing Service).
  • Failover testing: Kill one service and see if the application handles the failure gracefully (e.g., showing a friendly error instead of crashing).
  • Log tracing: Use Correlation IDs to track a single request across multiple service logs.

  • Defining Strategy: Deciding the "how" (tools, scope, environments).
  • Resource Management: Assigning tasks based on team strengths.
  • Risk Assessment: Identifying potential blockers (e.g., environment delay) and creating mitigation plans.
  • Estimation: Negotiating timelines with the Product Owner.
  • Stakeholder Communication: Providing transparency through dashboards and status reports.
Q50 How do you estimate testing effort for a large project? ▼
Leadership
✦ Model Answer

I use a combination of techniques:

  • Work Breakdown Structure (WBS): Breaking the project into small modules and estimating each.
  • Historical Data: Looking at similar features from past sprints.
  • 3-Point Estimation: Calculating (Optimistic + 4*Likely + Pessimistic) / 6.
  • Buffer: I always add a 10-15% buffer for environment issues, bug re-testing, and meetings.
Q51 Difference between positive and negative testing. ▼
Basic
✦ Model Answer

Positive Testing: Verifying the system works as expected with valid inputs. Success path.

Negative Testing: Verifying the system handles invalid inputs or unexpected user behavior gracefully without crashing, usually showing correct error messages.

Q52 What is soak testing and when is it used? ▼
Advanced
✦ Model Answer

Soak testing (Endurance testing) involves testing the system with a typical production load for a long period (e.g., 24-48 hours). It's primarily used to find memory leaks, resource depletion, and performance degradation over time.

Q53 How do you test database integrity manually? ▼
Intermediate
✦ Model Answer
  • Data Mapping: Ensure UI fields match DB columns and types.
  • ACID Properties: Verify transactions follow Atomicity, Consistency, Isolation, and Durability.
  • Referential Integrity: Check that Primary and Foreign keys work as expected.
  • Data Accuracy: After a UI update, query the DB to ensure the correct values were saved.
  • Trigger/Stored Proc: Manually trigger events to ensure DB-side logic executes correctly.
Q54 What is ad-hoc testing? Pros and cons. ▼
Basic
✦ Model Answer

Ad-hoc testing is informal, unstructured testing performed without any documentation or formal test design techniques. It's usually done after formal testing is complete.

  • Pros: Fast, can find "unique" bugs, requires no planning.
  • Cons: Not systematic, bugs are hard to reproduce, no coverage guarantee, cannot be easily automated.
Q55 How do you manage test data for complex scenarios? ▼
IntermediateProcess
✦ Model Answer
  • Data Masking: Using production-like data but masking sensitive info (PHI/PII).
  • Automation Scripts: Using scripts to "pre-seed" the DB with necessary states before a test runs.
  • Centralized Repository: Storing reusable test data in Excel or JSON files.
  • Cleanup: Ensuring data is rolled back or deleted after testing to keep environments clean.
Q56 Difference between error, defect, and failure. ▼
Basic
✦ Model Answer
  • Error: A mistake made by a human (developer or BA).
  • Defect (Bug): An error found in the code or document before release.
  • Failure: When the end-user experiences an issue because of a defect that leaked into production.
Q57 What is the difference between static and dynamic testing? ▼
Basic
✦ Model Answer

Static Testing: Testing performed without executing the code. Examples include Code Reviews, Inspections, Walkthroughs, and Analysis of Requirements documents. It's about finding defects early in the lifecycle.

Dynamic Testing: Testing performed by executing the code and comparing the actual results with expected results. Examples include Functional testing, Integration testing, and System testing.

Q58 Explain the concept of 'Shift Left' testing. ▼
IntermediateProcess
✦ Model Answer

'Shift Left' means moving testing activities to the earlier stages of the SDLC. Instead of waiting for a "finished build," QA gets involved in requirement analysis, design reviews, and works closely with developers during coding (e.g., TDD/BDD).

Benefits: Defects are caught when they are cheapest to fix, requirements are clearer, and the overall quality is higher from the start.

Q59 What is the role of a QA in a DevOps environment? ▼
AdvancedStrategy
✦ Model Answer

In DevOps, QA transitions from being a "gatekeeper" to being a "quality enabler":

  • Automation First: Building and maintaining automated regression suites in CI/CD pipelines.
  • Monitoring: Helping set up alerts for production issues.
  • Collaboration: Working tightly with Dev and Ops to ensure "deployability."
  • Continuous Testing: Ensuring tests run at every stage (Commit, Build, Deploy).
Q60 How do you test for accessibility (WCAG)? ▼
Intermediate
✦ Model Answer

I test against WCAG 2.1 (A, AA, AAA) standards:

  • Keyboard Navigation: Ensure every element is reachable via 'Tab' and has a visible focus indicator.
  • Screen Readers: Use tools like NVDA or VoiceOver to check if ARIA labels and alt-text are meaningful.
  • Color Contrast: Use plugins to ensure text-to-background contrast ratios are at least 4.5:1.
  • Automated tools: Run scans using Axe DevTools or Lighthouse for quick baseline checks.
Q61 Explain the difference between severity and priority with 4 combinations. ▼
BasicIntermediate
✦ Model Answer
  • High Sev / High Priority: System crash on the main login page.
  • High Sev / Low Priority: Database crash that only happens on a legacy module used by 1 user once a year.
  • Low Sev / High Priority: The company logo is wrong on the home page (branding issue).
  • Low Sev / Low Priority: A small typo in the FAQ section footer.
Q62 What is monkey testing and how it differs from ad-hoc? ▼
Basic
✦ Model Answer

Monkey Testing: Randomly providing inputs and clicking anywhere without knowing the system logic (often automated by scripts). Goal: To crash the system.

Ad-hoc Testing: Unstructured but performed by a tester who knows the application, trying to find bugs using intuition and experience.

Q63 How do you test a web application's compatibility across different browsers? ▼
Intermediate
✦ Model Answer
  • Identify Target Browsers: Check analytics (GA) to see what users actually use (e.g., Chrome 70%, Safari 20%).
  • Functional Parity: Ensure core JS functions work in all (e.g., ES6 features).
  • UI Consistency: Check CSS layouts using tools like BrowserStack or LambdaTest.
  • Automation: Run the Selenium/Playwright suite against multiple browser drivers in parallel.
Q64 What is a test harness? ▼
AdvancedProcess
✦ Model Answer

A test harness is a collection of software and test data configured to test a program unit in isolation. It includes the test engine, the test scripts, and the mocks/stubs required to simulate the rest of the system. It's common in integration and unit testing.

Q65 Explain the purpose of a Test Summary Report. ▼
Leadership
✦ Model Answer

The Test Summary Report (TSR) provides stakeholders with a final assessment of the quality of the software before a release. It includes:

  • Pass/Fail status of all test cases.
  • List of open defects with their workarounds.
  • Test coverage achieved.
  • The QA's "Go/No-Go" recommendation.
Q66 How do you handle a situation where a developer says "it's not a bug"? ▼
IntermediateConflict Resolution
✦ Model Answer
  • Evidence First: I double-check the requirement/story (AC).
  • Reproduce: I show the dev exactly how it happens on my machine.
  • Impact Analysis: I explain how this "behavior" affects the end-user or business.
  • Neutral Arbitration: If we still disagree, I pull in the Product Owner (PO) to clarify if the behavior is acceptable or a bug.
Q67 What is localization and internationalization testing? ▼
Intermediate
✦ Model Answer

Internationalization (I18N): Designing the app to support multiple languages/regions (e.g., using resource files, supporting UTF-8).

Localization (L10N): Testing the app for a specific region (e.g., checking if German date formats, currency symbols, and cultural nuances are correct).

Q68 What is the difference between re-testing and regression testing? ▼
Basic
✦ Model Answer

Re-testing: Performed specifically to verify that a fixed bug is actually fixed. We run only the failed test cases.

Regression Testing: Performed to ensure that the bug fix hasn't broken other parts of the application. We run a larger set of existing test cases.

Q69 Explain the concept of 'Defect Density'. ▼
Intermediate
✦ Model Answer

Defect Density is a metric used to measure the quality of a software module. It is calculated as the Total number of defects found / Size of the module (e.g., per 1000 lines of code or per requirement). High density indicates a module that might need refactoring.

Q70 What is 'Mutation Testing'? ▼
Advanced
✦ Model Answer

Mutation testing is a technique to evaluate the effectiveness of your test cases. Small changes (mutations) are made to the source code (e.g., changing a > b to a < b). If your existing tests don't fail, it means your tests are weak and need improvement.

Q71 How do you perform end-to-end testing in a complex ecosystem? ▼
AdvancedProcess
✦ Model Answer
  • Map the Journey: Identify the critical path from user login to final outcome (e.g., Account creation → Payment → Invoice generation).
  • Environmental Sync: Ensure all integrated systems (DBs, 3rd party APIs) are pointed to the same test environment.
  • Data Orchestration: Use scripts to flow data through all systems.
  • Validation: Check not just the UI, but also DB entries and logs at each step.
Q72 What are the key components of a bug report? ▼
Basic
✦ Model Answer

A high-quality bug report must include:

  • Defect ID & Summary: Short and descriptive.
  • Severity & Priority: Impact vs Urgency.
  • Environment: OS, Browser, Device, Build Version.
  • Steps to Reproduce: Clear, numbered steps.
  • Expected vs Actual Result: What was supposed to happen vs what did.
  • Attachments: Screenshots, video recordings, or log files.
Q73 What is 'Gray Box' testing? ▼
Intermediate
✦ Model Answer

Gray box testing is a combination of black box (functional) and white box (internal code knowledge) testing. The tester has access to internal data structures, DB schemas, or architecture diagrams but tests from the outside. Common in API testing and Integration testing.

Q74 How do you test a system's 'usability'? ▼
BasicIntermediate
✦ Model Answer

I focus on the "5 E's":

  • Efficiency: How quickly can users complete tasks?
  • Easability: Is the UI intuitive or do I need a manual?
  • Error Prevention: Does the UI prevent common mistakes?
  • Explorability: Can users navigate back easily?
  • Engagement: Is the interface pleasant to look at?
Q75 Explain 'Risk-based testing' in detail. ▼
AdvancedStrategy
✦ Model Answer

Risk-based testing involves identifying, analyzing, and prioritizing test cases based on the probability of a defect and its impact on the business. I create a Risk Matrix (Likelihood x Impact) and focus my manual and automation efforts on the "High Risk" quadrant first.

Q76 What is a 'Test Catalyst' or 'Test Oracle'? ▼
Advanced
✦ Model Answer

A Test Oracle is the source of truth used to determine if a test passed or failed. It could be a Requirements document, an older version of the software, a competitor's product, or even a subject matter expert's (SME) knowledge.

Q77 How do you perform 'Sanity Testing' after a build? ▼
Basic
✦ Model Answer

Sanity testing is a subset of regression testing. After a build with minor fixes, I focus only on the components that were modified. I ensure the bugs are fixed and the immediate features around them are still working correctly. If sanity fails, I reject the build immediately to save time.

Q78 Explain 'Integration Testing' strategies (Top-down, Bottom-up, Sandwich). ▼
Intermediate
✦ Model Answer
  • Top-Down: Testing from top-level modules to lower-level ones. We use Stubs to simulate missing lower-level modules.
  • Bottom-Up: Testing from lower-level modules to higher-level ones. We use Drivers to simulate missing higher-level modules.
  • Sandwich (Hybrid): A combination of both approaches, testing from both ends towards the middle.
Q79 What is the difference between Decision Table and State Transition testing? ▼
Intermediate
✦ Model Answer

Decision Table: Used when business logic depends on combinations of inputs. It's "stateless"—the outcome depends only on current inputs.

State Transition: Used when the system's behavior depends on its *history* (current state) and an event. It's "stateful"—the same input might lead to different results depending on the previous state.

Q80 How do you test 'non-functional' requirements? ▼
Advanced
✦ Model Answer

Non-functional requirements (NFRs) define "how" the system works. I test them using:

  • Performance: Using JMeter/Gatling for load and stress.
  • Security: Checking for OWASP Top 10 vulnerabilities.
  • Usability: Heuristic evaluation and user feedback.
  • Compatibility: Testing across OS/Browser combinations.
  • Scalability: Checking if the system handles increased users by adding resources.
Q81 What is 'Security Testing' from a manual perspective? ▼
Advanced
✦ Model Answer

Manual security testing involves:

  • Authentication: Checking if passwords are masked, account lockout works, and MFA is enforced.
  • Authorization: Ensuring a regular user cannot access admin URLs.
  • Input Validation: Trying basic SQL injection (' OR 1=1) or XSS scripts (<script>alert(1)</script>) in fields.
  • Session Management: Checking if the session expires after logout or inactivity.
Q82 Explain the 'V-Model' in software testing. ▼
Intermediate
✦ Model Answer

The V-Model (Verification and Validation Model) suggests that for every development phase, there is a corresponding testing phase. It emphasizes early test planning (e.g., writing UAT cases while requirements are being defined). It's more disciplined than Waterfall but less flexible than Agile.

Q83 What is 'Agile Testing' and how is it different from Waterfall? ▼
Intermediate
✦ Model Answer

Waterfall: Testing is a distinct phase after "Development" is complete. Expensive to fix bugs found late.

Agile: Testing is continuous and iterative. QA works in the same sprint as Dev. We use BDD/TDD and frequent regression (automated) to ensure quality in every increment.

Q84 How do you document a 'Test Strategy'? ▼
Leadership
✦ Model Answer

A Test Strategy is a high-level document that defines the overall testing approach for the organization or a large project. It includes: Scope, Test Levels (Unit, System, etc.), Environment requirements, Tools to be used, Defect management process, and Exit criteria.

Q85 What is 'User Acceptance Testing' (UAT)? ▼
Intermediate
✦ Model Answer

UAT is the final phase of testing performed by the end-users or the business team. Goal: To ensure the software is "fit for use" in a real-world business environment. It focuses on end-to-end business flows rather than individual technical features.

Q86 Explain 'Alpha' vs 'Beta' testing. ▼
Basic
✦ Model Answer

Alpha: Performed by internal QA/staff at the developer's site. Controlled environment.

Beta: Performed by a limited set of real users in their own environment ("in the wild"). Helps find issues related to varied hardware, networks, and unexpected user behavior.

Q87 What is 'Crowdsourced Testing'? ▼
Intermediate
✦ Model Answer

Crowdsourced testing involves using a large, distributed group of freelancers (often via platforms like Applause or uTest) to test your app. Pros: Diverse devices/locations, fast results, pay-per-bug. Cons: Security risks, variable quality of reports, management overhead.

Q88 How do you test for 'performance' manually? ▼
BasicIntermediate
✦ Model Answer

Manual performance testing involves observing perceived latency:

  • Using a stopwatch to measure page load time.
  • Checking for UI "jank" or freezing during heavy operations.
  • Monitoring browser Network Tab for slow API calls.
  • Checking for memory spikes in Task Manager.
Q89 What is an 'Environment-related defect'? ▼
Intermediate
✦ Model Answer

A defect that is not caused by the code itself but by the setup of the test environment (e.g., wrong DB version, expired SSL certificate, firewall blocking a port, or incorrect config files). These are common in integration testing.

Q90 Explain 'Baseline Testing'. ▼
Basic
✦ Model Answer

Baseline testing is the process of establishing a "benchmark" for future comparisons. We run tests on a stable version of the app to document the expected performance/behavior, which then serves as the standard for future builds.

Q91 What is 'Recovery Testing'? ▼
Advanced
✦ Model Answer

Recovery testing verifies how well the system recovers after a crash, hardware failure, or network loss. We deliberately force a failure (e.g., unplugging the DB) and check if the system resumes state, logs errors correctly, and maintains data integrity.

Q92 How do you perform 'Cloud Testing' manually? ▼
Intermediate
✦ Model Answer

Cloud testing involves testing applications hosted on AWS/Azure/GCP. Key focus areas: Scalability (checking if new instances spin up under load), Reliability (multi-region failover), and Security (checking IAM roles and S3 bucket permissions manually).

Q93 What is 'ETL Testing' from a manual QA perspective? ▼
AdvancedProcess
✦ Model Answer

ETL (Extract, Transform, Load) testing ensures data is correctly moved from source systems to a data warehouse. Manual QA checks: Data Completeness (row counts match), Data Transformation (business logic applied correctly), and Data Quality (no nulls/duplicates where forbidden).

Q26 What is the difference between verification and validation? ▼
BasicIntermediate
✦ Model Answer

Verification — "Are we building the product right?" It's a static process checking that the software conforms to specifications. Examples: code reviews, requirement reviews, design walkthroughs. No execution required.

Validation — "Are we building the right product?" It's a dynamic process — actually running the software to ensure it meets the user's needs. Examples: functional testing, UAT, system testing. Execution required.

Scenario: If a requirement says "login button must be blue" and the developer builds a blue button that opens the wrong page — verification passes (blue button exists) but validation fails (wrong page). Both are necessary.

Q31 Explain the Software Testing Life Cycle (STLC). ▼
BasicIntermediate
✦ Model Answer

STLC consists of specific phases to ensure quality goals are met:

  • Requirement Analysis: QA interacts with stakeholders to understand requirements. Are they testable?
  • Test Planning: Preparation of Test Strategy/Plan, resource allocation, and effort estimation.
  • Test Case Development: Creating detailed test cases, test data, and RTM.
  • Environment Setup: Configuring the test environment (servers, databases, devices).
  • Test Execution: Running the tests and logging defects.
  • Test Closure: Analysis of results, creating Test Summary Reports, and lesson-learned sessions.
Q32 What are the 7 principles of software testing? ▼
Basic
✦ Model Answer
  • 1. Testing shows presence of defects — it reduces the probability of undiscovered defects but cannot prove absence
  • 2. Exhaustive testing is impossible — test with risk-based and priority-driven approaches
  • 3. Early testing saves money — defects found in requirements are cheaper to fix than in production
  • 4. Defect clustering — most defects are found in a small number of modules (80/20 rule)
  • 5. Pesticide paradox — repeating the same tests finds fewer new bugs; tests must evolve
  • 6. Testing is context dependent — safety-critical testing differs from e-commerce testing
  • 7. Absence of errors fallacy — a bug-free system that doesn't meet user needs still fails

Q36 Explain equivalence partitioning and boundary value analysis with examples. ▼
Basic
✦ Model Answer

Equivalence Partitioning (EP): Divides input data into logical groups (partitions) where each value in a group is expected to be processed similarly. We test only one value from each partition.

Boundary Value Analysis (BVA): Focuses on testing at the boundaries of these partitions, where bugs are most likely to occur (minimum, maximum, just inside, and just outside).

Example: A field accepts age between 18 and 60.
EP: Valid (18-60), Invalid (>60), Invalid (<18).
BVA: 17, 18, 19 and 59, 60, 61.

Q38 How do you perform root cause analysis of production defects? ▼
AdvancedProcess
✦ Model Answer

I follow a structured "5 Whys" approach or a fishbone diagram:

  1. Gather Data: Collect logs, screenshots, environment details, and steps to reproduce.
  2. Identify the Gap: Determine why the bug wasn't caught in QA. Was it a missing test case? An environment difference? Or a data-specific issue?
  3. Root Cause: Dig deep. Example: Why did the server crash? (Memory leak). Why? (Unclosed DB connection). Why? (Missing finally block in code).
  4. Corrective Action: Fix the bug and add a regression test.
  5. Preventive Action: Update the test strategy or add code review checklists to ensure similar issues don't recur.

Q40 Explain decision table testing technique. ▼
Intermediate
✦ Model Answer

Decision table testing is used for features containing complex business logic with multiple true/false conditions. We create a table listing all possible combinations of inputs (Conditions) and the resulting actions (Outcomes).

Example: A banking app allows transfer if (1) Account is active and (2) Balance > Transfer Amount. Each combination (Y/Y, Y/N, N/Y, N/N) becomes a test scenario.
Q41 What is state transition testing? ▼
Intermediate
✦ Model Answer

State transition testing is used when the application behavior depends on "states" and certain events that trigger transitions between them. We test all valid transitions and ensure invalid transitions are blocked.

Example: An ATM. States: Idle → Card Inserted → Pin Entered → Transaction Completed. We test that you can't go from 'Idle' to 'Transaction Completed' without entering a PIN.

Q44 Difference between smoke, sanity, and regression testing. ▼
Basic

Q46 What is defect leakage and defect age? ▼
Intermediate
✦ Model Answer

Defect Leakage: When a bug is not caught by the QA team but is found by the end-user in production. Formula: (Bugs in Prod / Total Bugs) * 100.

Defect Age: The time difference between when a bug was reported and when it was closed. It helps measure the efficiency of the dev-QA cycle.

Q47 Explain pairwise testing with an example. ▼
Advanced
✦ Model Answer

Pairwise (All-Pairs) testing is a combinatorial technique that assumes most bugs are caused by the interaction of at most two parameters. It significantly reduces the number of test cases while maintaining strong coverage.

Example: Testing a printer with 3 params: OS (Win, Mac), Connection (USB, Wifi), Paper (Plain, Photo).
Instead of 2*2*2 = 8 cases, pairwise selects combinations (e.g., Win/USB, Win/Wifi, Mac/USB, Mac/Wifi) ensuring every pair is tested.

Q97 List the different locator strategies in Selenium. ▼
Basic
✦ Model Answer

Starting from the most preferred:

  1. ID: Fastest and most unique.
  2. Name: Good if unique.
  3. CSS Selector: Faster and more stable than XPath.
  4. XPath: Most powerful (can navigate backwards).
  5. LinkText / PartialLinkText: For anchor tags.
  6. ClassName: For elements with unique classes.
  7. TagName: To find a group of elements (e.g., all <a> tags).
Q98 What is the difference between absolute and relative XPath? ▼
Basic
✦ Model Answer
  • Absolute XPath: Starts from the root node (/html/body/...). It is very brittle because any small change in the UI structure breaks the locator.
  • Relative XPath: Starts from anywhere in the DOM (//div[@id='login']). It is more stable and preferred in automation.
Q99 How do you handle dynamic elements in Selenium? ▼
Intermediate
✦ Model Answer

Dynamic elements have attributes (like IDs) that change on every reload. Strategies include:

  • XPath Functions: Using contains(), starts-with(), or text(). Example: //button[contains(@id, 'login')].
  • Following/Parent Axes: Finding a stable neighbor and navigating to the target.
  • CSS Selectors: Using wildcards like ^= (starts-with), $= (ends-with), or *= (contains).
Q100 Explain implicit, explicit, and fluent waits. ▼
Intermediate
✦ Model Answer
  • Implicit Wait: A global wait applied to the driver. It waits for a specific time before throwing a NoSuchElementException.
  • Explicit Wait: Used to wait for a *specific condition* (e.g., elementToBeClickable) for a specific element. It's more efficient than implicit.
  • Fluent Wait: A type of explicit wait where you can define the maximum time to wait, the polling frequency, and which exceptions to ignore (e.g., NoSuchElementException).

Q214 What is the core principle of the Page Object Model (POM)? ▼
Basic
✦ Model Answer

The core principle is **encapsulation**. It separates the test logic (assertions) from the UI structure (locators and interactions). Each web page has a corresponding class that contains its elements and methods to interact with them.

Q215 In POM, what should be Private vs Public? ▼
Intermediate
✦ Model Answer
  • Private: Locators (By objects) and raw IWebElement properties. This prevents tests from interacting with elements directly.
  • Public: Action methods (e.g., Login(user, pass), GetTitle()) that represent user behavior.
Q216 Explain the primary benefits of using POM. ▼
Basic
✦ Model Answer
  • Maintainability: If a locator changes, you update it in one place, not in dozens of tests.
  • Reusability: Page methods can be shared across multiple tests.
  • Readability: Test scripts read like English (e.g., LoginPage.Login("u", "p")).
  • DRY Principle: Eliminates code duplication.
Q217 What is a BasePage and why is it useful? ▼
Intermediate
✦ Model Answer

A BasePage is a parent class that all specific Page Objects inherit from. It centralizes common functionality like the IWebDriver instance, common explicit waits, logging, and shared elements like navigation bars or headers.

Q218 What are "Component Objects" in POM? ▼
Advanced
✦ Model Answer

For large applications, a single page class can become too big. **Component Objects** break the page down into smaller, reusable UI pieces like a HeaderComponent, SidebarComponent, or FooterComponent. These components are then instantiated or included via properties in the main Page Objects.

Q219 Page Factory vs Standard By locators. ▼
Intermediate
✦ Model Answer
  • Standard By: Locators are explicitly called using driver.FindElement(By.Id("...")) every time.
  • Page Factory: Uses attributes like [FindsBy] to initialize elements. It uses **Lazy Initialization**; elements are only searched when they are first used.
Q220 Why is Page Factory often avoided in modern Selenium 4 frameworks? ▼
Advanced
✦ Model Answer

Selenium 4 officially deprecated the original .NET implementation of Page Factory. Reasons include: reliance on reflection, difficulty in handling complex waits (it often leads to StaleElementReferenceException), and its inability to handle dynamic locators effectively compared to standard By locators with wrappers.

Q221 How to handle dynamic elements in POM? ▼
Intermediate
✦ Model Answer

Instead of hardcoded locators, we use methods that return an element based on a parameter. Example: public IWebElement GetColumn(string colName) => driver.FindElement(By.XPath($"//td[text()='{colName}']"));. This keeps the POM flexible.

Q222 What is the Builder Pattern in automation? ▼
Advanced
✦ Model Answer

It helps in creating complex test data objects or configuring the test environment step-by-step. In POM, it's often used to chain actions: new RegistrationPage().WithEmail("x").WithPassword("y").Build().Submit();. It improves readability and makes code more declarative.

Q223 What is the Fluent Interface pattern in POM? ▼
Intermediate
✦ Model Answer

Methods in the Page Object return the object itself (return this;) or the next page object in the flow. This allows method chaining in the test: LoginPage.EnterUsername("u").EnterPassword("p").ClickLogin();. It creates a "fluent" reading experience.

Q224 Where should assertions live: POM or Test? ▼
Intermediate
✦ Model Answer

In the Test. Page Objects should ideally only provide services (UI interactions and data retrieval). Assertions in the POM make it less reusable across different test scenarios where the expected result might differ.

Q225 Explain Page Chaining. ▼
Intermediate
✦ Model Answer

When an action on one page leads to another page, the POM method should return an instance of that new page. Example: public DashboardPage Login() { ... return new DashboardPage(driver); }. This enforces valid transitions in the test flow.

Q226 What is a LoadableComponent? ▼
Advanced
✦ Model Answer

It’s a standard Selenium class that ensures a page is fully loaded before the driver interacts with it. It has two main methods: load() (navigates to the page) and isLoaded() (asserts that a specific element exists to confirm readiness).

Q227 How to manage multiple frames in POM? ▼
Intermediate
✦ Model Answer

Each frame should ideally be treated as its own component or nested class. The page methods should handle the SwitchTo().Frame() and SwitchTo().DefaultContent() calls internally, so the test doesn't have to worry about the underlying frame structure.

Q228 Organizing locators: Strings vs By objects vs Elements. ▼
Intermediate
✦ Model Answer

Using By objects (e.g., private By _loginBtn = By.Id("btn");) is preferred. It is more flexible than raw strings and safer than IWebElement properties, which can go stale if the page refreshes before the interaction.

Q229 Is POM a testing framework? ▼
Basic
✦ Model Answer

No, POM is a design pattern. A testing framework (like NUnit + Selenium) *implements* the POM pattern to structure its code. POM only dictates *how* classes are organized, not how tests are run or reported.

Q230 Drawbacks of using POM. ▼
Intermediate
✦ Model Answer
  • High initial effort to create page classes.
  • Can be overkill for very small automation projects.
  • Requires strong object-oriented programming (OOP) knowledge from the QA team.
Q231 Scaling POM for large enterprise apps. ▼
Advanced
✦ Model Answer

Use a layered structure: Divide pages into modules/folders. Use composition heavily (nested components). Implement an interface-driven approach if testing across multiple platforms (Web/Mobile) to share common logic while specializing locators.

Q232 Lazy Initialization in POM. ▼
Advanced
✦ Model Answer

Create a dedicated Page Object for the modal. Since modals are often shared (e.g., a 'Feedback' modal on every page), they can be instantiated inside the BasePage or returned by a general method to avoid code repetition in every specific page class.

Q234 How does POM apply to Mobile Testing (Appium)? ▼
Intermediate
✦ Model Answer

The logic is identical. Instead of IWebDriver, you use AppiumDriver. You organize screens into Screen Classes (Screen Object Model). Locators might use accessibility IDs instead of XPaths, but the encapsulation principle remains the same.

Q235 What is the Screenplay Pattern vs POM? ▼
Advanced
✦ Model Answer

Screenplay is an evolution of POM. While POM focuses on "Pages" (UI structure), Screenplay focuses on **"Actors" and "Tasks"**. It follows SOLID principles more strictly, avoiding the "God Object" problem often seen in massive Page Object classes.

Q236 How to use ByChained in C#? ▼
Advanced
✦ Model Answer

ByChained is used to find nested elements. It looks for a parent first, then searches for the child *only* within that parent. Example: new ByChained(By.Id("table"), By.ClassName("row")). It’s cleaner than writing a very long, complex XPath.

Q237 Handling Table interactions in POM. ▼
Advanced
✦ Model Answer

Create a dedicated TableComponent class. It should have methods like GetRowCount(), GetCellText(row, col), and ClickButtonInRow(name). Internally, use dynamic XPaths that build queries based on the parameters passed.

Q238 What is "Page Discovery"? ▼
Advanced
✦ Model Answer

It's the ability of a framework to identify the current page based on visible elements and automatically instantiate the correct Page Object. It's often used in "Magic" automation frameworks to reduce boilerplate instantiation code in scripts.

Q239 POM vs DRY: When not to refactor? ▼
Intermediate
✦ Model Answer

Don't refactor different pages just because they *happen* to look the same. For example, a 'Contact Us' form and a 'Lead Gen' form might have the same fields today, but they serve different business purposes. Keeping them separate avoids "Tight Coupling" that breaks when one form changes but the other doesn't.

Q240 Thread safety in POM instance management. ▼
Advanced
✦ Model Answer

When running tests in parallel, each thread must have its own instance of the Page Objects and the underlying WebDriver. Use ThreadLocal<IWebDriver> or a DI container (like Autofac or Microsoft.Extensions.DependencyInjection) configured with Scoped lifetime.

Q241 Handling multi-lingual sites with POM. ▼
Advanced
✦ Model Answer

Avoid using text-based locators (like By.LinkText("Home")). Stick to IDs, Data-Attributes (data-qa="home"), or name attributes which remain constant across different language versions of the site.

Q242 Shadow DOM in POM. ▼
Advanced
✦ Model Answer

Selenium cannot find elements inside a Shadow DOM using standard FindElement. You must use GetShadowRoot(): var shadowRoot = hostElement.GetShadowRoot(); var element = shadowRoot.FindElement(By.CssSelector("#inner"));. POM action methods should encapsulate this traversal.

Q243 What are Custom Element wrappers? ▼
Intermediate
✦ Model Answer

Instead of interacting with raw IWebElement, you create classes like Checkbox or Calendar that wrap the element. These wrappers provide higher-level methods like Check(), Uncheck(), or SelectDate(dateTime), hiding the repetitive JS clicks or typing logic.

Q244 Using Interfaces for Pages. ▼
Advanced
✦ Model Answer

Useful for cross-platform apps. Define ILoginPage with Login(u,p). Implement WebLoginPage and AndroidLoginPage. Your test script then interacts with ILoginPage, making the script platform-agnostic.

Q245 Handling Lazy elements manually. ▼
Intermediate
✦ Model Answer

Instead of declaring elements as properties, use private methods or Lazy<IWebElement> in C#. This ensures the driver only reaches out to the browser the moment you access the element, preventing stale or missing element errors during page initialization.

Q246 Logging strategies inside Page Objects. ▼
Intermediate
✦ Model Answer

Inject a logger (e.g., NLog, Serilog) into your BasePage. Every action method should log a meaningful step: _logger.Info($"Entering username: {user}");. This makes the execution logs much easier for developers to debug when a test fails in CI.

Q247 Error handling inside POM. ▼
Advanced
✦ Model Answer

Don't just catch exceptions; throw custom ones. Example: If login fails, throw UserAuthenticationException("Failed to login after 3 attempts") with a screenshot. Custom exceptions provide much better context than a generic WebDriverException.

Q248 POM for Single Page Applications (SPA). ▼
Intermediate
✦ Model Answer

In SPAs, the "Page" doesn't reload, but the UI changes significantly. View the "Views" or "Components" as the Page Objects. Because there's no reload, **Explicit Waits** are even more critical to handle the transient DOM states during React/Angular updates.

Q249 How to test a Search functionality using POM? ▼
Intermediate

ClickSearchButton();
return new SearchResultsPage(driver);
}
Q250 How to handle dynamic IFrames that share IDs? ▼
Advanced
✦ Model Answer

Use XPath to find the frame by its position, relative labels, or specific attributes other than ID. Your POM method should always explicitly switch back to DefaultContent() after the operation to avoid getting the driver "stuck" in a frame that might disappear.

Q251 Utility class vs BasePage method? ▼
Intermediate
✦ Model Answer
  • Utility: Helper methods that don’t need the driver (e.g., generating random emails, parsing dates, connecting to a DB).
  • BasePage: Methods that actively use the driver to interact with the UI.
Q252 Testing complex charts/graphs in POM. ▼
Advanced
✦ Model Answer

You can't easily find elements inside a Canvas or SVG using ID. Use JS Executor to query the application's internal data model, or use Visual Testing (Applitools/Percy) to compare the chart's rendering against a baseline.

Q253 Best practices for naming in POM. ▼
Basic
✦ Model Answer
  • Pages: [Feature]Page (e.g., OrderPage.cs)
  • Locators: _xxxLocator (e.g., _submitBtnLocator)
  • Methods: Verb-Noun (e.g., SubmitOrder(), CancelPromotion())
Q254 Page Factory in C# (SeleniumExtras). ▼
Advanced
✦ Model Answer

The original PageFactory moved to the DotNetSeleniumExtras package. While it still works using PageFactory.InitElements(driver, this), it's generally recommended to move away from it for better control over element lifecycle and wait logic.

Q255 How to handle Hover menus in POM? ▼
Intermediate
✦ Model Answer

Encapsulate the Actions logic. public void NavigateToSubMenu(string sub) { new Actions(driver).MoveToElement(mainMenu).Perform(); wait.Until(EC.ElementIsVisible(subMenuByText(sub))).Click(); }. This hides the complexity from the test script.

Q256 Using WaitForms or Splash Screens. ▼
Intermediate
✦ Model Answer

Handle these in your BasePage or WaitWrapper. Create a method WaitForSpinnerToDisappear(). Call this at the start of every page action that triggers a backend call. It's the #1 way to eliminate "flaky" automation.

Q257 Integrating POM with ExtentReports. ▼
Advanced
✦ Model Answer

Use a TestManager or ThreadLocal to store the ExtentTest instance. Your Page Objects can then retrieve this instance to log fine-grained UI steps directly into the beautiful ExtentReport HTML file.

Q258 Handling AJAX loading indicators globally. ▼
Advanced
✦ Model Answer

You can use JavaScript Executor to check the state of the Page: return (bool)driver.ExecuteScript("return jQuery.active == 0");. Including this check in your global `Wait` utility ensures your automation never clicks while the page is still "talking" to the server.

Q259 What is the Facade Pattern in automation? ▼
Advanced
✦ Model Answer

A Facade wraps multiple Page Objects to provide a single, high-level business action. Example: A CheckoutFacade that coordinates the CartPage, AddressPage, and PaymentPage. The test simply calls `facade.CompleteOrder()`, shielding it from the multi-page workflow complexity.

Q260 Comparing POM with the Factory Pattern. ▼
Advanced
✦ Model Answer

Factory creates the Page Objects; POM defines their structure. They work together: var page = PageFactory.GetPage(driver);. This centralizes object creation and can handle environment-specific page versions automatically.

Q261 How to avoid "God Objects" in POM? ▼
Advanced
✦ Model Answer

If a class has more than 20-30 methods, it's becoming a "God Object." Break it down using Composition. Move functional areas into sub-components. A Page Object should ideally just be a conductor of smaller, focused components.

Q262 Using Dependency Injection with POM. ▼
Advanced
✦ Model Answer

Instead of new LoginPage(driver), use DI: private readonly ILoginPage _loginPage;. This decouples the test from the implementation and makes it trivial to swap in mock objects or different platform implementations for the same page interface.

Q263 Handling StaleElementReferenceException in wrappers. ▼
Advanced
✦ Model Answer

Your click/type wrappers in BasePage should have a retry mechanism. try { element.Click(); } catch (StaleElementReferenceException) { // Re-find the element and try again }. This significantly reduces random test failures caused by JS-driven DOM refreshes.

Q264 Benefit of private readonly for locators. ▼
Intermediate
✦ Model Answer

Using private readonly By _locator ensures that the locator cannot be accidentally changed by any page method at runtime. It enforces immutability and makes the code safer and easier to reason about.

Q265 What is a "Block Object"? ▼
Advanced
✦ Model Answer

Similar to Component Objects, but specifically for repeating blocks inside a list (e.g., a single card in a search result list). A SearchResultBlock handles interactions with *one* result, and the SearchPage handles the list of blocks.

Q266 How to test a "File Download" verification in POM? ▼
Advanced
✦ Model Answer

Don't rely on the browser's download manager. Configure the driver to download to a specific folder in your project. Then, have a FileSystemUtility method called in your POM that checks if the file exists and is not empty after the download click.

Q267 Using Action delegates for generic interactions. ▼
Advanced
✦ Model Answer

In C#, you can pass Action or Func to your wait wrappers to execute complex logic until a condition is met. This is much more flexible than just waiting for an element to be visible, as it can handle multi-step polling scenarios.

Q268 Cross-browser concerns in Page Objects. ▼
Intermediate
✦ Model Answer

Sometimes Safari or IE require different interaction types (e.g., JS click instead of native click). Your BasePage should detect the browser type from the driver and apply those specific workarounds once, so the individual Page Objects stay clean.

Q269 How to deal with "Obstructed Elements"? ▼
Advanced
✦ Model Answer

When an element is covered by a 'Loading' overlay or 'Cookie Banner', the normal click fails. Your POM should handle clearing the obstruction first (e.g., accept cookies) or use a JavaScript Click as a fallback to bypass the obstruction if it's purely visual.

Q270 Verifying visual elements (CSS/Styles) through POM. ▼
Intermediate
✦ Model Answer

Use element.GetCssValue("color"). Your Page Object should have methods like IsButtonHighlighted() that return a boolean. This is useful for accessibility testing and ensuring UI standards are met during regression.

Q271 When to use "Self-healing" locators in POM? ▼
Advanced
✦ Model Answer

Self-healing (AI-based) locators are useful when dealing with highly dynamic, unstable front-ends. They should only be used as a SECONDARY fallback in your POM when the primary reliable locator (like a QA ID) is not available.

Q272 POM for legacy systems (Silverlight/Flash). ▼
Advanced
✦ Model Answer

Selenium cannot interact with these directly. You normally have to use an external tool (like WinAppDriver or image-based automation) wrapped inside your POM classes. The Page Object acts as a bridge between Selenium and these external automation engines.

Q274 What is Behavior-Driven Development (BDD)? ▼
Basic
✦ Model Answer

BDD is an Agile software development process that encourages collaboration between developers, QA, and non-technical business stakeholders. It focuses on defining the **behavior** of the system using simple, structured language (Gherkin) before any code is written.

Q275 Explain the Given-When-Then syntax. ▼
Basic
✦ Model Answer
  • Given: Pre-condition or initial state.
  • When: The action or event performed by the user.
  • Then: The expected outcome or verification.
Q276 What are Feature Files in SpecFlow? ▼
Basic
✦ Model Answer

Feature files (.feature) are plain text files that contain system requirements written in Gherkin. They act as "Living Documentation" and as the source of truth for what a specific feature should do.

Q277 What are Step Definitions? ▼
Intermediate
✦ Model Answer

Step Definitions are C# methods that the BDD framework executes when it encounters a matching line in a feature file. They use attributes like [Given], [When], and [Then] with regex or cucumber expressions to map text to code.

Q278 Scenario vs Scenario Outline. ▼
Intermediate
✦ Model Answer
  • Scenario: A single test case run once.
  • Scenario Outline: A template run multiple times using different sets of data provided in an **Examples** table.
Q279 What is the role of Examples tables? ▼
Basic
✦ Model Answer

Found inside a Scenario Outline, the Examples table provides a structured way to run the same scenario with different inputs and expected outputs, which is highly efficient for data-driven testing.

Q280 What is Gherkin? ▼
Basic
✦ Model Answer

Gherkin is the Business Readable, Domain Specific Language (DSL) used by tools like Cucumber and SpecFlow. It defines a set of grammar rules to structure human-readable scenarios.

Q281 Who should write Gherkin scenarios? ▼
Leadership
✦ Model Answer

Ideally, it’s a collaborative effort (The Three Amigos: Business/PO, Developer, and QA). However, the Product Owner or Business Analyst should drive the requirements part, while QA ensures they are testable.

Q282 Benefits of BDD for cross-functional teams. ▼
Leadership
✦ Model Answer

It reduces ambiguity, ensures everyone has a shared understanding of the requirements, provides clear "Definition of Done," and creates tests that also serve as documentation.

Q283 What are SpecFlow Hooks? ▼
Intermediate
✦ Model Answer

Hooks allow you to execute code at specific points in the test lifecycle, such as [BeforeScenario], [AfterScenario], [BeforeStep], etc. They are used for setup/teardown logic (e.g., clearing the DB or starting the driver).

Q284 What is Context Injection in SpecFlow? ▼
Advanced
✦ Model Answer

It’s the built-in Dependency Injection (DI) mechanism in SpecFlow. It allows you to share state between multiple step definition classes by injecting common objects through the constructor.

Q285 ScenarioContext vs FeatureContext. ▼
Advanced
✦ Model Answer
  • ScenarioContext: Stores data for the duration of a single scenario (Reset per scenario).
  • FeatureContext: Stores data for all scenarios within a single feature file.
Q286 Sharing data between steps using ScenarioContext. ▼
Intermediate
✦ Model Answer
_scenarioContext["Key"] = value;
var value = _scenarioContext["Key"];

While useful, using POCO objects via Context Injection is usually cleaner for large projects.

Q287 Reusing steps across features. ▼
Basic
✦ Model Answer

Steps are globally available as long as they are in the same assembly or referenced assembly. If you write "Given I logout" once, you can use it in 50 different feature files.

Q288 Parameterizing steps. ▼
Intermediate
✦ Model Answer

Using regular expressions or placeholders to capture data from a step. Example:
[When(@"I enter ""(.*)"" into the field")]
public void WhenIEnterValue(string value) { ... }.

Q289 Using Data Tables in Gherkin. ▼
Intermediate
✦ Model Answer

Used to pass a complex set of data to a single step. Represented by pipes | field | value | directly under the step. Accessed in C# via the Table object arguments.

Q290 What is SpecFlow.Assist? ▼
Advanced
✦ Model Answer

A set of extension methods for the Table class that allow easy mapping between Gherkin tables and C# objects using methods like CreateInstance<T>() and CreateSet<T>().

Q291 What are Tags in SpecFlow? ▼
Basic
✦ Model Answer

Metadata identifiers (prefixed with @) used to categorize and filter scenarios for execution. Example: @smoke, @regression, @ignore.

Q292 Using Tags for test filtering. ▼
Intermediate
✦ Model Answer

You can tell the test runner (NUnit/VSTest) to only include or exclude certain tags in a run. Example: dotnet test --filter "Category=smoke".

Q293 Background keyword in Gherkin. ▼
Basic
✦ Model Answer

Used to define steps that should run before **every** scenario in the feature file. Eliminates duplication of common Given steps (e.g., "Given I am logged in").

Q294 Imperative vs Declarative Gherkin. ▼
Advanced
✦ Model Answer
  • Imperative: Detailed UI steps ("When I enter 'x' in 'ID_1', When I click 'ID_2'"). Hard to maintain.
  • Declarative: High-level business intent ("When I login as a valid user"). Much easier to read and maintain.
Q295 Common Gherkin anti-patterns. ▼
Leadership
✦ Model Answer
  • Writing scenarios that are too long.
  • Using too many technical details (IDs, DB table names).
  • Combining multiple behaviors into one scenario.
  • Missing the "Three Amigos" collaboration.
Q296 Simplifying "step-heavy" scenarios. ▼
Advanced
✦ Model Answer

If a scenario has 20 steps, it's often a sign it should be broken down. Use the **Background** for common setup, move detailed UI fields into a **Data Table**, or refactor the steps to be more **Declarative** (batching multiple actions into one step).

Q297 Living Documentation (SpecFlow+ LivingDoc). ▼
Leadership
✦ Model Answer

A tool that transforms your Gherkin feature files into beautiful HTML reports that show both the requirements and the latest test results. It makes automation accessible to the whole company.

Q298 Cucumber vs SpecFlow. ▼
Intermediate
✦ Model Answer

They are functionally identical in their use of Gherkin. **Cucumber** originated in Ruby and has Java/JS versions. **SpecFlow** was built specifically for the .NET ecosystem and integrates tightly with Visual Studio and NuGet.

Q299 SpecFlow configuration: specflow.json. ▼
Intermediate
✦ Model Answer

A JSON file used in modern SpecFlow versions to configure project-wide settings like the feature file language, unit test provider (NUnit/xUnit), and step assembly references.

Q300 Parallel execution in SpecFlow. ▼
Advanced
✦ Model Answer

SpecFlow supports parallel execution at the feature level by default if the underlying test runner (like NUnit) supports it. You must ensure your **Context Injection** objects are thread-safe and not shared across threads incorrectly.

Q301 Handling setup/cleanup for specific tags. ▼
Intermediate
✦ Model Answer

Hooks can be scoped using tags. Example: [BeforeScenario("web")] will only run before scenarios marked with @web. This is great for only starting a browser when needed.

Q302 Custom Step Argument Transformations. ▼
Advanced
✦ Model Answer

Used to convert a Gherkin string into a complex C# object automatically. Example: Transforming "3 days from now" into a DateTime object before it even hits your step definition method.

Q303 Regular Expressions in Matchers. ▼
Advanced
✦ Model Answer

Used for powerful step matching. Example: [When(@"I (add|remove) the item")] can handle both "I add the item" and "I remove the item" in a single method by capturing the verb as a parameter.

Q304 Scope binding in SpecFlow (Binding by Tag). ▼
Advanced
✦ Model Answer

Using the [Scope(Tag = "mytag")] attribute on a step definition class or method restricts those steps to only work for scenarios with that tag. This allows you to have different implementations for the same step text based on context.

Q305 Using External Data with SpecFlow (Excel/JSON). ▼
Advanced
✦ Model Answer

SpecFlow doesn't natively support Excel for Example tables. You either use a plugin (like SpecFlow.ExternalData) to pull from CSV/JSON, or you write a [BeforeScenario] hook that reads the file and injects data into the ScenarioContext.

Q306 Handling "And" and "But" steps. ▼
Basic
✦ Model Answer

In Gherkin, And and But inherit the keyword of the step above them to improve readability. In SpecFlow, you still map them using [Given], [When], or [Then] attributes based on which functional category they belong to.

Q307 Best practices for writing readable scenarios. ▼

A Test Runner is a tool (like vstest.console.exe or the VS Test Explorer) that scans your assemblies for test methods, executes them, and aggregates the results into a report (like a TRX or XML file).

Q336 What are Test Attributes (Annotations)? ▼
Basic
✦ Model Answer

Attributes are metadata tags (e.g., [Test], [SetUp]) placed above methods to tell the Test Runner how to handle that code. They define the entry points and the lifecycle of the test execution.

Q337 [Test] vs [TestMethod]. ▼
Basic
✦ Model Answer

They are functionally equivalent but belong to different frameworks: [Test] is the NUnit attribute, and [TestMethod] is the MSTest attribute used to identify a runnable test case.

Q338 Setup and Teardown methods. ▼
Intermediate
✦ Model Answer
  • Setup: Runs before **each** individual test method in the class (Example: initializing the driver).
  • Teardown: Runs after **each** individual test method (Example: quitting the driver).
Q339 OneTimeSetup vs OneTimeTeardown. ▼
Intermediate
✦ Model Answer

These run only **once** per class (or namespace). Use them for heavy initialization that can be shared across all tests, like starting a database container or reading a massive config file.

Q340 Soft Assertions vs Hard Assertions. ▼
Advanced
✦ Model Answer
  • Hard Assertion: Fails the test immediately on the first failure.
  • Soft Assertion: Records failures but allows the test to continue, reporting all accumulated failures at the end. Useful for verifying multiple fields on a single page.
Q341 What are Fluent Assertions? ▼
Intermediate
✦ Model Answer

A library that provides a more readable, natural language syntax for assertions. Example: totalCount.Should().Be(10).And.NotBeNegative(); instead of Assert.AreEqual(10, totalCount);.

Q342 Data-driven testing with [TestCase]. ▼
Basic
✦ Model Answer

An NUnit attribute that allows you to provide multiple sets of inputs to a single test method. Example: [TestCase("user1", "pass1")] [TestCase("user2", "pass2")] will run the test twice with different data.

Q343 Using [TestCaseSource] for complex data. ▼
Advanced
✦ Model Answer

When your test data is too large for attributes, use [TestCaseSource]. It points to a method or property that returns an IEnumerable of test data, which can be pulled from a JSON, CSV, or Database.

Q344 How to assert that an Exception occurred? ▼
Intermediate
✦ Model Answer

Use Assert.Throws<TException>(() => methodToTest()). This verifies that the code raises the correct exception type and allows you to inspect the exception message.

Q345 How to skip a test programmatically? ▼
Intermediate
✦ Model Answer

Use Assert.Ignore("Reason") or Assert.Inconclusive() in NUnit. This is useful when a test depends on a resource (like a specific DB) that is currently unavailable.

Q346 Test Categories/Traits for filtering. ▼
Basic
✦ Model Answer

Use the [Category("Name")] attribute. It allows you to group tests (e.g., smoke, nightly, slow) so the CI pipeline can choose which subsets to run.

Q347 Parallelization at class vs method level. ▼
Advanced
✦ Model Answer
  • Class Level: Different classes run in parallel, but tests *within* a class run sequentially. Safest for complex state.
  • Method Level: Every single test method can run in parallel. Fastest, but requires perfect thread safety.
Q348 Asserting on Collections. ▼
Intermediate
✦ Model Answer

NUnit provides CollectionAssert. Useful methods include Contains(), AreEqual() (checks order), and AreEquivalent() (checks elements regardless of order).

Q349 Asserting on private methods. ▼
Advanced
✦ Model Answer

Technically possible via Reflection, but a **bad practice**. You should test the public surface of the class; if a private method is complex enough to need its own test, it probably belongs in its own helper class with public methods.

Q350 Comparing objects: Deep vs Shallow. ▼
Advanced
✦ Model Answer
  • Shallow: Compares if the references are exactly the same.
  • Deep: Compares the values of all properties within the objects. Usually requires a library like Fluent Assertions or a JSON-based comparison.
Q351 Why is "Determinism" important in tests? ▼
Leadership
✦ Model Answer

A deterministic test gives the same result every time it's run on the same code. Non-deterministic (flaky) tests destroy team trust and lead to the "crying wolf" effect where real failures are ignored.

Q352 How to handle "Flaky Tests"? ▼
Leadership
✦ Model Answer

Quarantine them immediately. Move them to a [Category("flaky")], fix the underlying race condition or environment issue, and only move them back to the main suite after they pass 10+ times in a row.

Q353 Integrating with NLog/Serilog. ▼
Intermediate
✦ Model Answer

Instantiate a logger in your BaseTest. Configure it to write to both a file (for the run artifact) and the Console (so the test runner captures it and displays it in the test results window).

Q354 Capturing Console output in test results. ▼
Basic
✦ Model Answer

Anything written to Console.WriteLine() or TestContext.WriteLine() is automatically captured by the .NET test runner and stored alongside the individual test result in the TRX file.

Q355 What is the TRX file format? ▼
Intermediate
✦ Model Answer

TRX (Visual Studio Test Results) is an XML-based file format used by Microsoft tools to store comprehensive data about a test run, including timings, machine info, and stdout/stderr.

Q356 Allure Reporting integration. ▼
Advanced
✦ Model Answer

Allure converts raw test results into a rich, interactive web dashboard. Integrating it requires adding the Allure.NUnit NuGet package and configuring an allureConfig.json to define the output directory.

Q357 ExtentReports initialization. ▼
Advanced
✦ Model Answer

Initialize the ExtentReports and ExtentHtmlReporter in the [OneTimeSetUp] helper. Create a new ExtentTest instance for each test case in the [SetUp] method to ensure logs are correctly isolated.

Q358 Logging screenshots to reports. ▼
Intermediate
✦ Model Answer

Inside the [TearDown], check if TestContext.CurrentContext.Result.Outcome.Status == TestStatus.Failed. If so, take a screenshot and use the reporter's AddScreenCaptureFromPath() method to link it to the test report.

Q359 App.config vs appsettings.json. ▼
Intermediate
✦ Model Answer
  • App.config: Legacy XML-based config for .NET Framework.
  • appsettings.json: Modern, hierarchical JSON-based config for .NET Core/.NET 5+, supporting environment overrides (e.g., appsettings.Development.json).
Q360 Environment-based config switching. ▼
Advanced
✦ Model Answer

Use the ASPNETCORE_ENVIRONMENT variable. Your framework's ConfigurationBuilder should load the base JSON first, then optionally provide an override based on the environment variable, effectively allowing you to run the same code against Staging or Prod by just changing one variable.

Q361 Global Setup/Cleanup (SetUpFixture). ▼
Advanced
✦ Model Answer

In NUnit, a class marked with [SetUpFixture] outside any namespace (or inside a parent namespace) will run its Setup/Teardown exactly once for the entire assembly run. Ideal for starting/stopping the heavy infrastructure.

Q362 Retrying failed tests (Retry Attribute). ▼
Advanced
✦ Model Answer

NUnit provides [Retry(3)]. It will rerun a failed test up to 3 times. If it passes on any attempt, the whole run is marked as green. Use this sparingly to manage noise in highly unstable environments.

Q363 Measuring Test Execution Time. ▼
Basic
✦ Model Answer

Every modern Test Runner measures the duration from start to finish. In code, you can use Stopwatch or access TestContext.CurrentContext.Result.Duration during teardown for custom benchmarking.

Q364 Mocking vs Stubbing. ▼
Intermediate
✦ Model Answer
  • Stub: A simple fake that provides a specific response to a call (State-based).
  • Mock: A more complex fake that verifies that specific interactions occurred (Behavior-based). In automation, we mostly use Mocks to simulate downstream APIs.
Q365 Using Moq or NSubstitute. ▼
Advanced
✦ Model Answer

These are libraries used to create mock objects in C#. var mockRepo = new Mock<IRepository>();. They allow you to test your automation framework components (like a DB helper or an API client) without needing the actual external service to be running.

Q366 Testing Async methods. ▼
Intermediate
✦ Model Answer

Your test methods should be marked async Task and you must await every call to the system under test. NUnit and xUnit fully support async tests, ensuring the runner waits for the task to complete before moving to teardown.

Q367 Thread-safe driver management. ▼
Advanced
✦ Model Answer

When running tests in parallel, use [ThreadStatic] or ThreadLocal<IWebDriver>. This ensures that every test thread gets its own isolated instance of the browser, preventing one test from closing another test's window.

Q368 Parameterized tests from CSV. ▼
Intermediate
✦ Model Answer

Create a static method that reads a CSV file and yields TestCaseData objects. Reference this method in your test using [TestCaseSource("MethodName")]. This keeps your test data decoupled from your C# code.

Q369 Parameterized tests from Excel. ▼
Advanced
✦ Model Answer

Use a library like ExcelDataReader. Since Excel is a heavy dependency, it’s best to only use it when strictly required by business stakeholders; otherwise, CSV or JSON is preferred for speed and simplicity in CI.

Q370 Parameterized tests from Database. ▼
Advanced
✦ Model Answer

Use Dapper or ADO.NET inside a TestCaseSource. This is powerful for "Production Mirror" testing where you pull real IDs from a staging DB to verify they function correctly in the UI.

Q371 Role of a BaseTest class. ▼
Leadership
✦ Model Answer

A BaseTest centralizes common logic: driver instantiation, logging setup, screenshot capture on failure, and environment configuration reading. It ensures a consistent "Standard Operating Procedure" for every test in the suite.

Q372 Dependency Injection in Test Frameworks. ▼
Advanced
✦ Model Answer

Modern frameworks use DI (like Microsoft.Extensions.DependencyInjection) to inject Page Objects and Service Helpers into the test classes. This makes tests loosely coupled and significantly easier to unit test themselves.

Q373 Handling Secrets in local development. ▼
Advanced
✦ Model Answer

Use UserSecrets (via dotnet user-secrets). This stores sensitive data (like DB passwords or API keys) in a JSON file outside your project folder, ensuring they are never accidentally committed to Git.

Q374 Best practices for test naming. ▼
Leadership
✦ Model Answer

Use a standard pattern like [UnitOfWork]_[StateUnderTest]_[ExpectedBehavior]. Example: Login_ValidCredentials_RedirectsToDashboard. The name should act as documentation for what is being verified.

Q375 What is the AAA Pattern? ▼
Basic
✦ Model Answer
  • Arrange: Set up the test (data, mocks, page objects).
  • Act: Perform the action being tested.
  • Assert: Verify the result is correct.
Q376 Single Responsibility Principle in Tests. ▼
Advanced
✦ Model Answer

A test should have exactly **one** reason to fail. If you are testing the login process AND the profile update in a single test case, you are violating SRP, making failures harder to diagnose.

Q377 Test Data Builders pattern. ▼
Advanced
✦ Model Answer

A builder class provides a fluent way to create complex test data. var user = new UserBuilder().WithRole("Admin").Enabled().Build();. This keeps test setup readable and prevents "The Object Mother" bloat.

Q378 Objective of a "Smoke" suite. ▼
Leadership
✦ Model Answer

To verify the **most critical** paths of the application (e.g., Can I login? Does the home page load?). It runs in minutes and provides immediate confidence that a build is not "dead on arrival."

Q379 Objective of a "Regression" suite. ▼
Leadership
✦ Model Answer

To ensure that **previous functionality** still works after a new change. It is comprehensive, takes much longer to run, and is usually triggered nightly or on a release branch.

Q380 Objective of a "Sanity" suite. ▼
Leadership
✦ Model Answer

To verify that a **specific bug fix** or a small area of the system is working as expected. It's more focused than regression but broader than unit testing.

Q381 Verifying Logs as part of a test. ▼
Advanced
✦ Model Answer

In some safety-critical systems, or when testing audit trails, your automation should read the application logs or DB audit tables to verify that the correct "Silent Behavior" occurred in the background.

Q382 Cleaning up test data after execution. ▼
Advanced
✦ Model Answer

Always endeavor to clean up! Use the [TearDown] or [AfterScenario] to delete any records created during the test. If deletion is impossible, append unique IDs (like GUIDs) to every created entity to avoid naming collisions in the next run.

Q383 Handling Database Transactions in tests. ▼
Advanced
✦ Model Answer

A pro-tip is to wrap your test in a TransactionScope. When the test finishes (Pass or Fail), you simply **dispose** the transaction without committing it. This leaves the database in the exact same state it was in before the test started.

Q384 Feature Toggles in Automation. ▼
Intermediate
✦ Model Answer

Your framework should be able to query the application's feature toggle state (e.g., via an API) and skip or adapt tests accordingly. Don't let your automation fail just because a feature was turned off in the environment.

Q385 Comparing JSON responses in tests. ▼
Intermediate
✦ Model Answer

Use JToken.DeepEquals() from Newtonsoft.Json or JsonSerializer in System.Text.Json. This ignores whitespace and property order, focusing only on whether the data structure matches.

Q386 Waiting for events vs waiting for elements. ▼
Advanced
✦ Model Answer

Element-based waits look at the DOM. Event-based waits look at things like "Has the JS active count dropped to zero?" (AJAX) or "Has a specific event listener fired?". Event-based waits are generally much more stable for modern reactive SPAs.

Q387 Test Impact Analysis (TIA). ▼
Advanced
✦ Model Answer

A smart optimization that only runs the tests that are "impacted" by a code change. It maps code paths to test cases. If you only changed the PaymentService, TIA will skip the LoginTests, saving time in CI.

Q388 Value of specialized assertion libraries. ▼
Advanced
✦ Model Answer

Libraries like **Shouldly** or FluentAssertions provide much better failure messages. Instead of "Expected 10 but got 5", they might say "Collection 'orders' should contain 10 items but only found 5 because [list of items]". Faster diagnosis = faster fixes.

Q389 Reporting test coverage (OpenCover/dotCover). ▼
Advanced
✦ Model Answer

While useful for Unit tests, **Functional coverage** (testing requirements) is more important for QA. Use coverage tools to find "Dead Code" that is never hit by any automation, which helps identify untested areas of the app.

Q390 Integrating checks for memory leaks. ▼
Advanced
✦ Model Answer

In high-stability systems, your framework can capture memory snapshots before and after a test run. If memory usage doesn't return to the baseline, the test fails, helping catch leaks before they reach Production.

Q391 Using [Timeout] attribute. ▼
Basic
✦ Model Answer

This NUnit attribute sets a hard limit on how long a test can run (e.g., [Timeout(10000)] for 10 seconds). If it exceeds this, the test is forcibly killed and marked as failed, preventing hung tests from blocking the CI runner.

Q392 Using [Order] attribute. ▼
Intermediate
✦ Model Answer

It forces tests to run in a specific sequence. **Use it with caution!** Tests should ideally be independent. If TestA must run before TestB, you have "Test Interdependency," which makes parallelization and debugging much harder.

Q393 Final summary: What makes a "World Class" framework? ▼
Leadership
✦ Model Answer

A world-class framework is readable (anyone can understand the tests), stable (no random failures), fast (parallel and optimized), and maintainable (uses POM/SRP). It doesn't just "find bugs" — it provides a safety net that allows the team to deploy with total confidence.

Advanced
✦ Model Answer

Especially for Serverless/Azure Functions. The pipeline should hit the endpoint, measure the first response time (cold), then hit it again (warm) to ensure the cold start delay isn't violating the SLA.

Q449 Monitoring server logs during tests. ▼
Advanced
✦ Model Answer

The build agent can use a script to "tail" the server logs while tests are running. If a 'NRE' or 'Exception' appears in the log, the test should fail even if the UI looks okay, catching silent background errors.

Q450 Best practices for pipeline maintenance. ▼
Leadership
✦ Model Answer

Keep your YAML files modular. Use templates. Review your agent queue regularly. Delete old branches and their associated pipelines. A clean DevOps environment is a fast and secure one.

Q451 Integrating Slack/Teams for failures. ▼
Intermediate
✦ Model Answer

Configured via "Service Hooks" or the official Azure Pipelines app for Teams. It provides an immediate alert in the dev channel when a post-merge build fails, ensuring the team knows right away.

Q452 What is Artifact Tagging? ▼
Intermediate
✦ Model Answer

Applying meta-data (e.g., "ReadyForProd") to a build artifact after it has passed all testing stages. This tag acts as a permanent stamp of approval that survives even if the build run data is purged.

Q453 Final summary: Ultimate goal of robust CI/CD? ▼
Leadership
✦ Model Answer

To eliminate "Deployment Fear." A robust pipeline ensures that if code is in the main branch, it is already built, tested, and ready for Production. It turns a stressful manual release into a boring, automated non-event.

Agile, Scrum & QA Ceremonies
Sprints, DoD, Estimation, Three Amigos, Retrospectives, Velocity
Q454–Q513
Q454 What is Agile project management? ▼
Basic
✦ Model Answer

Agile is an iterative approach to project management and software development that helps teams deliver value to their customers faster and with fewer headaches. It focuses on continuous improvement, flexibility, and high-quality results rather than a single, massive delivery at the end.

Q455 The 4 values of the Agile Manifesto. ▼
Intermediate
✦ Model Answer
  • Individuals and interactions over processes and tools.
  • Working software over comprehensive documentation.
  • Customer collaboration over contract negotiation.
  • Responding to change over following a plan.
Q456 What are the 12 Principles of Agile? ▼
Leadership
✦ Model Answer

Twelve core guidelines that expand on the values, focusing on customer satisfaction, welcoming change, delivering frequently, team collaboration, technical excellence, and simplicity (the art of maximizing work NOT done).

Q457 The 3 roles in Scrum. ▼
Basic
✦ Model Answer
  • Product Owner: Represents the business and customer; manages the backlog.
  • Scrum Master: Facilitates the process and removes blockers for the team.
  • The Development Team: Self-organizing professionals who deliver the work (Developers, QA, Designers).
Q458 Responsibilities of a Product Owner (PO). ▼
Intermediate
✦ Model Answer

The PO is solely responsible for the "Value" produced. They must define project vision, prioritize backlog items based on business ROI, and ensure the team understands what they are building and why.

Q459 Responsibilities of a Scrum Master (SM). ▼
Intermediate
✦ Model Answer

The SM is a "Servant Leader." They protect the team from external distractions, facilitate ceremonies, coach the team in Agile practices, and ensure the Scrum process is followed to reach maximum velocity.

Q460 The Role of QA in the Development Team. ▼
Leadership
✦ Model Answer

In Agile, QA is not a separate phase. The QA professional is a core part of the team, involved from the start in story definition, providing "quality-first" thinking, and ensuring automation and manual checks are completed within the sprint.

Q461 Sprint Planning ceremony. ▼
Basic
✦ Model Answer

A meeting at the start of each sprint where the team selects stories from the prioritized backlog and commits to a specific "Sprint Goal" based on their capacity and velocity.

Q462 The Daily Standup (Daily Scrum). ▼
Basic
✦ Model Answer

A 15-minute time-boxed meeting to align the team. Everyone answers: 1) What was achieved yesterday? 2) What is the plan for today? 3) Are there any blockers? It focuses on "moving the ball" toward the sprint goal.

Q463 Sprint Review vs Sprint Retrospective. ▼
Intermediate
✦ Model Answer
  • Review: Demoing the "Working Software" to stakeholders for feedback on the product.
  • Retrospective: Looking inward at the "Process." Discussing what went well and what the team can improve in the next sprint.
Q464 What is Backlog Refinement (Grooming)? ▼
Intermediate
✦ Model Answer

An ongoing process of adding detail, estimates, and order to items in the backlog. It ensures that stories are "Ready" for planning and that the PO has enough refined work for the next 1-2 sprints.

Q465 INVEST criteria for User Stories. ▼
Advanced
✦ Model Answer
  • Independent: Story can stand alone.
  • Negotiable: Open to discussion.
  • Valuable: Delivers business value.
  • Estimable: Team can gauge effort.
  • Small: Fits within a sprint.
  • Testable: Has clear acceptance criteria.
Q466 Story Points vs Hours. ▼
Intermediate
✦ Model Answer

Hours are absolute; story points are relative. Points estimate complexity, risk, and effort collectively. This accounts for the fact that a "2-hour" task for a senior dev might take a junior dev 8 hours, but they both agree it’s a "3-point" story.

Q467 What is Planning Poker? ▼
Basic
✦ Model Answer

A consensus-based estimation technique. Team members use cards (Fibonacci sequence) to vote on story points. It prevents "anchoring" (where the loudest voice sets the estimate) and forces discussion on outliers.

Q468 Definition of Ready (DoR). ▼
Intermediate
✦ Model Answer

A checklist used by the team to determine if a story is refined enough to be pulled into a sprint. Example: Clear AC, wireframes attached, external dependencies identified.

Q469 Definition of Done (DoD). ▼
Intermediate
✦ Model Answer

The "Quality Standard" for a story. A story is not "Done" until every item on this list is checked (e.g., Code Reviewed, Unit Tests pass, Regression tested, Documentation updated).

Q470 What is Agile Velocity? ▼
Intermediate
✦ Model Answer

The average amount of "Story Points" a team completes per sprint. It is a historical metric used for planning future sprints and forecasting release dates, not for measuring individual performance.

Q471 Burndown Chart interpretation. ▼
Intermediate
✦ Model Answer

A graph showing remaining work (Y-axis) vs. time (X-axis). If the line is above the "Ideal" diagonal, the team is behind schedule; if below, they are ahead. It is a vital tool for the SM to flag risks early.

Q472 Kanban vs Scrum (WIP limits). ▼
Advanced
✦ Model Answer
  • Scrum: Fixed time-boxes (sprints) with a committed scope.
  • Kanban: Continuous flow of work. Uses **WIP (Work in Progress) limits** to prevent bottlenecks and ensure the team only starts what they can finish.
Q473 Concept of "Scrum of Scrums". ▼
Advanced
✦ Model Answer

A scaling technique where ambassadors from multiple Scrum teams meet regularly to discuss inter-team dependencies, blockers, and shared risks. Essential for large enterprise projects.

Q474 Scaled Agile Framework (SAFe) overview. ▼
Leadership
✦ Model Answer

A rigorous framework for scaling Agile to thousands of people. It introduces concepts like the **Agile Release Train (ART)** and **Program Increments (PI)** to align many teams around a single large-scale goal.

Q475 Why have a Sprint Goal? ▼
Intermediate
✦ Model Answer

The Goal gives the team a single, clear objective for the sprint. It allows for flexibility—if a story turns out to be massive, the team can swap it for something else as long as they still meet the overall Goal.

Q476 Handling "Bugs" in a Sprint. ▼
Intermediate
✦ Model Answer
  • Sprint Bugs: Found in new code; must be fixed in the same sprint to meet DoD.
  • Backlog Bugs: Found in legacy code; must be prioritized by the PO and brought into a future sprint after refinement.
Q477 What is a "Spike" story? ▼
Advanced
✦ Model Answer

A story used for research and investigation, not building code. If a requirement is too vague to estimate, a Spike is used to explore technical solutions and provide the info needed for a real estimate in the next sprint.

Q478 Technical Debt in Agile. ▼
Leadership
✦ Model Answer

The cost of "cutting corners" to meet a sprint deadline. Agile teams should manage debt by dedicating a percentage of every sprint (e.g., 20%) to refactoring and improving the code base's health.

Q479 Empirical Process Control. ▼
Advanced
✦ Model Answer

The three pillars of Scrum: 1) **Transparency** (everything visible to those responsible), 2) **Inspection** (check progress toward goals), and 3) **Adaptation** (adjust process if outcomes are deviating).

Q480 Acceptance Criteria vs DoD. ▼
Intermediate
✦ Model Answer
  • AC: Unique to each story (What do I build?).
  • DoD: Universal across all stories (How well do I build it?). You need both to declare a story "Done."
Q481 Role of QA in Backlog Refinement. ▼
Intermediate
✦ Model Answer

QA’s "Negative Intelligence" is vital here. While the PO focus is on usage, QA asks "What if X happens?", "How do we handle error Y?", and ensures the AC is unambiguous and testable from day one.

Q482 What is a Cross-functional Team? ▼
Basic
✦ Model Answer

A team that has all the skills necessary to turn a requirement into a working increment without relying on anyone outside the team. It is the opposite of a siloed "Dev Team" and "QA Team."

Q483 What is a Self-organizing Team? ▼
Leadership
✦ Model Answer

A team that decides *how* to deliver the work best without being micromanaged. They don't wait for a manager to assign tasks; instead, they pull work from the sprint backlog based on capacity and skill.

Q484 The "Three Amigos" discussion. ▼
Intermediate
✦ Model Answer

A collaborative meeting between the PO (Business), Developer (Technical), and QA (Testing). They discuss a story *before* it starts to align on behavior, clarify acceptance criteria, and uncover edge cases early (Shift Left).

Q485 Scope Creep in Agile. ▼
Leadership
✦ Model Answer

Adding new requirements mid-sprint. It endangers the Sprint Goal. Agile handles this by allowing the PO to add to the *Backlog*, but once a sprint starts, the scope is ideally locked unless something is removed to make room.

Q486 Measuring Agile Quality (Metrics). ▼
Leadership
✦ Model Answer

Avoid "Count of Bugs." Instead, use: 1) **Defect Leakage** (Bugs found in Prod), 2) **Test Pass Rate** over time, 3) **Automation ROI**, and 4) **Lead Time** (time from idea to production).

Q487 What is Defect Leakage? ▼
Intermediate
✦ Model Answer

A metric that calculates the percentage of defects that weren't caught by the QA team and "leaked" into Production. Formula: (Prod Defects / Total Defects) * 100. A high percentage indicates a gap in the testing strategy.

Q488 Definition of "Agile Testing." ▼
Basic
✦ Model Answer

Testing that follows the principles of agile software development. It involves cross-functional teams, continuous feedback, and the belief that quality is a shared responsibility, not just the "QA's job" at the end.

Q489 What are the Agile Testing Quadrants? ▼
Advanced
✦ Model Answer

A framework by Brian Marick/Lisa Crispin: Q1 (Unit/Auto - Support team), Q2 (Functional/Manual - Support team), Q3 (Exploratory/UAT - Critique product), Q4 (Performance/Security/Tools - Critique product).

Q490 Auto vs Manual testing in a Sprint. ▼
Intermediate
✦ Model Answer

Automate repetitive, high-risk regression cases to save time. Use manual testing for exploratory work, UX feedback, and complex new features that aren't stable yet. You need a healthy balance of both.

Q491 Regression testing in Agile. ▼
Intermediate
✦ Model Answer

Since Agile involves frequent changes, regression testing must be fast. This is why automated regression suites running in a CI/CD pipeline are non-negotiable for successful Agile delivery.

Q492 In-Sprint Automation. ▼
Advanced
✦ Model Answer

The goal of automating a new feature *during* the same sprint it’s built in. It’s challenging because the UI is often changing. Strategy: Automate API tests first, then stable UI elements last.

Q493 Exploratory Testing in Agile. ▼
Intermediate
✦ Model Answer

Unscripted testing where the tester "explores" the app to find bugs that automated and scripted manual tests might miss. It relies on the tester's creativity and domain knowledge.

Q494 The concept of "Done-Done". ▼
Leadership
✦ Model Answer

A phrase used to emphasize that a story isn't just "Dev Complete" (Done) but has also been tested, verified, and meets every single requirement of the Definition of Done (Done-Done).

Q495 Agile Estimating (Relative sizing). ▼
Basic
✦ Model Answer

Comparing stories against each other rather than looking at them in isolation. "Story A is twice as complex as Story B." This is faster and more accurate for long-term forecasting than absolute time estimates.

Q496 What is a Release Burndown Chart? ▼
Intermediate
✦ Model Answer

Similar to a sprint burndown, but it tracks progress toward a major release goal over multiple sprints. It helps the PO decide if they need to cut scope or move the release date.

Q497 Burnup Chart vs Burndown Chart. ▼
Advanced
✦ Model Answer
  • Burndown: Shows how much work is *remaining*.
  • Burnup: Shows total work done vs. total project scope. It’s better for projects where the scope is frequently changing (growing).
Q498 Lead Time vs Cycle Time. ▼
Advanced
✦ Model Answer
  • Lead Time: Total time from when a request is made to when it is delivered.
  • Cycle Time: The time it takes for the team to actually work on and complete the item once it's started.
Q499 Cumulative Flow Diagram (CFD). ▼
Advanced
✦ Model Answer

A Kanban tool that shows the distribution of work across different states (To Do, Doing, Testing, Done). Widening bands indicate bottlenecks (e.g., if the 'Testing' band keeps getting thicker, you need more QA capacity).

Q500 Five Whys Technique. ▼
Basic
✦ Model Answer

An iterative interrogative technique used to explore the cause-and-effect relationships underlying a particular problem. By asking "Why?" five times, you move past symptoms to the actual root cause.

Q501 Root Cause Analysis (RCA) in Agile. ▼
Intermediate
✦ Model Answer

The process of identifying the underlying cause of a defect or process failure. In Agile, this is often done during Retrospectives or after a critical production defect is resolved to prevent recurrence.

Q502 Pair Testing (Co-testing). ▼
Intermediate
✦ Model Answer

Two people (often a Dev and a QA) testing the same feature together on one machine. It encourages knowledge sharing, cross-training, and catches bugs faster than testing in silos.

Q503 Session-Based Test Management (SBTM). ▼
Advanced
✦ Model Answer

A method specifically for managing exploratory testing. Testers work in uninterrupted "sessions" (usually 90 mins) with a specific charter, and produce a standardized report of their findings.

Q504 Crowd Testing in Agile. ▼
Advanced
✦ Model Answer

Outsourcing testing to a large group of external testers. While hard to fit into a 2-week sprint, it’s excellent for multi-device/multi-region compatibility testing just before a major release.

Q505 The importance of Feedback Loops. ▼
Leadership
✦ Model Answer

Agile is entirely built on feedback loops: Unit tests (seconds), CI build (minutes), Daily Standup (hours), Sprint Review (weeks). The goal is to shorten these loops so errors are caught as close to the source as possible.

Q506 Transparency, Inspection, Adaptation. ▼
Intermediate
✦ Model Answer

The three pillars of Empiricism. Without transparency, you can't inspect properly. Without inspection, you won't know when to adapt. Without adaptation, you aren't really Agile; you're just doing "Iterative Waterfall."

Q507 Handling team turnover in a Sprint. ▼
Leadership
✦ Model Answer

If someone leaves or joins, the team's velocity will drop while they adjust. A good SM/Lead manages stakeholder expectations and doesn't force the team to over-commit during this transition phase.

Q508 Mentoring in an Agile Team. ▼
Intermediate
✦ Model Answer

Agile teams should be "T-shaped" individuals (deep skill in one area, broad skills in others). Mentoring ensures that knowledge isn't siloed, reducing the "Bus Factor" and making the team more resilient.

Q509 Aligning QA Strategy with Sprint Goals. ▼
Leadership
✦ Model Answer

QA shouldn't just "test everything." They should prioritize testing the stories that are most critical to the **Sprint Goal**. If time is tight, focus on the Goal-critical items first.

Q510 Refresher: The Agile Testing Pyramid. ▼
Intermediate
✦ Model Answer

A graphical representation of the distribution of testing. Most tests at the bottom (Unit), then Service/API, then few UI tests at the top. This ensures a fast, stable, and cost-effective test suite.

Q511 Managing Product Backlog Health. ▼
Intermediate
✦ Model Answer

A healthy backlog is DEEP: Detailed appropriately, Estimated, Emergent, and Prioritized. QA helps by ensuring stories at the top of the backlog meet the Definition of Ready.

Q512 Role of QA Lead in Agile Transformation. ▼
Leadership
✦ Model Answer

They act as a coach, helping traditional "Waterfall" testers transition to an Agile mindset, advocating for automation investment, and breaking down the silos between Dev and QA at the leadership level.

Q513 Final summary: Why Agile works. ▼
Leadership
✦ Model Answer

Because it acknowledges that software development is complex and unpredictable. Instead of trying to plan everything perfectly upfront, Agile emphasizes "Learning through doing" and adapting to change as it happens.

API & Integration Testing
REST, HTTP methods, Response validation, Contract testing, Postman, RestSharp
Q514–Q573
Q514 What is a REST API? ▼
Basic
✦ Model Answer

REST (Representational State Transfer) is an architectural style for providing standards between computer systems on the web. It is stateless, uses standard HTTP methods, and represents resources as nouns (e.g., /api/users) rather than verbs.

Q515 Differences between SOAP and REST. ▼
Intermediate
✦ Model Answer
  • REST: Lightweight, supports JSON/XML/plain text, stateless, uses HTTP methods.
  • SOAP: Heavier, XML only, has built-in security features (WS-Security), stateful possibilities, uses a WSDL (Web Services Description Language).
Q516 JSON vs XML in API communication. ▼
Basic
✦ Model Answer
  • JSON: JavaScript Object Notation. Very lightweight, easy for browsers to parse, and overwhelmingly preferred for modern REST APIs.
  • XML: Extensible Markup Language. More verbose, supports rigid schemas (XSD), historically common in legacy SOAP systems.
Q517 Standard HTTP Verbs and their uses. ▼
Basic
✦ Model Answer
  • GET: Retrieve a resource.
  • POST: Create a new resource.
  • PUT: Replace/update an entire resource.
  • PATCH: Partially update a resource.
  • DELETE: Remove a resource.
Q518 What is Idempotency in APIs? ▼
Advanced
✦ Model Answer

A method is idempotent if performing it multiple times has the same effect as performing it once. GET, PUT, and DELETE are idempotent. POST is typically NOT idempotent (multiple POST calls result in multiple resources being created).

Q519 Anatomy of an HTTP Request. ▼
Basic
✦ Model Answer
  • Endpoint/URL: The address of the resource.
  • Method: The verb (GET, POST, etc.).
  • Headers: Metadata like Content-Type or Authorization.
  • Body: The actual data payload (usually JSON).
Q520 Anatomy of an HTTP Response. ▼
Basic
✦ Model Answer
  • Status Code: Indicating success or failure (e.g., 200 OK).
  • Headers: Response metadata (like Date or Server).
  • Body: The data returned by the server.
Q521 Common HTTP Status Codes to know. ▼
Basic
✦ Model Answer
  • 2xx: Success (200 OK, 201 Created).
  • 3xx: Redirection (301 Moved Permanently).
  • 4xx: Client Error (400 Bad Request, 401 Unauthorized, 404 Not Found).
  • 5xx: Server Error (500 Internal Server Error, 503 Service Unavailable).
Q522 Query Parameters vs Path Parameters. ▼
Intermediate
✦ Model Answer
  • Path: Identifies a specific resource: /users/123.
  • Query: Filters/sorts the resource list: /users?role=admin&sort=desc.
Q523 Common Request Headers (Content-Type vs Accept). ▼
Intermediate
✦ Model Answer
  • Content-Type: Tells the server the format of the data I AM SENDING (e.g., application/json).
  • Accept: Tells the server the format of the data I WANT BACK.
Q524 What is JSON Schema validation? ▼
Advanced
✦ Model Answer

A declarative language for validating the structure of a JSON response. Instead of checking every field one-by-one, we use a schema to ensure mandatory fields are present, types are correct (string/int), and constraints (max-length) are met.

Q525 What is Postman? ▼
Basic
✦ Model Answer

Postman is a popular API client that allows developers and QA engineers to create, share, and test individual HTTP requests or complex automated test suites in a user-friendly UI.

Q526 Postman Collections and Environments. ▼
Intermediate
✦ Model Answer
  • Collections: A group of saved requests organized into folders.
  • Environments: A set of variables (like baseUrl or apiKey) that allow you to reuse the same collection across different servers (Dev, QA, Prod).
Q527 Writing tests in Postman. ▼
Intermediate
✦ Model Answer

Postman uses JavaScript for testing. You write scripts in the "Tests" tab. Example: pm.test("Status is 200", function () { pm.response.to.have.status(200); });. These run automatically after a response is received.

Q528 What are Postman Pre-request scripts? ▼
Advanced
✦ Model Answer

JavaScript code that runs *before* the actual request is sent. Commonly used to generate dynamic timestamps, calculated HMAC signatures, or fetch an OAuth token that will be used in the main request.

Q529 What is Newman? ▼
Advanced
✦ Model Answer

Newman is the command-line collection runner for Postman. It allows you to run your Postman test suites directly from the terminal or as a task in a CI/CD pipeline (Jenkins/Azure DevOps).

Q530 What is RestSharp? ▼
Advanced
✦ Model Answer

A popular .NET library used for making HTTP calls. It simplifies the process of sending requests and automatically handles deserializing JSON/XML responses back into C# objects.

Q531 Serialization vs Deserialization. ▼
Intermediate
✦ Model Answer
  • Serialization: Converting a C# object into a JSON string to be sent in a request body.
  • Deserialization: Converting the JSON response string back into a C# object for easy manipulation in code.
Q532 How does Basic Authentication work? ▼
Basic
✦ Model Answer

It involves sending the username and password in the request header, encoded in Base64. Header format: Authorization: Basic [Base64-encoded string]. Note: Base64 is NOT encryption, so this is only safe over HTTPS.

Q533 What is a Bearer Token (JWT)? ▼
Intermediate
✦ Model Answer

A JSON Web Token (JWT) is a secure way to transmit information between parties as a JSON object. "Bearer" means the holder of this token is granted access. Header: Authorization: Bearer [Token].

Q534 Using API Keys for authentication. ▼
Basic
✦ Model Answer

A simple way to identify a project or caller. The key is usually sent as a custom header (e.g., X-API-Key) or as a query parameter in the URL. Easier to implement but less secure than OAuth.

Q535 Basic OAuth 2.0 Flow. ▼
Advanced
✦ Model Answer

1) User authorizes the app. 2) App gets an Authorization Code. 3) App exchanges the code for an Access Token. 4) App uses the token to call the API. This ensures the app never sees the user's actual password.

Q536 What is Contract Testing? ▼
Advanced
✦ Model Answer

Testing used to ensure two systems (e.g., a microservice and its client) agree on the "Contract" of their interaction. If the API producer changes a field name, the "Contract" fails, flagging the breakage before deployment.

Q537 Mocking vs Stubbing in API testing. ▼
Advanced
✦ Model Answer
  • Stubbing: Providing a hardcorded response for a specific call (e.g., "always return 200").
  • Mocking: Simulates a real service and can verify interactions (e.g., "verify this endpoint was called exactly twice").
Q538 Testing for "Statelessness". ▼
Advanced
✦ Model Answer

In REST, every request must contain all the info needed to process it. You test this by ensuring that if you run a request on a totally fresh server session (with only the auth token), it still works without needing previous "state" setup.

Q539 Negative Testing for APIs. ▼
Intermediate
✦ Model Answer

Sending "bad" data to ensure the API fails correctly. Example: Send a string for an integer field (expect 400), or send an expired token (expect 401). A robust API should never return 500 for user errors.

Q540 Testing Rate Limiting (429). ▼
Intermediate
✦ Model Answer

Intentional load testing where you bombard an endpoint to ensure it correctly returns a 429 Too Many Requests status code once a threshold is reached, protecting the backend from being overwhelmed.

Q541 What to check in Pagination testing? ▼
Intermediate
✦ Model Answer
  • Does page=1&size=10 return exactly 10 records?
  • Does navigating to page=2 show a different set of records?
  • Does the response meta-data (totalResults) correctly match the DB?
Q542 Sorting and Filtering via API. ▼
Intermediate
✦ Model Answer

Verify that parameters like ?sort=name,asc or ?filter=status:active actually change the response body as expected. Test combinations of multiple filters and sorts at once.

Q543 API Headers: CORS and Security headers. ▼
Advanced
✦ Model Answer

Ensure standard security headers are present: Strict-Transport-Security, Content-Security-Policy, and Access-Control-Allow-Origin. This prevents cross-site scripting and unauthorized domain access.

Q544 Testing File Uploads via API. ▼
Advanced
✦ Model Answer

Use multipart/form-data. Test with different file types (PDF, PNG), large files (size limits), and zero-byte files. Verify the server stores the file correctly and returns a valid path/ID.

Q545 Testing File Downloads via API. ▼
Advanced
✦ Model Answer

Verify the Content-Disposition header (attachment; filename=...). Download the byte stream and compare it against the expected file checksum (MD5) to ensure no corruption occurred during transit.

Q546 API Versioning strategies. ▼
Intermediate
✦ Model Answer
  • URL versioning: /api/v1/users (Most common, visible).
  • Header versioning: Accept: application/vnd.company.v1+json (Cleaner URLs, harder to test manually).
Q547 Testing breaking changes in versioning. ▼
Advanced
✦ Model Answer

When V2 is released, V1 must still pass its original tests (Backward Compatibility). Verify that removing a field in V2 doesn't break V1 consumers. Use Contract Tests to automate this verification.

Q548 SOAP: What is a WSDL? ▼
Intermediate
✦ Model Answer

Web Services Description Language. It's an XML file that describes everything about the service: endpoints, available methods, and data structures. It acts as the technical "manual" for a SOAP API.

Q549 GraphQL basics vs REST. ▼
Advanced
✦ Model Answer

REST uses multiple endpoints for different resources. GraphQL uses a SINGLE endpoint. The client specifies exactly which fields they need in the request body, avoiding "over-fetching" data.

Q550 Testing GraphQL Queries and Mutations. ▼
Advanced
✦ Model Answer
  • Queries: Test fetching nested data and ensuring only requested fields are returned.
  • Mutations: Test creating/updating data. Verify the returned object reflects the change. Check for proper error messages when schema is violated.
Q551 API Security: SQL Injection via params. ▼
Advanced
✦ Model Answer

Attempt to send SQL commands in query strings (e.g., ?id=1' OR '1'='1). A secure API uses parameterized queries and should return a 400 error, never executing the malicious SQL.

Q552 API Security: XSS in responses. ▼
Advanced
✦ Model Answer

Ensure that script tags (<script>alert(1)</script>) sent in a POST body are correctly sanitized or encoded by the API before they are stored and later returned in a GET response.

Q553 What is Broken Object Level Authorization (BOLA)? ▼
Advanced
✦ Model Answer

A critical security flaw where a user can access another user's data by guessing their ID in the URL (e.g., User A changing /api/my-profile/101 to 102). Test this by using a token for User A to try and access User B's resource.

Q554 What is an API Gateway? ▼
Intermediate
✦ Model Answer

A single entry point for all clients. It handles cross-cutting concerns like Authentication, Rate Limiting, SSL termination, and Routing to various microservices. Examples: Azure API Management, Kong, Zuul.

Q555 Testing in a Service Mesh (Istio). ▼
Advanced
✦ Model Answer

Service Mesh manages service-to-service communication. For testers, it provides features like **Fault Injection** (testing how the system handles 503 errors) and **Traffic Shifting** (Canary releases).

Q556 Distributed Tracing (Correlation IDs). ▼
Advanced
✦ Model Answer

In microservices, one request may hit 10 services. A **Correlation ID** is passed in the header across all calls. If a failure occurs, you search logs for that ID to see exactly where in the chain the error happened.

Q557 Parallel API testing. ▼
Intermediate
✦ Model Answer

Running multiple API tests at once to reduce total execution time. Must ensure test data is isolated (e.g., each parallel thread creates its own user) to prevent race conditions or data collisions.

Q558 Load Testing vs Functional API testing. ▼
Intermediate
✦ Model Answer
  • Functional: Does the API return the correct data for one user?
  • Load: Does the API still return correct data (and fast) for 10,000 concurrent users?
Q559 Latency vs Throughput. ▼
Advanced
✦ Model Answer
  • Latency: Time taken for a SINGLE request to get a response (measured in ms).
  • Throughput: Number of requests the API can handle per second (RPS).
Q560 Testing SSL/TLS handshake for APIs. ▼
Advanced
✦ Model Answer

Verify the API only accepts secure connections (TLS 1.2+). Attempt to connect via HTTP (should redirect or fail) or older, weak TLS versions to ensure the security configuration is tight.

Q561 Data-masking in API responses. ▼
Intermediate
✦ Model Answer

Check that sensitive data like credit card numbers or SSNs are masked in the response (e.g., ****-****-****-1234) unless the specifically authorized user has permissions to see the full data.

Q562 What is Swagger/OpenAPI? ▼
Basic
✦ Model Answer

A standard specification for documenting REST APIs. It provides a machine-readable JSON/YAML file that describes all endpoints, their inputs, and expected outputs.

Q563 Benefits of Swagger UI for testers. ▼
Basic
✦ Model Answer

It provides an interactive web page where you can "Try it out"—sending real requests to the API and seeing the responses without writing any code. It’s the best way to start exploratory testing on a new API.

Q564 Testing API + Event Bus (RabbitMQ). ▼
Advanced
✦ Model Answer

Post a request to the API (e.g., Create Order). Instead of checking a DB, verify that the API correctly published a OrderCreated message to the RabbitMQ exchange with the right payload.

Q565 How to test WebHooks? ▼
Advanced
✦ Model Answer

WebHooks are "Reverse APIs" (the server calls YOU). Test this by using a service like **RequestBin** or **Webhook.site** to provide a temporary URL. Trigger the event and verify the payload sent to the WebHook URL is correct.

Q566 Idempotency Keys in financial APIs. ▼
Advanced
✦ Model Answer

Test by sending a Payment request with a unique Idempotency-Key header. Send exact same request 5 times. Verify the bank only charges the user ONCE and returns the same success response for all 5 calls.

Q567 Testing "HATEOAS". ▼
Advanced
✦ Model Answer

Hypermedia as the Engine of Application State. Verify the API response contains LINKS (e.g., "rel": "next") that tell the client what they can do next. The client should "navigate" the API by following these links.

Q568 Content Negotiation testing. ▼
Intermediate
✦ Model Answer

Send a request with Accept: application/xml and verify the API returns XML. Send the same request with Accept: application/json and verify it returns JSON. This ensures the API correctly honors headers.

Q569 Testing for "Gzip" compression. ▼
Advanced
✦ Model Answer

Q570 API Caching (ETags). ▼
Advanced
✦ Model Answer

1) Get resource, save ETag from header. 2) Call GET again with If-None-Match: [ETag]. 3) Verify server returns 304 Not Modified and an empty body, saving bandwidth.

Q571 Microservices testing strategy. ▼
Leadership
✦ Model Answer

Q572 Troubleshooting with Fiddler/Charles. ▼
Intermediate
✦ Model Answer

Q573 Final summary: The API-First future. ▼

In modern architecture, the UI is just a "wrapper" around APIs. Testing the API tier is the highest ROI activity for a QA. By catching bugs here, you prevent them from ever surfacing in the complex and expensive-to-test UI layer.

Q574 What is a Relational Database (RDBMS)? ▼
Basic
✦ Model Answer

An RDBMS (Relational Database Management System) organizes data into tables (relations) with rows and columns. It uses SQL for data manipulation and ensures integrity via keys and constraints. Common examples: SQL Server, MySQL, Oracle, PostgreSQL.

Q575 Primary Key vs Foreign Key. ▼
Basic
✦ Model Answer
  • Primary Key (PK): A unique identifier for each record in a table (e.g., PatientID). Cannot be NULL.
  • Foreign Key (FK): A field in one table that refers to the PK in another table, creating a relationship (e.g., PatientID in the Appointments table).
Q576 Importance of NULL in data testing. ▼
Intermediate
✦ Model Answer

NULL represents "missing" or "unknown" data, not zero or empty. Testers must verify how the application handles NULLs—for example, ensuring mandatory fields reject NULLs and optional fields display them gracefully in the UI.

Q577 Testing UNIQUE constraints. ▼
Intermediate
✦ Model Answer

UNIQUE ensures all values in a column are different (e.g., Email). Test by attempting to INSERT two records with the same email; the DB should throw a constraint violation error, which the UI should handle as a validation message.

Q578 Testing NOT NULL constraints. ▼
Basic
✦ Model Answer

NOT NULL ensures a field must have a value. Test by attempting to save a record with a blank mandatory field; verify the DB rejects the transaction and the API/UI returns the correct 'Field is required' error.

Q579 SELECT: Basic syntax for data verification. ▼
Basic
✦ Model Answer

Used to retrieve data. Syntax: SELECT column1, column2 FROM TableName;. QA uses this to verify that data entered in a UI form has actually landed correctly in the database tables.

Q580 Filtering data with the WHERE clause. ▼
Basic
✦ Model Answer

Used to extract only records that fulfill a specific condition. Example: SELECT * FROM Patients WHERE Status = 'Active';. Vital for testers searching for their specific test-run data in a shared environment.

Q581 Logical operators: AND, OR, NOT. ▼
Basic
✦ Model Answer
  • AND: All conditions must be true.
  • OR: At least one condition must be true.
  • NOT: Records where the condition is not true.
Q582 Using IN and BETWEEN for range testing. ▼
Intermediate
✦ Model Answer
  • IN: Check if a value matches any value in a list (e.g., WHERE City IN ('London', 'Paris')).
  • BETWEEN: Check for values within a range (e.g., WHERE Age BETWEEN 18 AND 65).
Q583 LIKE operator: Wildcard/Search testing. ▼
Basic
✦ Model Answer

Used in a WHERE clause to search for a specified pattern. Wildcards: % (multiple chars) and _ (single char). Example: WHERE Name LIKE 'J%' find names starting with J. Used to test "Search" functionality in UIs.

Q584 ORDER BY: Verifying UI sorting. ▼
Basic
✦ Model Answer

Used to sort the result-set in ascending (ASC) or descending (DESC) order. QA uses this to verify that a "Sort" button in a table works correctly by comparing the DB output with the UI display.

Q585 DISTINCT: Finding unique values/duplicates. ▼
Basic
✦ Model Answer

The SELECT DISTINCT statement is used to return only distinct (different) values. QA uses this to check data integrity, such as verifying if there are duplicate User accounts that shouldn't exist.

Q586 Aggregate functions: COUNT, SUM, AVG. ▼
Intermediate
✦ Model Answer
  • COUNT(): Returns the number of rows.
  • SUM(): Returns the total sum of a numeric column.
  • AVG(): Returns the average value of a numeric column.
Q587 Aggregate functions: MIN, MAX. ▼
Basic
✦ Model Answer
  • MIN(): Returns the smallest value.
  • MAX(): Returns the largest value.

Used to verify boundary values in datasets (e.g., finding the oldest and newest patient records).

Q588 GROUP BY: Testing reporting/dashboard data. ▼
Intermediate
✦ Model Answer

Groups rows that have the same values into summary rows. Example: SELECT Country, COUNT(*) FROM Customers GROUP BY Country;. Vital for verifying Charts and Analytics on a dashboard.

Q589 HAVING vs WHERE. ▼
Intermediate
✦ Model Answer

WHERE filters individual rows before grouping. HAVING filters the results of a GROUP BY. Example: "Show countries with more than 10 customers" → GROUP BY Country HAVING COUNT(*) > 10.

Q590 Inner Join: Verifying related data. ▼
Intermediate
✦ Model Answer

Returns records that have matching values in both tables. Example: Orders JOIN Customers ON Orders.CustomerID = Customers.CustomerID. Used to verify the relationship between users and their actions.

Q591 Left Join: Finding orphan records or missing data. ▼
Intermediate
✦ Model Answer

Returns all records from the left table, and matched records from the right. If no match, result is NULL on the right. QA uses this to find "Orphan" records (e.g., Patients who have no assigned primary doctor).

Q592 Right Join basics. ▼
Basic
✦ Model Answer

Inverse of Left Join. Returns all records from the right table. Less common in daily QA but important to understand for full-set data reconciliation.

Q593 Full Outer Join basics. ▼
Intermediate
✦ Model Answer

Returns all records when there is a match in either left or right table. Useful for comparing two huge datasets to see what exists in 'A' but not 'B', and vice-versa.

Q594 Self Join: Testing hierarchical data. ▼
Advanced
✦ Model Answer

A table is joined with itself. Used to represent relationships like "Employee reports to Manager" where both are in the same 'Employees' table. Vital for testing org-chart or directory features.

Q595 Subqueries: Testing nested logic. ▼
Advanced
✦ Model Answer

A query within a query. Example: SELECT Name FROM Patients WHERE ID IN (SELECT PatientID FROM Appointments WHERE Date = '2023-10-01'). Helps verify complex business rules that aren't visible via simple joins.

Q596 EXISTS vs IN. ▼
Advanced
✦ Model Answer

EXISTS is generally faster for checking existence, while IN is better for a static list. Testers use both to verify if a record exists in a related table without pulling all its data.

Q597 INSERT: Setting up test data via SQL. ▼
Basic
✦ Model Answer

QA uses INSERT to "seed" the database with known state before running an automated test. Syntax: INSERT INTO Table (Cols) VALUES (Vals);. Essential for repeatable, predictable tests.

Q598 UPDATE: Modifying state for scenario setup. ▼
Basic
✦ Model Answer

QA uses UPDATE to put an object into a specific state, like setting an Appointment to 'Expired' to test that the UI correctly hides the 'Cancel' button. Always use a WHERE clause to avoid updating the whole table!

Q599 DELETE: Cleaning up test data. ▼
Basic
✦ Model Answer

Vital for the [Teardown] phase of testing. QA runs DELETE commands to remove the patient records, orders, or logs created during the test run, keeping the environment clean for the next person.

Q600 TRUNCATE vs DELETE. ▼
Intermediate
✦ Model Answer
  • DELETE: DML (Data Manipulation). Can be rolled back, logs every row removal, triggers fire.
  • TRUNCATE: DDL (Data Definition). Faster, removes ALL rows, cannot be rolled back easily, resets identity seeds. Use for complete clean-room reset.
Q601 ACID properties: Why they matter to testers. ▼
Advanced
✦ Model Answer

Atomicity, Consistency, Isolation, Durability. These guarantee that DB transactions are processed reliably. For a tester, an ACID violation might look like a bank transfer where money is deducted from A but never arrives at B.

Q602 Testing Database Transactions. ▼
Intermediate
✦ Model Answer

Verify that if one step in a multi-step process fails (e.g., deducting stock fails after creating order), the entire transaction reverts, leaving no "dirty" data. Test by intentionally breaking the second half of a process.

Q603 ROLLBACK: Verifying cleanup. ▼
Intermediate
✦ Model Answer

When a transaction is "rolled back," the DB state returns to exactly how it was before. QA must verify that after a failed operation (like a timeout), NO partial data was accidentally saved to the DB.

Q604 COMMIT: Verifying persistence. ▼
Basic
✦ Model Answer

Verify that once a transaction says "Success" in the UI, it is permanently saved. Test by killing the application process immediately after a COMMIT; the data should still be there when the app restarts.

Q605 Parameterized Queries: Preventing SQL Injection. ▼
Advanced
✦ Model Answer

Using placeholders (like @ID) instead of string concatenation. Testers verify this by attempting a SQL injection attack (e.g., entering '; DROP TABLE users -- in a search box) and ensuring the app treats it as a search string, not a command.

Q606 Stored Procedures: Testing logic in DB. ▼
Intermediate
✦ Model Answer

A Stored Procedure is code saved on the DB server. Test them by calling them directly with varied inputs (Null, Negative, Max strings) and verifying the output and DB side-effects are correct.

Q607 Testing SQL Views. ▼
Intermediate
✦ Model Answer

A View is a "Virtual Table" based on a stored query. QA verifies that the View correctly masks sensitive data or correctly aggregates statistics from multiple underlying tables.

Q608 Database Triggers: Automation perspective. ▼
Advanced
✦ Model Answer

Triggers are SQL code that runs automatically on INSERT/UPDATE/DELETE. Testers must verify they fire correctly (e.g., an "Audit Log" is created every time a patient's address is updated).

Q609 Database Indexes: Testing for performance gaps. ▼
Advanced
✦ Model Answer

Indexes speed up retrieval but slow down INSERTs. In performance testing, QA should identify fields that are frequently used in 'Search' filters but lack indexes, causing slow UI response times.

Q610 Data Masking (PHI/PII) in Lower Environments. ▼
Advanced
✦ Model Answer

Check that "Staging" or "QA" databases have sensitive user data scrambled (e.g., Real Name → User 123, Real Phone → 555-0101). Vital for GDPR/HIPAA compliance in high-security industries like healthcare.

Q611 Referential Integrity: Testing FK violations. ▼
Intermediate
✦ Model Answer

Verify that the system prevents deleting a "Parent" record if "Child" records exist (e.g., deleting a Patient who still has Appointments). This ensures database consistency.

Q612 Testing Database Migrations. ▼
Advanced
✦ Model Answer

When updating the database schema (adding tables/cols), QA must verify: 1) Data was NOT lost, 2) Data was correctly moved to new columns, 3) The 'Rollback' script successfully reverts the DB if the migration fails.

Q613 SQL Profiler/Execution Plans for testers. ▼
Advanced
✦ Model Answer

Tools like 'SQL Profiler' let you see exactly what SQL queries the API is sending to the server. If a UI button click is slow, QA uses this to find the specific "bottleneck" query for optimization.

Q614 NoSQL vs SQL: Testing differences. ▼
Advanced
✦ Model Answer
  • SQL: Rigid schema, Joins, ACID compliance. Testing focus = Relationships.
  • NoSQL (MongoDB): Dynamic schema (JSON-like docs), horizontal scaling. Testing focus = Data structure consistency within documents.
Q615 What is ETL Testing? ▼
Advanced
✦ Model Answer

Extract, Transform, Load. Testers verify data is correctly pulled from source (E), transformed according to business rules (T), and loaded into a destination Data Warehouse (L) without corruption or record counts mismatched.

Q616 Schema Validation testing. ▼
Basic
✦ Model Answer

Verify that Table names, Column names, Data types (int/varchar), and lengths (Max 50 chars) strictly match the technical design specification document.

Q617 Bulk data testing. ▼
Intermediate
✦ Model Answer

Verifying how the application handles thousands of records at once—for example, ensuring that a "Download all Patient Records as PDF" feature doesn't time out when the DB has 100,000 entries.

Q618 Verifying Data Integrity after UI update. ▼
Basic
✦ Model Answer

The "Golden Rule" of Database Testing: Every success in the UI must be verified in the DB. If User says Account is deactivated, the DB Status flag MUST be '0' or 'Inactive'. Don't trust the UI alone.

Q619 Connecting C# to SQL (ADO.NET/Dapper) for automation. ▼
Advanced
✦ Model Answer

In automation code, use libraries like Dapper to run SQL queries directly. Example: var user = dbConnection.Query("SELECT * FROM Users WHERE ID=@id", new {id=1});. Used to assert DB state directly from technical test scripts.

Q620 Mocking a Database in Unit Tests. ▼
Advanced
✦ Model Answer

Instead of hitting a real SQL server (slow & fragile), use Moq or an 'In-Memory DB' to simulate database responses. This allows unit tests to run in milliseconds and prevents external dependency failures.

Q621 Best practices for Test Data Management. ▼
Intermediate
✦ Model Answer
  • Use isolated data per test (no sharing Users between tests).
  • Clean up after yourself (DELETE created data).
  • Avoid hardcoded IDs (fetch the generated ID after INSERT).
  • Use Data-driven inputs from external files (CSV/JSON).
Q622 Golden Datasets for regression. ▼
Advanced
✦ Model Answer

A "Golden Dataset" is a verified, static snapshot of database data used as the baseline for regression testing. Testers compare current output against this Golden set to find unexpected changes.

Q623 Final summary: Why every tester needs SQL. ▼
Leadership
✦ Model Answer

SQL is the universal language of data. By knowing it, a QA moves from being a "UI user" to a "Systems analyst," capable of verifying backend triggers, setting up complex test scenarios, and finding root causes of bugs that others miss. It's the #1 technical skill for high-level QA roles.

Q747 Expert: Verifying "Eventual Consistency" in NoSQL. ▼
